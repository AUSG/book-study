## CPU는 어떻게 입출력 작업을 처리할까?

- CPU에 레지스터가 있는 것처럼 장치에도 레지스터가 있다
	- 데이터를 저장하는 레지스터
		- 키보드 누르면 어떤 키를 눌렀는지 등을 저장
	- 제어 정보와 상태 정보를 저장하는 레지스터
		- 이를 통해 장치를 제어하거나 장치 상태를 볼 수 있음


- CPU 기계 명령어처럼 장치 레지스터를 읽고 쓰는 특정한 기계 명령어를 설계할 수 있다
- 이를 input and output instruction이라고 하는데 x86의 `IN`, `OUT`이 이에 해당
- 그렇다면 어떤 장치 레지스터를 써야 하는지 어떻게 알 수 있을까?
- 이런 구현 방식에서는 장치마다 고유한 주소가 부여되고, 입출력 명령에 장치 주소를 지정


- 이 외에도 방법이 있을까요?
- 사실 메모리도 외부 장치이며, 이미 `LOAD`, `STORE` 명령이 있으니 이를 활용할 수 있지 않을까?
- 그러나 그렇다면 CPU가 `LOAD`, `STORE` 명령 실행하면 이는 메모리를 읽고 쓰는 걸까요, 아니면 장치 레지스터를 읽고 쓰는 걸까요?


- `LOAD`, `STORE`는 메모리 주소 공간을 전달
- 주소 공간 일부에 장치를 할당하면, 메모리를 활용해서 장치를 할당할 수 있다

![](files/CleanShot%202026-03-08%20at%2017.43.06@2x.png)

- CPU가 명령을 실행할 때 내부 하드웨어가 이 정보를 보고, 앞 4비트가 모두 1이면 장치를 대상으로 동작하고 아니면 일반적인 메모리 읽기 명령어로 동작
- 이를 메모리 사상 입출력(memory mapping input and output)이라고 한다
- `LOAD R1 0xFE00` 같은 코드로 장치의 값을 레지스터에 가져올 수 있다


- 그렇다면 데이터를 가져오는 방법은 알겠는데, 사용자가 키보드를 언제 누를지 모르는데 이 시점을 어떻게 알 수 있을까?
- 장치 상태 레지스터를 읽으면 CPU는 장치를 읽을 수 있는지 쓸 수 있는지 알 수 있다


- 그렇다면 장치 상태 레지스터를 지속적으로 읽으면 될까?
- 물론 폴링으로 가능하지만 매우 비효율적이다
- CPU 속도와 달리 외부 장치의 속도는 그렇게 빠르지 않다
- 따라서 이를 인터럽트로 처리하면 효율적으로 구현할 수 있다

![](files/CleanShot%202026-03-08%20at%2017.47.37@2x.png)

- 이를 인터럽트 구동식 입출력(interrupt driven input and output)이라고 하며, 1954년 DYSEAC 시스템에서 처음 사용되었다
- 다만 여전히 두 문제가 남아 있다
	- CPU는 인터럽트 신호를 어떻게 감지하는가
	- 중단된 프로그램의 실행 상태를 저장하고 복원하는 방법은 무엇인가


- 인터럽트 신호가 감지되면, 하던 걸 멈추고 인터럽트를 처리해야 한다
- 물론 인터럽트의 우선순위가 지금 실행 중인 프로그램보다 높지 않다면 처리하지 않을 수도 있다

![](files/CleanShot%202026-03-08%20at%2017.52.39@2x.png)

- 인터럽트를 처리할 때는 먼저 중단된 작업 상태를 보존해야 한다
- 그렇다면 인터럽트 처리와 함수 호출이 다를까?


- 일반적인 함수 호출은 단일 스레드 내에서 발생하지만, 인터럽트 처리는 서로 다른 두 실행 흐름을 포함하므로 상태 정보가 더 복잡하다

![](files/CleanShot%202026-03-08%20at%2017.53.38@2x.png)


- 인터럽트 처리 중에 인터럽트가 발생하는 것도 가능하다


![](files/CleanShot%202026-03-08%20at%2017.54.07@2x.png)
![](files/CleanShot%202026-03-08%20at%2017.54.46@2x.png)

- 이는 스택으로 구현할 수 있고, 이 스택은 반드시 커널에 있어야 한다

## 디스크가 입출력을 처리할 때 CPU가 하는 일은 무엇일까?

- 최신 컴퓨터에서는 디스크가 입출력을 처리할 때 CPU 개입이 필요 없다

![](files/CleanShot%202026-03-08%20at%2017.57.33@2x.png)

- 한 스레드가 디스크 입출력을 기다리고 있으면 다른 스레드를 실행한다

![](files/CleanShot%202026-03-08%20at%2017.58.42@2x.png)

- 장치가 점점 발전하면서 자체적인 프로세서와 펌웨어를 지니고 있고, CPU의 도움 없이도 복잡한 작업을 처리할 수 있다
- 또한 자신만의 버퍼나 레지스터를 가지고 있어 이를 활용하기도 한다


- 그렇다면 CPU가 직접 데이터를 복사해야 할까?
- 이를 직접 하는 건 리소스 낭비이므로, CPU 개입 없이 장치와 메모리 사이에 데이터를 전송할 수 있는 방법을 설계했고, 이를 DMA(Direct Memory Access)라고 한다

![](files/CleanShot%202026-03-08%20at%2017.59.45@2x.png)

- CPU가 직접 복사하진 않지만 CPU가 DMA에게 어떻게 데이터를 복사할지 정도는 알려줘야 한다
- 방향은 어디로인지, 얼마나 기록할 건지, 어디 위치에 기록할 건지 등등
- DMA는 버스를 중재하고, 장치를 작동시켜 데이터를 읽고 버퍼에 놓고, DMA가 지정한 메모리 주소에 데이터를 쓴다


- 그러나 가상 메모리와 캐시가 또 문제가 된다
- DMA가 읽은 데이터는 가상 주소와 물리 메모리 주소 중 어디에 저장해야 할까?
- 한 가지 해결책은 OS가 DMA에 필요한 가상 주소와 물리 메모리 주소 사이의 맵핑 정보를 저장하는 것. 이를 이용하면 DMA가 가상 주소 기반으로 직접 데이터를 전송할 수 있다
- 캐시 일관성에 대한 해결책은 상응하는 캐시 데이터를 즉시 메모리에 갱신하여 일관성 문제가 나타나지 않도록 하는 것


- 그렇다면 CPU는 데이터 전송이 완료되었다는 건 어떻게 알 수 있을까?
- 데이터 전송을 완료하면 DMA가 인터럽트를 보낸다

![](files/CleanShot%202026-03-08%20at%2018.03.35@2x.png)

## 파일을 읽을 때 프로그램에는 어떤 일이 발생할까?

- 메모리 관점에서 입출력은 단순히 메모리 복사

![](files/CleanShot%202026-03-08%20at%2018.07.51@2x.png)

- 파일에서 데이터를 읽으면, 프로세스는 대기하는 동안 DMA는 디스크에서 파일을 읽어서 이를 메모리 버퍼로 복사한다
- 사실 입출력 데이터는 OS 내부로 먼저 복사되고, 이후 OS가 프로세스의 주소 공간으로 복사한다
- 즉, OS 복사라는 한 과정이 더 들어가 있는 것
- 이를 우회해서 즉시 프로세스 주소 공간에 복사하고 싶다면 이를 zero copy라고 한다

## 높은 동시성의 비결 : 입출력 다중화

- 유닉스, 리눅스에선 모든 입출력 장치를 파일로 추상화한다
- 파일을 사용하려면 File Descriptor라는 번호를 받는다
- 이 숫자로 파일과 관련된 정보를 얻고, 작업할 수 있다
- 결국 프로그래머는 FD만 잘 처리하면 모든 파일을 처리할 수 있다

![](files/CleanShot%202026-03-08%20at%2018.11.32@2x.png)

```c
if(read(socket_fd1, buff) > 0)
{
    // 첫 번째 처리
    do_something();
}

if(read(socket_fd2, buff) > 0)
{
    // 두 번째 처리
    do_something();
}
```

- 그렇다면 여러 개의 파일 처리를 어떻게 효율적으로 할 수 있을까?
- 블로킹 되어서는 안 된다
- 여기서 핵심은 FD 하나에 대응하는 입출력 장치가 읽을 수 있는지, 쓸 수 있는 상태인지 미리 알 수 없다는 것
- 그런 상태에서 IO 요청을 보내면 스레드가 블로킹된다


- 더 나은 방법은 커널이 필요할 때 응용 프로그램에게 통지하는 것
- 10000개의 FD가 있으니, 읽거나 쓸 수 있는 FD가 생기면 알려줘라~
- 이를 IO Multiplexing이라고 한다


- Linux에는 `select`, `poll`, `epoll`이 있다
	- `select`
		- FD가 1024개를 넘을 수 없다
		- `select`가 호출될 때 대응하는 프로세스 또는 스레드는 블로킹 되었다가 하나라도 가능해지면 깨어난다
		- 여기서 문제는 프로세스가 깨어날 때 어떤 FD가 읽고 쓸 수 있는지 알 수 없어 처음부터 다시 다 확인해야 한다는 것
		- 그래서 대량의 FD를 감시할 때 효율이 떨어진다
	- `poll`
		- 이는 `select`와 유사하며, 감시 가능한 FD가 1024개라는 제한을 해결한 것
		- `poll`도 FD가 늘어날수록 성능이 저하되는 문제가 있다 
	- `epoll`
		- 이를 위해 커널에 필요한 데이터 구조를 만들었다
		- 준비 완료 목록을 따로 만들고, 프로세스를 깨우면서 이 목록을 전달한다

## mmap: 메모리 읽기와 쓰기 방식으로 파일 처리하기

- 메모리 작업은 편하지만 파일 작업은 어렵다
- 메모리 작업하는 것처럼 파일 작업을 할 수는 없을까?
- 가상 메모리는 '가상'이므로 좀 수작을 부려서 파일을 메모리 공간에 사상할 수 있다

![](files/CleanShot%202026-03-08%20at%2018.23.34@2x.png)

- 길이 200 바이트 파일을 600~799 주소 범위에 위치한다고 가정하면, 이 주소 공간에서 바이트 단위로 파일을 처리할 수 있다
- 대응되는 파일이 아직 메모리에 없다면 page fault가 발생한다
- 이후 CPU가 인터럽트를 처리하면 디스크 입출력이 발생하고, 메모리에 파일 데이터가 올라간다
- 메모리를 수정하면 OS는 다시 이를 디스크에 기록한다
- 여전히 디스크 작업은 이루어지지만 모두 OS가 진행하므로 사용자는 이를 몰라도 된다
- 그렇다면 이걸 왜 쓸까? `read`, `write` 있는데?


- `read`, `write` 쓰면 직접 파일 읽고 쓸 수 있다
- 그러나 매번 커널 상태로 데이터 가져와서 다시 사용자 상태로 복사하는 등의 작업이 수반됨 -> 큰 부담
- 반대로 `mmap`은 이런 문제가 없지만 커널은 프로세스 주소 공간과 파일의 사상 관계를 유지하기 위한 특정 데이터 구조가 필요하고, 이 또한 성능에 부담을 가져옴 + 페이지 누락 문제
- 페이지 누락 인터럽트 + 커널마다 구현이 달라 `mmap`이 항상 `read/write`보다 성능이 좋다고 할 수는 없다. 확인해봐야 한다


- `mmap`의 장점은 큰 파일을 다룰 때 이점이 있다
- `read/write`는 파일을 조금씩 나누어 메모리에 적재하고, 일부분 처리가 끝나면 다음 부분 처리하는 등의 방식을 택함
- 전체 파일에 임의로 접근해야 한다면 더 복잡해짐
- 그러나 `mmap`은 가상 메모리 덕분에 파일 크기와 무관하게 작업이 가능하며, 범위나 메모리 이동 등을 전혀 신경 쓰지 않아도 된다. 임의의 위치에 접근할 때도 훨씬 단순해진다
- 다만 역시 성능 이점은 테스트해봐야 알 수 있다

![](files/Pasted%20image%2020260308204518.png)

- 많은 프로세스가 한 파일을 참조하고, 읽기 전용인 경우도 있을까?
- 동적 링크 라이브러리가 대표적인 예시다
- 많은 프로세스가 동적 링크 라이브러리에 의존하므로, mmap으로 해당 라이브러리를 사용하는 모든 프로세스의 주소 공간에 직접 사상할 수 있다
	- 각자 자기 주소 공간에 적재되었다고 생각하겠지만, 실제 물리 메모리에서 이 라이브러리가 차지하는 공간은 한 개 크기뿐이다
