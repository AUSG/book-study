## 캐시. 어디에나 존재하는 것

![](files/CleanShot%202026-03-08%20at%2015.35.38@2x.png)

- 폰노이만 구조
- 명령어, 데이터는 메모리에 저장되어 있어야 함
- 결과 또한 메모리에 저장
- 따라서 프로그램이 실행되면서 CPU와 메모리는 빈번하게 상호작용
- 레지스터가 있긴 하지만 용량이 극히 제한되어 있어 여전히 빈번하게 메모리 접근 필요

![](files/CleanShot%202026-03-08%20at%2015.37.01@2x.png)

- 그렇다면 CPU와 메모리 성능은 얼마나 차이가 날까?
- 점점 속도 차이는 벌어지고 있고, 개선의 여지는 보이지 않음
- 메모리 속도는 보통 CPU의 100분의 1 정도

![](files/CleanShot%202026-03-08%20at%2015.40.45@2x.png)

- 그래서 캐시를 두고, 일부 데이터에 대해 빠르게 접근할 수 있도록 한다
- L1 : 4클럭, L2 : 10클럭, L3 : 50클럭 정도
- 캐시 단계에 따라서 속도는 낮아져도 용량이 증가한다


- 다만, 이 또한 대가를 치러야 한다
- 메모리에 기록할 때 이제 캐시 데이터도 갱신이 필요하다. 불일치 문제 해결 필요
- 이를 해결하는 가장 쉬운 방법은 캐시 갱신할 때 메모리도 갱신하는 것
- 이를 `write-through`라고 한다
- 그러나 캐시 업데이트에서 메모리 접근이 필요하고, 이는 동기 방식임

- CPU가 메모리에 기록할 때는 캐시를 직접 갱신하지만, 반드시 메모리가 갱신될 때까지 CPU가 기다릴 필요는 없다
- 캐시 용량에도 한계가 있으므로, 캐시에서 데이터가 제거될 때 변경되었다면 이를 메모리에 갱신하는 것도 가능하다
- 이를 `write-back`이라고 한다
- 훨씬 복잡하지만 성능적으로 더 좋다.

![](files/CleanShot%202026-03-08%20at%2015.45.17@2x.png)

- 그러나 다중 코어가 등장하면서 더 복잡해졌다
- CPU마다 캐시가 있는데 서로 같은 변수에 접근한다면?
- 이를 가장 쉽게 해결하는 방법은 캐시를 업데이트할 때 다른 코어에도 값이 존재하면 이 또한 갱신하기
- 즉, 이를 위해서는 각 CPU에 어떤 캐시가 있는지 파악이 필요하다
- 최신 CPU에는 고전적인 MESI protocol 같은 다중 코어 캐시 일관성 유지 규칙이 있다.


- 물론 이를 빈번하게 유지하면 당연히 성능 상 손해를 본다


- 이런 문제는 디스크에도 존재한다
- 메모리 접근 속도는 디스크보다 10만 배 가량 빠르다
- 따라서 메모리를 디스크의 캐시로 사용한다


- 보통 메모리는 100%를 다 쓰는 게 아니라 일부 공간이 남아 있다
- 이를 디스크의 캐시로 활용하고, 페이지 캐시의 기본 원리이기도 하다


- 단, 캐시처럼 메모리에만 값을 저장하면 데이터가 날아갈 수 있으므로 동기화 혹은 flush 기능을 제공한다.


- 가상 메모리도 비슷하다
- 각 프로세스는 자체적인 주소 공간을 지니며, 실제 메모리를 쓸 때 이를 물리적인 메모리에 올리고 더 이상 쓰지 않는 메모리를 디스크에 내림으로써 메모리 공간을 확보한다
- 이 모든 일은 OS가 뒤에서 알아서 하므로 프로그래머는 신경쓸 필요가 없다
- 다르게 말하면 프로세스 주소 공간의 데이터는 디스크로 대체될 수 있으며, 메모리 사용률이 높은 경우 디스크 접근이 빈번해질 수 있다.


- CPU는 어떻게 메모리를 읽을까?
- CPU가 읽는 건 모두 가상 메모리 주소
- 이는 실제로 읽을 때 물리 메모리 주소로 변환되어야 하며, 변환이 끝나면 캐시를 검색하고 없는 경우에만 실제 메모리에 접근한다
- 단, 가상 메모리로 인해 실제 데이터는 메모리가 아니라 디스크에 있을 수도 있으므로 디스크 접근이 필요할 수도 있다 


- 대용량의 데이터 저장 문제는 어떻게 할까?
- 여러 장치에 나눠서 저장한다
- 이를 distributed file system이라고 한다

![](files/CleanShot%202026-03-08%20at%2015.54.31@2x.png)

- 이 경우 실제로 분산 파일 시스템에 있는 데이터를 쓸 때는 로컬 디스크에 저장해서 쓰므로, 로컬 디스크를 일종의 분산 파일 시스템의 캐시로 간주할 수 있다
- 혹은 원격 분산 파일 시스템의 데이터를 stream 형태로 로컬의 메모리로 가져올 수도 있다
- Kafka 등이 이렇다


## 어떻게 캐시 친화적인 프로그램을 작성할까?

- 지역성의 원칙은 프로그램이 매우 규칙적으로 메모리에 접근한다는 거다
- 시간적 지역성은 한 번 접근한 메모리를 또 참조할 확률이 높다는 거고
- 공간적 지역성은 인접한 메모리에 접근할 확률이 높다는 것


- 메모리가 필요할 때마다 `malloc` 등으로 할당받으면 실제 메모리는 힙 여러 곳에 흩어져 있을 확률이 높다
- 따라서 메모리 풀을 사용할 수 있는데, 이는 한 번에 큰 메모리를 받으므로 인접한 메모리를 받을 확률이 높다
- 따라서 공간적 지역성이 높다


- 혹은 구조체를 정의하는 방법에 따라 캐시 친화율이 달라질 수도 있다

```c
#define SIZE 100000

struct List
{
    List* next;
    int arr[SIZE];
    int value;
};
```

- `next`, `value` 등을 자주 접근해서 사용하는데 `arr`에 의해 둘 간의 거리가 멀어 캐시 친화적이지 않다


- 위 구조체는 더 최적화될 수 있다
- `next`, `value` 등은 자주 접근하지만 `arr`은 거의 접근하지 않는다고 해보자
- `arr`을 다른 구조체로 두고 포인터를 가지면, 여러 노드에 접근할 때 캐시에 저장할 수 있는 노드가 많아지므로 성능이 올라간다
- 이는 cold와 hot 데이터를 구분하는 것


- 또 유명한 건 2차원 배열에 접근할 때 인접한 메모리부터 접근하도록 순서를 바꾸는 것


- 가장 중요한 건 이러한 최적화는 꼭 프로파일링 후 병목이 되는 경우에만 적용하기

## 다중 스레드 성능 방해자

![](files/CleanShot%202026-03-08%20at%2016.01.57@2x.png)

- CPU가 4바이트 정수에 접근하는데 캐시가 적중하지 않는다면?
- CPU는 단순히 4바이트 정수를 가져오지 않는다
- 공간적 지역성을 위해서 데이터를 가져올 때 인접한 데이터까지 한 번에 가져온다
- 이를 `cache line`이라고 한다
- 캐시와 메모리 상호 작용의 기본 단위는 캐시 라인이다
- 일반적으로 64바이트 정도

```c++
atomic<int> a;

void threadf()
{
    for (int i = 0; i < 500000000; i++)
    {
        ++a;
    }
}

void run()
{
    thread t1 = thread(threadf);
    thread t2 = thread(threadf);

    t1.join();
    t2.join();
}
```

- 위 코드는 싱글 스레드 코드보다 느리다
- 이는 캐시 일관성 때문이다
- 전역 변수 `a`는 각 스레드의 캐시에 각각 저장된다
- 한 CPU에서 `a` 값을 바꾸면 다른 코어의 캐시도 무효화해야 한다
- 서로가 이를 반복하면서 비효율적이게 된다


![](files/CleanShot%202026-03-08%20at%2016.04.22@2x.png)

- 이를 `cache line bouncing` 혹은 `cache ping-pong`이라고 한다.

```c++
struct data
{
    int a;
    int b;
};

struct data global_data;

void add_a()
{
    for (int i = 0; i < 500000000; i++)
    {
        ++global_data.a;
    }
}

void add_b()
{
    for (int i = 0; i < 500000000; i++)
    {
        ++global_data.b;
    }
}

void run()
{
    thread t1 = thread(add_a);
    thread t2 = thread(add_b);

    t1.join();
    t2.join();
}
```

- 이 코드는 두 스레드가 같은 변수를 공유하지 않으므로 괜찮을까?
- 아니다
- 두 스레드는 변수를 공유하지 않지만, 두 변수는 동일한 캐시 라인에 있을 가능성이 매우 높다
- 이를 `false sharing`이라고 하며, 캐시 튕김 문제를 발생시킨다

```c++
struct data
{
    int a;
    int arr[16];
    int b;
```

- 이를 해결하는 쉬운 방법은 두 변수 사이에 사용하지 않는 데이터를 채우는 것

## 봉화희제후와 메모리 장벽

```
스레드 1        스레드 2
 X = 1;          Y = 1;
 a = Y;          b = X;
```
- 스레드 1이 먼저 실행되어 0, 1
- 스레드 2가 먼저 실행되어 1, 0
- 동시에 코드 첫 줄을 실행해서 1, 1
- 근데 실제로는 0, 0 케이스도 있다


- 원래 CPU는 코드 순서대로 명령어를 실행하지 않는다
- 컴파일러는 최적화 과정에서 명령어의 순서를 바꿀 수 있다
- `volatile` 등의 명령어로 명령어 재정렬을 막을 수 있다
- 그러나 CPU가 명령어를 실행할 때도 순서를 바꿀 수도 있다

1. 기계 명령어(opcode)를 가져옵니다.
2. 명령어의 피연산자(operand)가 레지스터에 저장되는 등 이미 준비 완료 상태라면 명령어는 실행 단계에 들어갑니다. 가끔 명령어의 피연산자가 아직 메모리에서 레지스터로 저장되지 않아 준비가 완료되지 않았다면, CPU는 피연산자가 메모리에서 레지스터로 저장될 때까지 기다려야 합니다. 이는 CPU 속도에 비해 메모리의 접근 속도가 매우 느리기 때문입니다.
  3. 데이터가 이미 준비되었다면 명령어가 실행되기 시작합니다.
  4. 실행 결과를 다시 기록합니다.

- 그러나 이는 피연산자가 준비되지 않았다면 CPU가 대기해야 하므로 비효율적이다

1. 기계 명령어를 가져옵니다.
2. 명령어를 대기열(reservation station)에 넣고 명령어에 필요한 피연산자를 읽습니다.
3. 명령어는 대기열에서 피연산자의 준비가 완료될 때까지 대기합니다. 이때 이미 준비가 완료된 명령어가 먼저 실행 단계에 들어갈 수 있습니다.
4. 기계 명령어를 실행하면 실행 결과를 대기열에 넣습니다.
5. 이전 명령어의 실행 결과가 기록될 때까지 기다렸다가 현재 명령어의 실행 결과를 기록합니다. 이는 명령어의 원래 실행 순서에 따라 유효한 결과를 얻기 위한 것입니다.

- 이 과정에서 명령어 실행 순서가 달라질 수 있으며, 이를 `Out of Order Execution (OoOE)`라고 한다
- CPU가 피연산자를 기다리는 동안 파이프라인에 빈 슬롯이 생기고, 이미 준비가 완료된 명령어로 이 공간을 메울 수 있다면 명령어 실행 속도를 끌어올릴 수 있다
- 다만 CPU 외부에서 볼 때는 명령어가 순차적으로 실행되는 것처럼 보인다
- 다만, 이는 전후 관계의 두 명령어가 서로 어떤 의존 관계 없을 때에 한하며, CPU가 이를 지원하는 경우 뒤 명령어를 미리 실행하는 게 가능하다 정도만 알면 된다

![](files/CleanShot%202026-03-08%20at%2016.13.12@2x.png)

- 그렇다면 캐시 입장에선 어떨까?
- 캐시 갱신 및 일관성 유지 작업은 많은 시간을 필요로 한다
- 이를 최적화하기 위해 `store buffer` 등의 대기열을 둔다
- 대기열에 기록하므로 캐시는 즉시 갱신되지 않고, CPU는 다음 명령어를 즉시 실행할 수 있다
- CPU의 기록 작업은 사실 비동기 과정이며 CPU는 기록할 때 캐시나 메모리 갱신을 기다리지 않고 다음 명령어를 실행한다.

```
// a의 초깃값은 0, y의 초기값은 100
a = 1;
b = y;
```
- 위 코드에서 CPU의 코어 A가 `a=1` 실행할 때 데이터는 아직 코어 A 캐시에 반영되지 않았지만 저장 버퍼에 있어, 코어 A는 1이라는 데이터가 캐시에 완전히 갱신되기 전에 `b=y`를 실행할 수 있다
- 이때 `b`의 값은 100이 된다
- 코어 B 입장에서도 `b` 값이 `100`이지만 여전히 `a`는 코어 B 입장에서 초깃값인 `0`일 수도 있다


- 다만 스레드 내부에서는 비순차적인 실행을 볼 수 없다
- CPU 설계가 이는 보장하므로


- 따라서 이런 비순차적인 실행은 자기 자신 이외의 또 다른 코어가 해당 코어를 바라볼 때만 나타나는 현상
- 단일 스레드에서는 이를 신경쓸 필요가 없다
- 이런 메커니즘은 모든 CPU에 있는 건 아니고 CPU에 따라서 다르다
- 잠금을 걸고 프로그래밍하면 문제가 없지만, lock free programming을 하고 있다면 이 문제를 신경써야 한다
- 메모리 장벽을 활용하면 이를 해결할 수 있다
- 이 부분은 다른 수작 부리지 말고 성실하게 순서에 따라서 실행해 다른 코어가 이 코어를 바라보았을 때 순차적으로 실행하는 것처럼 보여야 한다

- `load`, `store`  명령이 있으므로 `LoadLoad`, `StoreStore`, `LoadStore`, `StoreLoad` 등의 4가지 메모리 장벽 유형이 존재한다
- 이는 모두 비순차적인 실행을 금지


- `LoadLoad`
	- CPU가 `Load` 명령 실행할 때 다음 `Load`가 먼저 실행되는 것을 방지한다
- `StoreStore`
	- CPU가 `Store` 명령 실행할 때 다음 `Store`가 먼저 실행되는 것을 방지한다
	- 시스템 캐시 등이 추가되면 쓰기 작업도 비동기로 처리될 수 있지만, 이 장벽을 쓰면 다른 코어에서 보기에 변수의 갱신 순서와 코드 순서가 일치하게 된다
- `LoadStore`
	- CPU가 `Load` 명령 실행할 때 다음 `Store`가 먼저 실행되는 것을 방지한다
	- 쓰기 작업이 더 무거운데 어떻게 `Load`보다 먼저 실행될 수 있을까?
	- `Load` 명령이 캐시 미스되면 `Store` 명령이 미리 실행될 수도 있다
- `StoreLoad`
	- CPU가 `Store` 명령 실행할 때 다음 `Load`가 먼저 실행되는 것을 방지한다
	- 가장 무거운 메모리 장벽이다
	- 쓰기 작업이 더 무거운데, 이를 사용하면 쓰기 명령이 얼마나 복잡하고 얼마나 오래 걸리든 이 기간 동안 CPU는 관련 없는 읽기 작업을 미리 실행할 수 없고, 메모리 장벽 이전에 쓰기 작업이 모든 다른 코어에서 반드시 확인 가능해야 한다
	- 즉, 이를 활용하면 메모리 장벽 이전의 변수를 다른 코어가 장벽 이후에 읽으면 그 값이 반드시 최신 값이라는 걸 보장할 수 있다
	- `StoreStore` 장벽은 이를 보장하지 않는다
	- 따라서 이는 동기 작업이다
	- 아래 코드가 0, 0이 나오지 않음을 보장해주는 유일한 장벽이다

```
스레드 1        스레드 2
 X = 1;          Y = 1;
 StoreLoad();    StoreLoad();
 a = Y;          b = X;
```


- 획득 해제 의미론
- 다중 스레드 프로그래밍 시 두 가지 문제에 직면한다
	- 공유 데이터에 대한 상호 배타적인 접근
	- 스레드 간 동기화 문제
- 획득 해제 의미론(acquire-release semantics)은 두 번째 문제를 해결하는데 쓰이지만 공식적인 정의는 없다

![](files/CleanShot%202026-03-08%20at%2016.29.33@2x.png)

- 획득 의미론은 메모리 읽기 작업에 대한 것
	- `Load` 뒤에 있는 모든 메모리 작업은 이 `Load` 작업 이전에는 실행이 불가능하다
	- `LoadLoad`, `LoadStore`

![](files/CleanShot%202026-03-08%20at%2016.29.38@2x.png)

- 해제 의미론은 메모리 쓰기 작업에 대한 것
	- `Store` 앞에 있는 모든 메모리 작업이 이 `Store` 작업 이후에는 실행 불가능하다
	- `StoreStore`, `LoadStore`

- 흥미로운 것은 획득-해제 의미론을 얻기 위해 `StoreLoad`처럼 매우 무거운 메모리 장벽이 필요하지 않고, 나머지 세 가지 메모리 장벽만 사용하면 된다

![](files/CleanShot%202026-03-08%20at%2016.30.43@2x.png)

- 한 곳에서는 해제 의미론을 사용해 모든 메모리의 읽기 및 쓰기 작업을 봉화 신호 설정한 후 하지 않도록 하고
- 한 스레드에서는 획득 의미론을 사용해 모든 메모리 읽기와 쓰기 작업을 봉화 신호 이전에는 하지 않도록 한다
- 이렇게 하면 동기화 문제를 해결할 수 있다

```
std::atomic<bool> is_enemy_coming(false);
int enemy_num = 0;

// 주유왕 스레드
void thread_zhouyouwang()
{
    enemy_num = 100000;

    // 해제 장벽
    std::atomic_thread_fence(std::memory_order_release);

    is_enemy_coming.store(true, std::memory_order_relaxed);
}

// 제후 스레드
void thread_zhuhou()
{
    int n;

    if(is_enemy_coming.load(std::memory_order_relaxed))
    {
        // 획득 장벽
        std::atomic_thread_fence(std::memory_order_acquire);

        n = enemy_num;
	}
}
```

![](files/CleanShot%202026-03-08%20at%2016.32.49@2x.png)

- 아키텍처 별로 재정렬 지원이 다르다
- Alpha, ARMv7, POWER 등 CPU에서는 거의 모든 유형의 재정렬을 볼 수 있는데 이를 weak memory model이라고 한다
- x86은 StoreLoad 재정렬 하나만 있다. 따라서 x86을 strong memory model이라고 한다
- 모든 유형의 CPU는 StoreLoad 재정렬을 가지고 있다


- 잠금 없는 프로그래밍을 할 때만 메모리 재정렬을 신경쓰면 된다
- 잠금을 사용하면 메모리 재정렬 문제를 자동으로 처리한다.
![](files/CleanShot%202026-03-08%20at%2016.34.47@2x.png)

- 다중 스레드에서는 일반적으로 두 종류의 잠금이 있다
	- 하나는 상호 배제로, 최대 스레드 하나만 상호 배제를 보유할 수 있고, 다른 스레드는 대기 상태로 진입한다
	- 다른 건 잠금을 요청하는 스레드가 대기 상태로 빠지는 게 아니라 이 여부를 반복적으로 확인한다. 이를 spinlock이라고 한다
- 어떤 종류든 결국 잠금을 획득하지 못하면 다음 코드로 넘어가지 못한다는 것


- 잠금 없는 프로그래밍은 어떤 스레드가 공유 자원을 사용하고 있을 때 대기하지 않고 일단 다른 필요한 작업으로 넘어가는 것
- 즉, 성능 향상보다는 대기 없이 어떤 일을 하도록 하는 것에 가치를 두고 있다
- 실시간 요구 사항이 높은 시스템에서는 매우 중요하지만 매우 많고 복잡한 리소스 경쟁 문제와 ABA 문제를 처리해야 하며, 훨씬 복잡하다
- 그러나 어떤 상황에서는 적은 수의 원자적 작업으로 이를 구현 가능하며, 이 경우 성능적인 향상을 얻을 수도 있다


- 그렇다면 왜 명령어 재정렬을 해서 이런 문제를 만드는 걸까?
- 하드웨어 입장에서는 이는 성능 향상에 큰 도움이 된다
- 그러나 이는 매우 어렵고, 많은 종류의 버그를 일으키는 주요 원인
- 당연히 하드웨어가 성능도 올리면서 이러한 문제도 막아주며 좋지만 당연히 어렵다
- 물론 이런 기술들도 언젠간 역사의 뒤편으로 사라질 수도 있다
