# Chapter 1 - 프로그래밍 언어부터 프로그램 실행까지, 이렇게 진행된다

**추상화의 과정**

CPU가 이해 가능한 명령(0101의 연속) → Instruction Set(어셈블리어) → 고급 프로그래밍 언어 

**해석의 과정**

고급 프로그래밍 언어 → 트리 형태를 지닌(재귀 원리) 구문 트리로 변환 → 표준 명령어 집합(CPU 시뮬레이션 프로그램인 ‘가상머신’에서 돌아갈 수 있음) → CPU가 이해 가능한 명령

**컴파일 과정**

자연어로 된 프로그래밍 언어 → 토큰으로 분해(어휘 분석 - Lexical Analysis) → 토큰의 의미를 해석하고 파싱하는 과정을 통해 AST 생성(구문 분석 - Syntax Analysis) → 구문 트리에 이상 없는지, 변수가 뭘 뜻하는지 등 깊게 의미 분석(Semantic Analysis) → 구문 트리 기반으로 중간 코드 생성 

**링커**

링커는 컴파일러가 생성한 대상 파일 여러 개를 하나로 묶어 하나의 최종 실행 파일을 생성하는 역할

- 재배치(Relocation): 컴파일러가 소스 파일을 컴파일 하는 시점에는 메모리 주소에 대해 알 수 없으므로, 표시만 해두면 이후에 링커가 표시들을 한 군데 모아 실행 파일을 생성하는 과정에서 함수의 정확한 주소를 확인 후 리졸빙
- 심벌(Symbol): 전역 변수(Global Variable)와 함수(Function)의 이름을 포함하는 모든 변수의 이름
    - 모듈 간 종속성이 올바르게 설정되어 있는지(인터페이스 구현이 종속된 모듈에서 사용 가능한지 확인)
    - 참조하고 있는 외부 심벌에 대한 실제 구현이 모든 모듈에 딱 하나 있는지 확인(Symbol Resolution)
- 심벌 테이블(Symbol Table): 소스 파일마다 외부에서 참조 가능한 심벌이 뭐가 있는지, 어떤 외부 심벌을 소스 파일이 참조하고 있는지를 기록한 테이블
- 심벌 해석: 각 대상 파일에서 사용할 외부 심벌이 심벌 테이블에서 유일한 정의를 발견 가능한지 확인

**정적 라이브러리와 동적 라이브러리**

컴파일의 경우 소스 파일마다 단독으로 진행되고, 이것들이 하나로 합쳐져 실행 파일이 된다

- 정적 라이브러리(Static Library): 공통으로 사용되는 대상 파일들을 패키지로 묶고, 구현된 모든 함수의 선언을 포함한 헤더 파일(header file)을 제공하는 것
    - 다시 컴파일할 필요 없이 링크 과정에서 그대로 실행 파일에 복제되기 때문에 컴파일 속도가 빨라짐
    - 이 과정을 정적 링크(static linking)이라고 함
    - 실행 파일에는 특수한 심벌인 `_start`가 있어 cpu는 이 심벌의 주소를 바탕으로 실행에 필요한 기계 명령어를 찾음. 이 명령을 실행한 후 main 함수 실행 시작
    - 라이브러리를 실행 파일에 직접 복사하기 때문에 모든 실행 파일에 중복 파일이 생겨 무거워진다는 단점이 존재함
- 동적 라이브러리(Dynamic Library):
    - 공유 라이브러리(Shared Library)라고도 불림, `.so` 확장자 사용, 접두사로 `lib` 사용
    - 실행 파일에 동적 라이브러리 이름, 심벌 테이블, 재배치 정보 등 필수 정보만 실행 파일 내 저장
    - 동적 링크(Dynamic Linking)이 일어날 때 그때 사용됨. 방식은 두 가지
        1. 프로그램이 메모리에 적재(loading)될 때
        2. 프로그램이 먼저 실행된 후, 런타임에서 코드가 직접 동적 링크를 실행하는 경우
    - 장단점
        - 장점:
            - 메모리 적재, 디스크 저장 리소스 아끼기
            - 동적 라이브러리 코드 수정에 유리, 버그 수정 유리
            - 여러 언어를 혼합하여 개발할 때 매우 유용
                **왜 여러 언어를 혼합할 수 있나?**
                
                **1) API가 아닌 ABI를 통한 통신**
                
                동적 라이브러리는 ABI를 통해 로딩됩니다. 언어가 달라도 ABI 규약만 맞으면 서로 호출할 수 있습니다.
                
                - API (소스 코드 수준의 인터페이스): 소스 코드에서 함수를 호출하는 방식 (예: 함수 이름, 매개변수 타입)
                - ABI (바이너리 호출 규약 수준의 인터페이스): 컴파일된 바이너리 수준에서 함수를 호출하는 규약 (예: 레지스터 사용법, 스택 구조)
                
                예를 들어, Python에서 작성한 코드가 C로 작성된 라이브러리를 호출하고, 그 라이브러리가 다시 Rust로 작성된 함수를 호출할 수 있습니다. 이는 언어 간 직접 통신이 아니라 "바이너리 계약"을 통한 통신입니다.
                
                대부분의 프로그래밍 언어는 C ABI를 지원(Python → CPython C API, Rust → extern "C", Go → cgo, Java → JNI)
                
                즉, C ABI는 여러 언어 간의 "최소 공통 분모" 역할 수행. 동적 라이브러리는 보통 이 C ABI를 사용하여 인터페이스를 노출
                
                **2) 정적 링크 vs 동적 링크: 빌드 시스템 관점**
                
                정적 링크의 제약**:** 컴파일러 버전, 링커 옵션, 표준 라이브러리, 런타임 등이 모두 맞아야 함
                
                동적 링크의 유연성**:** ABI만 유지하면 됨(내부 구현은 자유롭게 변경 가능, 다른 컴파일러로 빌드해도 사용 가능)
                
        - 단점:
            - 적재되는 시간 또는 실행 시간에 링킹되어야 하므로 성능 상 불리
            - 특정 메모리 주소와 독립적으로 동작하기 때문에 위치 독립 코드(position-indenpendent code)로 불리우고, 메모리에 단 하나의 복사본만 존재하고 해당 코드는 여러 프로세스가 공유할 수 있기 때문에 동적 라이브러리의 코드는 임의의 메모리 절대 주소(absolute address)를 참조할 수 없음
                - 동적 라이브러리의 변수를 참조할 때 좀 더 ‘간접적인 접근’을 해야 하지만, 동적 라이브러리를 사용해 얻는 이점이 훨씬 커서 사용한다
            - 종속된 동적 라이브러리를 제공하지 않거나 그 버전이 호환되지 않을 경우 프로그램 실행이 안 됨
    - 재배치
        - 메모리 주소를 확정할 수 없는 변수를 발견할 때마다 .relo.text에는 해당 명령어를 저장하고 .relo.data에는 해당 명령어와 관련된 데이터를 저장
