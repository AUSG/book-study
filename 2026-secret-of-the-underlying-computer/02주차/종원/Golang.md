
## Golang 메모리

[함께 보는 Go 메모리](https://siyul-park.github.io/system/memory-in-golang/)

- TCMalloc을 기반으로 한 자체 메모리 관리를 수행함
- 메모리를 68가지의 다른 크기의 블록들로 나누고, 같은 크기의 블록끼리 관리함
- 또한 고루틴(스레드)에서 잠금 없이 메모리를 빠르게 할당하고 해제하기 위한 로컬 스레드 캐시도 존재
	- 크기와 수명이 정해진 객체들을 저장하는 `stackcache`
	- 16B 보다 작은 객체를 저장하기 위한 `tiny`
	- 32KB보다 작은 값을 저장하기 위한 `alloc`

- 재밌는 건 golang의 스택은 사실 메모리 할당자가 힙에서 할당해준다.
	- 힙에서 영역을 할당받으면, 내부에서 자기 혼자 그걸 스택처럼 쓰는 것

- 그렇다면 golang은 왜 자체 메모리 관리를 수행하는가?
- 슬라이스(가변 배열) 생성하고, 문자열 연산할 때마다 OS에게 메모리 할당 요청하면 시스템 콜 날아감 -> 너무 느림
- 수천 개의 고루틴이 메모리 할당을 요청한다? -> XX

- 그래서 Go도 다른 언어처럼 OS에게 큰 덩어리로 메모리를 요청함 -> 보통 64MB씩
	- 정확히는 64MB를 예약하고
	- 실제로 필요할 때 4MB씩 청크로 요청하고 (시스템 콜)
	- 실제로 메모리에 쓸 때 OS는 물리 메모리에 씀
- 이 덩어리를 Arena라고 함
- 근데 64MB는 너무 큼 -> Arena를 8KB짜리 Page로 나눔
- 즉, 1 Arena = 8192 Page, runtime은 각 Page의 상태를 추적 및 관리
- 근데 여전히 8KB는 대부분 할당에 너무 크므로, 이걸 더 쪼갬 -> Span

- Span은 단일 크기의 객체를 담기 위한 하나 이상의 연속된 페이지
- 32 바이트 객체가 많이 필요하다면, Page 하나(8KB)를 32바이트 Span으로 쪼개서, 256개의 슬롯으로 나눔
- 각 슬롯은 32바이트 객체 하나 담을 수 있음
- span은 연속된 동일 사이즈의 블록이므로 검색할 필요도 없음. 그냥 인덱스로 찾고 관리 가능

- Span은 한 객체만 담는다
- 따라서 여러 크기의 Span이 필요
- 8 bytes부터 32KB까지 범위의 68개 size class를 정의해서 쓴다

- 32KB보다 큰 객체를 위한 Large 클래스가 있음
- 고정된 객체 크기가 없고, span 크기도 없음
- 객체를 담을 수 있을 만큼 페이지 할당 받고, 슬롯으로 분할하지도 않음

- 그렇다면 bool, tiny int 같이 아주 작은 객체는? 이 친구들에게는 8바이트도 낭비
- 따라서 여러 개의 tiny object를 16바이트 블록에 담는 tiny allocator가 따로 있음

- GC하려면 포인터를 따라가며 스캔이 필요함
- 객체가 많으면 이걸 매번 하는 건 무거우니 스캔이 필요한 객체들과 아닌 객체들을 따로 관리함

- 그렇다면 golang에서 스레드 로컬 캐시, 메모리는 어떻게 돌아가나?
- 기본적으로 고랭은 프로세서 수만큼 P가 있고, 각 고루틴이 P에 할당되는 형태

![](files/Pasted%20image%2020260227230706.png)

- 따라서 한 P에서는 한 고루틴만 동시에 실행됨
- 따라서 P마다 자체 `mcache`를 가지고, `mcache`는 각 span class마다 하나씩의 span으로 이루어진 private 컬렉션
- 따라서 P끼리 자원을 공유하지 않고, P에서는 한번에 한 고루틴만 실행되므로 경합이 전혀 발생하지 않음. fast path

- 그러나 `mcache`의 `span`을 다 쓰면 새로운 `span`을 받아야 함
- 이때 `mcentral`이 등장
- span class마다 `mcentral`이 있고, 공유 `span` 풀을 관리함
- 락이 필요하지만, 매우 짧고, span class 마다 각각 `mcentral`을 가지므로 경합이 심하지 않음

- `mcentral`에도 더 이상 남은 `span`이 없으면 `mheap`은 OS로부터 새 페이지를 할당 받고, 초기화하는 작업을 거침
- 또한 `mheap`은 전역 객체이며, 하나만 존재 -> 전역 락이 필요함
- 그러나 이 path를 자주 호출하지 않으므로 경합이 심하지 않음. slow path
