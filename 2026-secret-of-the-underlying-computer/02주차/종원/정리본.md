## 메모리의 본질, 포인터와 참조

메모리 셀에는 0 또는 1이라는 bit를 저장할 수 있다.
더 많은 정보를 표현하기 위해서 비트 8개를 묶어서 한 바이트를 만들었다
바이트마다 번호를 붙임 -> 이를 메모리 주소라고 함
메모리 주소 하나로 여러 개의 메모리 셀을 찾을 수 있으며, 이를 addressing이라고 한다.

바이트만으로도 정보를 표현하기에는 부족하므로 일반적으로 4바이트를 묶어 정수를 표현한다.
다양한 정보를 묶어서 하나를 표현하기 위해서 구조체 또는 객체 등을 만들기도 한다.


같은 값을 가리키기 위해 매번 값을 복사하면 메모리를 너무 많이 차지한다
따라서 메모리 주소를 가리키는 변수를 또 만들 수 있다
우리가 흔히 아는 포인터
사실 포인터가 주소라는 건 어셈블리어 수준에서 그렇고, 고급 언어에서는 포인터도 하나의 변수에 불과하다.
단지 이 변수에 저장하기 적합한 값이 메모리 주소일 뿐

포인터라는 건 PL/I 언어에서 처음 등장했는데, linked list의 처리 능력을 높이기 위해서 등장함

Multics는 PL/I 언어로 구현된 운영 체제
고급 언어를 이용해서 구현된 최초의 OS
이 프로젝트에 참여했던 사람들이 Unix와 C 언어를 만들었다


자바, 파이썬 등의 언어는 포인터가 존재하지 않아 메모리 상의 데이터를 직접 조작할 수 없다
그러나 C 언어 등은 메모리 주소를 추상화하지 않아 프로그래머가 더 할 수 있는 게 많다
포인터 개념이 있으면 메모리 같은 하드웨어를 직접 조작할 수 있다

단, 메모리를 직접 조작할 수 있으므로 실수하지 않게 조심해야 한다
물론 포인터가 늘 필요한 건 아니다. 자바, 파이썬 등의 언어도 성공했으니까
포인터가 없어도 포인터 대신 사용할 수 있는 추상화된 무언가가 있기 때문 -> 참조(reference)


참조를 쓰면 구체적인 메모리 주소를 얻을 수는 없다.
따라서 메모리 주소에 얼마를 더하고 빼는 등의 연산은 불가능
그러나 데이터를 복사할 필요는 없으므로 포인터와 동일한 효과를 얻을 수 있다
포인터를 한 번 더 추상화한 게 참조


사실 포인터가 사용하는 주소도 실제 물리 주소는 아니다. 가상 메모리 상의 주소

## 스택 영역: 함수 호출은 어떻게 구현될까?

함수 호출 및 반환은 스택 구조로 이루어진다
함수 호출에 필요한 정보들도 여기에 저장되는데, 이를 stack frame, call stack이라고 한다
이는 프로세스 스택 영역에 생성된다

![](files/Pasted%20image%2020260221173049.png)

함수 호출은 사실 제어권의 이전이다
이에는 두 가지 정보가 필요하다
- 반환(return) : 어디에서 왔는지에 대한 정보
- 점프(jump) : 어디로 가는지에 대한 정보

이런 정보는 어떻게 얻고 유지할까? 스택 프레임의 도움이 필요하다

![](files/Pasted%20image%2020260221173324.png)

```
call 0x400540
```

call 명령을 부르면 함수가 실행된다
call은 사실 함수로 점프하는 것 외에도 call 명령어 다음에 위치한 주소를 함수 A의 스택 프레임에 넣는다

![](files/Pasted%20image%2020260221173409.png)

함수 B가 실행된다

![](files/Pasted%20image%2020260221173426.png)

함수 B는 마지막 명령인 `ret`까지 실행되고, `ret`가 실행되면 함수 A의 스택 프레임에 저장된 반환 주소로 점프한다.


그렇다면 매개변수나 반환값은?
보통 이를 레지스터로 수행한다.

B를 호출하기 전에 레지스터에 값을 저장해두면 이에 접근할 수 있고, 함수를 반환하기 전에 레지스터에 저장해두면 A에서 이에 접근할 수 있다

그러나 레지스터 수는 제한되어 있는데 매개변수 수가 레지스터보다 많으면? 나머지 매개변수는 스택 프레임에 넣는다

![](files/Pasted%20image%2020260221173553.png)


그렇다면 지역 변수는? 이 또한 레지스터에 저장할 수 있지만 레지스터 수보다 많을 때는 마찬가지로 스택 프레임에 저장한다.

![](files/Pasted%20image%2020260221173622.png)

그렇다면 각 함수에서 레지스터를 활용하는데 함수 A가 레지스터에 기록한 값을 B가 덮어 쓰지는 않을까?

레지스터에 지역 변수를 저장하기 전에는 레지스터에 저장되어 있던 초기값을 다른 곳에 백업하고, 레지스터를 사용하고 나면 다시 복원해둬야 한다.
이 백업도 스택 프레임에 이루어진다.


![](files/Pasted%20image%2020260221173728.png)

![](files/Pasted%20image%2020260221173738.png)

이러한 이유로 지역 변수가 너무 크거나, 함수 호출 단계가 너무 많으면 안 된다.


그렇다면 매개변수가 너무 많아서 다 레지스터에 저장하지 못한다 등의 정보는 어떻게 알 수 있을까?

-> 애초에 OS 혹은 아키텍처에 따라 미리 정해져 있다. 첫 몇 개는 레지스터에, 이후는 스택에 등등
즉, 런타임이 아니라 컴파일 시점에 미리 다 결정되어 있음

스택 영역의 증가와 감소는 어떻게 구현할 수 있고, 이 구현은 누구 책임일까?

-> 기본적으로 함수 호출, 반환 등의 스택 변환은 위에서 말한 것처럼 `call`, `ret` 등에서 자동으로 처리
정확히는 위 명령은 CPU가 특수 레지스터의 값을 읽고 쓰도록 하는 용도이지, 이런 걸 잘 활용해서 레지스터에 값을 넣고 읽도록 하는 건 컴파일러의 책임

또한 ABI에 따라서 특정 규칙을 요구하는 경우가 있는데 (함수 엔트리에서 (%rsp - 8)이 16바이트 정렬된다거나) 등 이 또한 컴파일러가 충족해줌

## 힙 영역: 메모리의 동적 할당은 어떻게 구현될까?

함수 호출이 완료되면 기존 스택 프레임 정보는 더 이상 사용되지 않고, 스택은 줄어든다
따라서 함수 A -> B 호출하는 경우 B 스택 프레임에 저장되어 있던 내용은 무효화되므로 어떤 가정도 하면 안 된다.
즉, 함수 B의 지역 변수의 포인터를 쓴다거나 등등을 해서는 안 된다.

또한 지역 변수의 수명 주기는 함수 호출과 동일하므로 메모리 할당, 반환 문제를 신경 쓸 필요가 없지만 반대로 함수를 뛰어넘고 이 변수를 쓰는 게 불가능하다.

근데 사실 이게 go는 상관이 없습니다

```go
package main

import "fmt"

func main() {
	a := B()	
	fmt.Printf("%d\n", *a)
}

func B() *int {
    x := 42
    return &x
}
```

```
❯ go build -gcflags="-m" main.go
# command-line-arguments
./main.go:10:6: can inline B
./main.go:6:8: inlining call to B
./main.go:7:12: inlining call to fmt.Printf
./main.go:7:12: ... argument does not escape
./main.go:7:21: *a escapes to heap
./main.go:11:5: moved to heap: x
```

지역 변수가 외부로 유출되는 것을 보고 `escapes to heap`이 발생 -> 알아서 변수를 힙으로 옮겨줘서 이런 거 신경 쓰지 않고 쓸 수 있음
그러나 힙 메모리 할당이 발생하므로 기피하는 게 좋음

C 언어 등에서는 이런 걸 수동으로 하기 위해 `malloc`, `new` 등으로 힙 영역에 메모리를 요청

![](files/Pasted%20image%2020260221181212.png)

이런 메모리 할당자를 직접 구현한다면, 크게 두 문제를 풀면 된다
- `malloc` : 메모리를 요청하면 빈 공간을 찾아서 반환
- `free` : 메모리 영역의 사용이 완료되면 이를 메모리 영역에 반환

우리는 두 가지를 신경 써야 한다
- 빠르게 메모리를 반환할 수 있어야 한다
- 메모리 사용률을 극대화하기 위해서 잘 배치해야 한다

이를 위해서는
1. 어떤 메모리가 할당되었고, 비어 있는지 알 수 있어야 하고
2. 후보군이 여러 개라면 무엇을 할당해줄 것이고
3. 여유 메모리 공간이 크다면, 남은 메모리 공간은 어떻게 할 것인지
4. 반환된 메모리는 어떻게 처리할 것인지


보통 1번은 linked list로 많이 푼다
근데 연결 리스트는 메모리 할당이 필요하다

따라서 연결 리스트와 메모리 사용 정보를 메모리 조각 자체에 같이 저장해야 한다
메모리 조각은 할당 또는 해제된 전체 메모리 조각
여기에는 다음 노드가 어디에 있는지 알려주는 포인터가 없지만, 메모리 사용 정보로 다음 노드 위치를 유추하는 건 가능하다
- 해당 메모리 조각이 비어 있는지 flag
- 해당 메모리 조각의 크기를 기록한 숫자

단일 메모리 할당에 최대 크기는 2GB고, 메모리 정렬이 필요 없다고 해보자
2GB이므로 31 비트로 조각 크기를 기록하고, 나머지 1비트로 flag를 기록할 수 있다

![](files/Pasted%20image%2020260221181628.png)

이를 header라고 하며, 메모리 조각의 사용 정보를 저장
할당 가능한 메모리 조각은 payload라고 하며, `malloc` 호출하면 이 메모리 주소가 반환된다.

이렇게 구성하면 헤더 크기는 고정이고, 메모리 사이즈는 헤더에서 얻을 수 있으므로 다음 노드 시작 주소를 알 수 있다

![](files/Pasted%20image%2020260221181757.png)

연결 리스트처럼 메모리 할당자에도 tail node를 알려주는 특수한 표시(sentinel)가 필요하고, 이를 위해 마지막 4바이트를 사용한다.


그렇다면 메모리를 어떻게 할당해야 할까?

![](files/Pasted%20image%2020260221181934.png)

- 최초 적합 방식
	- 가장 먼저 요구사항을 만족하는 항목 반환. first fit
	- 단순하지만 앞 부분에 작은 메모리 조각이 많이 남고, 시간이 흐를수록 더 많은 여유 메모리 조각을 탐색해야 한다

![](files/Pasted%20image%2020260221181928.png)

- 다음 적합 방식
	- 최초 적합 방식과 비슷하지만 항상 처음부터 찾는 게 아니라 적합한 메모리 조각이 마지막으로 발견된 위치에서 시작한다
	- 이론적으로 최초 적합 방식보다 빠르게 메모리 조각을 탐색할 수 있다
	- 단, 메모리 사용률은 최초 적합 방식보다 별로인 거로 밝혀졌다

- 최적 적합 방식
	- best fit
	- 모든 후보를 찾고, 크기가 가장 작은 조각을 반환
	- 당연히 메모리는 더 잘 활용하지만 느리다


이제 메모리를 할당하자
헤더를 제외하고, 페이로드 시작 메모리 주소를 반환한다

그러나 메모리 사이즈가 많이 여유가 있다면?
조각 전부를 할당하면 메모리가 낭비되고, 내부 단편화가 발생한다

![](files/Pasted%20image%2020260221182058.png)

그래서 이런 경우 여유 메모리 조각을 두 개로 분할하고, 앞 부분은 할당 후 반환하고 뒷부분은 새로운 여유 메모리 조각으로 사용한다.


메모리 해제는 비교적 간단하다
`malloc`에서 얻은 주소를 반환하면, 반대로 헤더의 주소를 알 수 있다.
메모리 할당 flag만 바꿔주면 된다
이게 `free`에서 메모리 조각 크기는 없어도 되는 이유다.

단, 메모리를 할당 해제할 때 인접한 곳에 여유가 있는 메모리가 있을 수도 있다
이 경우 두 조각을 병합하면 나중에 메모리를 더 잘 활용할 수 있다

![](files/Pasted%20image%2020260221182223.png)

## 메모리를 할당할 때 저수준 계층에서 일어나는 일

![](files/Pasted%20image%2020260221184656.png)

CPU에는 privilege level이 존재한다
0이 특권이 높다

보통 시스템은 0, 3만 쓰는데 0이 kernel mode 3이 user mode다


CPU가 OS 코드 실행할 때는 커널 상태에 놓인다
모든 명령어를 실행할 수 있고, 모든 주소 공간에 접근할 수 있고, 제한 없이 하드웨어에 접근할 수 있다

반면에 프로그래머가 작성한 코드를 실행할 때는 유저 모드다
여러 제한을 받는데, 특히 특정 주소 공간에는 절대 접근할 수 없다
또한 특정 명령어는 실행할 수가 없다


그렇다면 응용 프로그램이 OS 서비스를 요청해야 할 때는 어떻게 해야 할까?
system call을 날리면 된다
이걸 날리면 OS가 대신 파일 입출력, 네트워크 통신 등을 처리해준다

![](files/Pasted%20image%2020260221184959.png)


system call은 OS마다 완전히 다르며, 이걸 직접 쓰면 특정 OS에서만 프로그램이 실행된다
이를 감추기 위해서 표준 라이브러리를 제공하며, 표준 라이브러리는 유저 모드에서 실행된다

![](files/Pasted%20image%2020260221185042.png)

프로그램의 힙 영역이 부족하면 어떻게 할까?
![](files/Pasted%20image%2020260221185125.png)

힙 영역에는 최상단을 가리키는 `brk`라는 변수가 있는데, 이를 위로 이동해서 힙 영역을 확장한다
OS는 `brk`라는 전용 시스템 호출을 제공하는데 이걸 이용해서 힙 영역의 크기를 조절할 수 있다
`mmap` 등도 같은 목적으로 쓸 수 있지만 `mmap`이 좀 더 구조가 유연하다

중요한 건 이런 시스템 호출이 있어 힙 영역이 부족하면 OS에게 힙 영역을 늘려달라고 요청해 여유 메모리를 늘릴 수 있다는 것

1. malloc를 호출해 메모리 할당을 요청. malloc는 표준 라이브러리다
2. malloc은 여유 메모리 조각을 찾고, 이를 할당한다. 여기까지는 유저 모드에서 실행된다
3. 여유 메모리를 찾지 못하면 `brk` 등을 통해 힙 영역을 늘려달라고 요청한다. `brk`는 운영 체제의 일부이므로 커널 상태에 놓인다.

![](files/Pasted%20image%2020260221185326.png)

사실 시스템은 가상 메모리를 쓰니, malloc이 받아오는 메모리는 가상 메모리다. 실제 물리 메모리가 전혀 할당되지 않은 상태
그렇다면 물리 메모리는 언제 할당되는가? 실제로 할당된 메모리가 쓰일 때 할당된다
이때 가상 메모리가 아직 실제 메모리와 연결되어 있지 않으면 page fault가 발생할 수 있다
OS는 이를 감지하고, 페이지 테이블을 수정해 물리 메모리를 할당해준다
즉, 프로그래머 입장에서는 애초에 메모리가 할당되어 있던 것처럼 보인다.

malloc은 메모리의 2차 할당에 불과하며, 할당받는 것도 가상 메모리다
이 과정은 다 유저 모드에서 처리된다
가상 메모리를 쓸 때는 실제 물리 메모리를 할당 받아야 하며, OS만 실제 물리 메모리를 할당할 수 있다.

즉, 정리하면 아래와 같다.

1. `malloc`이 여유 메모리 조각을 검색하고, 찾으면 할당
2. 못 찾으면 `brk` 시스템 콜을 호출해 힙 영역을 확장해서 찾는다
3. `malloc`이 `brk` 호출하면 커널 상태로 전환되는데, 이때 OS의 가상 메모리 시스템이 힙 영역을 확장한다. 다만 이때 확장되는 것도 가상 메모리에 불과하며, 아직 실제 물리 메모리가 할당되지 않았을 수 있다
4. `brk` 실행이 끝나면 `malloc`으로 제어권이 돌아오고, 다시 유저 모드로 전환된다. 확장된 걸 바탕으로 여유 메모리 조각을 찾아 반환한다
5. 실제로 요청된 메모리를 읽거나 쓰면 시스템 내에서 page fault interrupt가 발생하고, CPU는 커널 모드로 전환되어 OS가 실제 메모리를 할당하고, 페이지 테이블을 수정하고, 다시 유저 모드로 돌아간다

![](files/Pasted%20image%2020260221185719.png)

## 고성능 서버의 메모리 풀은 어떻게 구현될까?

malloc은 일종의 범용적인 프로그램
따라서 빈번하게 메모리를 요청하고 할당하는 경우에는 적합하지 않다
대신 특정 상황을 위해 자체적으로 메모리 할당 전략을 구현할 수 있고, 이를 **메모리 풀 기술**이라고 한다.


malloc은 표준 라이브러리지만, 메모리 풀은 응용 프로그램의 일부
메모리 풀은 보편적이지 않고, 특정 상황에 맞춰서 최적화된 것


메모리 풀은 자체적으로 큰 메모리 조각을 요청하고, 그 위에서 메모리 할당 및 해제를 관리해 표준 라이브러리와 OS를 우회한다
또한 사용 패턴에 따라 추가적인 최적화도 구현한다.
사용자의 요청을 처리할 때마다 여러 객체가 생성되었다가 해제된다면 미리 이런 객체를 생성해두는 것

![](files/Pasted%20image%2020260223185015.png)

여러 크기의 메모리를 할당하기 위해 여유 메모리 조각 관리가 필요
두 배 크기로 관리하면 vector 컨테이너 관리하는 것처럼 malloc을 너무 빈번하지 않게 호출하도록 할 수 있다

free처럼 메모리 조각 해제 기능은 제공하지 않고, 요청 처리가 완료되면 한 번에 전체 메모리 풀을 해제
이렇게 해서 메모리 해제 부담을 최대한 줄인다.

그렇다면 다중 스레드는 어떨까?

![](files/Pasted%20image%2020260223185157.png)

이렇게 구현하면 올바른 작동은 보장하지만, 잠금 경쟁이 매우 격렬해 시스템 성능이 저하될 수 있다.
따라서 스레드마다 메모리 풀을 유지해 스레드 간 경쟁을 근본적으로 해결하면 좋다 -> thread local storage

![](files/Pasted%20image%2020260223185225.png)

그렇다면 스레드 A가 메모리 조각을 쓰다가 수명 주기가 A를 넘어서 다른 스레드로 넘어가면 어떻게 해야 할까?
스레드 A에서 할당되었지만 해제는 B에서 맡는다 -> 어떻게 처리?

1. 스레드 B가 스레드 A에게 해제 요청을 한다. [tcmalloc](https://google.github.io/tcmalloc/design.html)이 하는 방법.
2. 애초에 이런 객체는 공유 풀에서 할당하고, 공동 관리한다.
3. 스레드 A가 스레드 B에게 소유권을 넘긴다. 효과적이지만 너무 복잡하다.

## 왜 SSD는 메모리로 사용할 수 없을까?

요즘 SSD는 성능이 좋다. 그렇다면 메모리로 쓸 수 있을까?
근데 여전히 메모리가 SSD보다 훠얼씬 빠르다.
근데 속도뿐만 아니라 다른 차이도 있다.


메모리 주소 지정 단위는 바이트다. 바이트마다 메모리 주소가 있음
그러나 SSD는 그렇지 않다. 조각 단위로 데이터를 관리하고, 이 조각 크기는 매우 다양하다.
근데 CPU가 특정 바이트에 직접 접근할 수 있는 방법이 없다.


또한 SSD에는 수명 제한이 있다.
기록 가능한 최대 바이트 TBW(TeraBytes Written) 등이 있고, 보통 수백 TB 수준이다.
그러나 메모리는 이러한 문제가 없다.
