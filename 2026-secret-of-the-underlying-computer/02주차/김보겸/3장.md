# Ch3. 저수준 계층? 메모리라는 사물함에서부터 시작해보자

## 메모리의 본질, 포인터와 참조
- memory cell: 1바이트 크기의 메모리
    - 0과 1을 저장할 수 있는 메모리, 8비트(=1바이트)로 256가지 상태를 표현할 수 있음
    - unsigned int의 경우 4바이트, struct의 경우 12바이트를 차지
- 변수, 포인터, 참조
    - 변수: 값 혹은 메모리 셀의 주소에 바인딩되는 이름
    - 포인터: 메모리 주소를 더 높은 수준으로 추상화한 것
    - 참조: 포인터를 더 안전하게 사용하기 위한 것, 하드웨어 직접 조작을 막지만 더 안전하게 포인터를 사용할 수 있음


## 프로세스는 메모리 안에서 어떤 모습을 하고 있을까?
- 메모리 내 프로세스는 실제 물리 메모리가 아니라 가상 메모리
- 실제 물리 메모리와 가상 메모리는 페이지 테이블을 통해 매핑됨
    - 프로세스의 주소 공간은 동일한 크기의 조각인 '페이지(page)'로 구성됨
    - 각 프로세스에는 단 하나의 페이지 테이블만 존재


## 스택 영역: 함수 호출은 어떻게 구현될까?
- 함수의 호출은 프로세스의 스택 영역에 '스택 프레임(stack frame)' 또는 '호출 스택(call stack)'을 쌓는 것으로 구현됨
    - 함수가 다른 함수를 호출하는 것은 스택에 새로운 스택 프레임을 쌓는 것
    - 함수가 반환하는 것은 스택에서 스택 프레임을 제거하는 것
    - 이러한 과정이 사실상 이진 트리(binary tree)를 탐색하는 것과 같음(트리 구조 순회로 재귀와 스택 구현 가능)
- 스택 프레임은 함수의 매개변수, 지역 변수, 반환 주소 등을 저장

- 함수 호출은 제어권의 이동
  - 제어권: 실제로 CPU가 어떤 함수에 속하는 기계 명령어를 실행하는지 의미
  - 제어권 이전에는 반환(return - 어디에서 왔는지에 대한 정보)과 점프(jump - 어디로 가야 하는지에 대한 정보)가 필요
    - 점프: 함수 내에서 호출하는 함수의 진입의 메모리 주소가 적혀있기 때문에 거기로 가면 됨
    - 반환: 호출하는 함수의 스택 프레임 하단에 함수 호출 다음 명령어의 주소가 적혀있기 때문에 거기로 가면 됨

- 매개변수 전달과 반환값
    - 매개변수: 호출하는 함수의 스택 프레임에 매개변수를 저장
    - 반환값: 호출하는 함수의 스택 프레임에 반환값을 저장
    - 지역 변수: 호출하는 함수의 스택 프레임에 지역 변수를 저장

- 주의할 점
    - 호출이 종료되면 스택 프레임은 제거됨
        - 따라서 스택 프레임에 저장된 지역 변수는 더 이상 사용할 수 없음
        - 만약 호출이 종료된 스택 프레임에 저장된 지역 변수를 사용하려고 하면 동작을 보장할 수 없음
    - 재귀 호출의 경우 스택 프레임이 계속 쌓이기 때문에 스택 오버플로우가 발생할 수 있음

## 힙 영역: 메모리의 동적 할당은 어떻게 구현될까?
- 특정 데이터를 여러 함수에 걸쳐 사용하되 전역 변수를 사용하고 싶지 않은 경우에 사용 가능
- 메모리 영역을 언제 요청할지와 데이터를 저장에 사용할 메모리 영역을 프로그래머가 직접 관리
- 여유 메모리 탐색 과정
    - LinkedList 형태로 구현하여 메모리 조각에 대한 정보를 탐색할 수 있도록 함
    - 각 노드의 앞뒤 4바이트는 조각의 크기와 할당 유무(f/a)를 저장하는 헤더(header)와 푸터(footer)로 활용
        - 헤더의 경우 요청된 메모리를 할당할 수 있을지 검증하는데 주로 사용된다면
        - 푸터의 경우 할당된 메모리를 해제할 때 앞뒤 메모리 조각과 병합을 위해 주로 사용
    - 할당 전략의 경우 최초 적합 방식, 다음 적합 방식, 최적 적합 방식 등이 존재


## 메모리를 할당할 때 저수준 계층에서 일어나는 일
- 특권 단계(privilege level)에 따라 메모리 접근 권한이 달라짐
    - x86 기준 4단계로 구성되는데 0 - 커널, 1 - 운영체제, 2 - 응용프로그램, 3 - 사용자 (보통 0, 3만 사용)
- 사용자 모드 -> 표준 라이브러리 -> 커널 모드
    - 표준 라이브러리는 사용자 모드가 커널 모드의 구현(서로 다른 운영체제)의 구현에 상관없이 동일한 기능을 제공
    - 표준 라이브러리는 커널 모드와 시스템 콜(System Call)을 통해 상호작용
        - malloc 같은 메모리 할당자는 운영 체제의 일부가 아닌 표준 라이브러리, 여러 종류의 메모리 할당자가 존재
        - 메모리 할당 요청 시 기존 힙 영역이 부족하면 brk 시스템 콜을 통해 힙 영역을 확장
- ⭐️ malloc 등을 이용한 메모리 할당은 실제 물리 메모리 영역을 할당해주는 것이 아닌 가상 메모리 영역을 할당해주는 것
    - 실제 물리 메모리는 프로그램이 할당된 가상 메모리를 사용할 때 커널 상태에서 실제 메모리가 할당됨
    - 운영 체제만이 실제 물리 메모리를 할당할 수 있음


## 고성능 서버의 메모리 풀은 어떻게 구현될까?
- 매번 시스템 콜을 통한 메모리 할당은 비용이 굉장히 큼
    - 1) malloc이 여유 메모리 조각 탐색 -> 적절한 크기의 조각을 찾아 할당
    - 2) 만약 적절한 크기의 조각이 없다면 brk 시스템 콜을 통해 힙 영역을 확장
    - 3) brk로 커널 상태로 전환, 힙 확장 후 사용자 모드로 전환
    - 4) 아직 가상메모리가 할당된 상태이기 때문에 코드가 새로 요청된 메모리를 읽거나 쓰면 페이지 폴트(page fault)가 발생
    - 5) 페이지 폴트가 발생하면 커널 상태로 전환, 실제 물리 메모리가 할당
    - 6) 실제 물리 메모리가 할당되면 사용자 모드로 전환
- malloc 같은 메모리 할당은 범용적인 메모리 할당을 위해 설계
    - 특정 크기의 메모리를 할당하는 데 최적화되어 있지 않음
    - 메모리 풀은 미리 일정 크기의 메모리를 할당받아두고, 필요할 때마다 메모리 풀에서 메모리를 할당
    - 메모리 풀은 여러 개의 메모리 조각으로 구성, 일정 크기의 메모리를 저장, 구현은 연결 리스트(linked list)로 구현
    - 스레드 안정성을 위해 thread local 등을 사용하여 각 스레드마다 메모리 풀을 가짐


## 대표적인 메모리 관련 버그
1) 지역 변수의 포인터 반환하기
    - 함수 내에서 지역 변수의 주소를 반환하면, 함수가 종료된 후 해당 지역 변수는 더 이상 사용할 수 없음
    - 따라서 지역 변수의 주소를 반환하면 동작을 보장할 수 없음
2) 포인터 연산의 잘못된 이해
    - 포인터 연산은 메모리 주소를 기반으로 연산을 수행
    - 포인터에 +1을 하면 메모리 주소가 1 증가하는 것이 아니라, 포인터가 가리키는 데이터 타입의 크기만큼 메모리 주소가 증가
3) 문제 있는 포인터 역참조하기
4) 초기화되지 않은 메모리 읽기
5) 이미 해제된 메모리 사용하기
6) 배열 첨자는 0부터 시작한다
7) 스택 오버플로우
8) 메모리 누수


## 왜 SSD는 메모리로 사용할 수 없을까?
- SSD의 경우 block이 4KB로 구성되어 있고, CPU가 1바이트만 읽어도 4KB를 읽어야 함
- CPU가 파일의 특정 바이트에 직접 접근할 방법이 없음 -> 바이트 단위 주소 지정이 지원되지 않음
- 가상 메모리의 제한: 32비트 시스템의 경우 4GB, 64비트 시스템의 경우 16EB로 제한 - 여전히 1TB 이상의 SSD를 메모리로 사용하기에는 무리가 있음
- SSD의 수명: 낸드 플래시 메모리는 읽기/쓰기 횟수에 제한이 있어 메모리로 사용하기에는 무리가 있음



# Deep Dive - Memory Overcommit

malloc 등을 이용한 메모리 할당은 실제 물리 메모리 영역을 할당해주는 것이 아닌 가상 메모리 영역을 할당해주는 것 -> 의도적인 설계 전략으로 활용 가능 -> 메모리 오버커밋(overcommit)


### 메모리 오버커밋이란?

물리 메모리보다 더 많은 메모리를 프로세스들에게 약속하는 것이다.

```
물리 메모리: 64GB

오버커밋 없이 (factor = 1):
  컨테이너 A: 16GB 할당
  컨테이너 B: 16GB 할당
  컨테이너 C: 16GB 할당
  컨테이너 D: 16GB 할당 요청 → 가용: 64 - 48 = 16GB → 성공
  컨테이너 E: 16GB 할당 요청 → 가용: 0GB → ❌ 거부

오버커밋 2배 (factor = 2):
  광고되는 총 메모리: 128GB (64GB × 2)
  컨테이너 A~H: 각 16GB 할당 → 모두 성공 ✅
  (단, 동시에 전부 사용하면 OOM 발생 가능)
```

### 왜 이것이 가능한가?

Ch3에서 배운 대로, `malloc`은 가상 메모리 주소 공간만 예약. 실제 물리 메모리는 프로그램이 해당 메모리에 접근(read/write)하는 시점에 페이지 폴트를 통해 할당

```
malloc(1GB) 호출
    │
    ▼
커널: 가상 메모리 1GB 매핑 (물리 메모리 사용: 0)
    │
    ▼
프로그램이 실제로 500MB만 사용
    │
    ▼
물리 메모리 500MB만 할당됨 (나머지 500MB는 물리 메모리 소비 없음)
```

- 대부분의 프로그램은 할당받은 메모리를 100% 사용하지 않음, 이 간극을 활용
- 오버커밋은 항공사의 오버부킹(overbooking)과 본질적으로 같은 전략

## Demand Paging — 오버커밋을 가능하게 하는 커널 메커니즘

Ch3에서 배운 내용을 다시 상기하자:

> 4) 아직 가상메모리가 할당된 상태이기 때문에 코드가 새로 요청된 메모리를 읽거나 쓰면 페이지 폴트(page fault)가 발생
> 5) 페이지 폴트가 발생하면 커널 상태로 전환, 실제 물리 메모리가 할당

이것이 바로 Demand Paging

### 페이지 테이블과 Valid 비트

```
가상 메모리 페이지 테이블:

페이지 0: [Valid=1, Frame=0x1000]  → 물리 메모리에 매핑됨 (이미 접근함)
페이지 1: [Valid=1, Frame=0x2000]  → 물리 메모리에 매핑됨
페이지 2: [Valid=0, Frame=없음]    → 아직 접근 안 함 (물리 메모리 소비 없음)
페이지 3: [Valid=0, Frame=없음]    → 아직 접근 안 함
   ...
페이지 N: [Valid=0, Frame=없음]    → malloc으로 예약만 된 상태

실제 물리 메모리 사용 = Valid=1인 페이지 수 × 페이지 크기
가상 메모리 예약량 = 전체 페이지 수 × 페이지 크기
```

### 페이지 폴트의 종류

페이지 폴트는 모두 같은 것이 아니다. 종류에 따라 성능 영향이 크게 다르다:

```
1. Minor Page Fault (소프트 페이지 폴트)
   - 물리 메모리에 페이지가 있지만 페이지 테이블에 매핑이 안 된 경우
   - 예: malloc 후 첫 접근, fork 후 Copy-on-Write
   - 비용: ~1μs (페이지 테이블 엔트리만 갱신)

2. Major Page Fault (하드 페이지 폴트)
   - 물리 메모리에 페이지가 없어서 디스크에서 읽어와야 하는 경우
   - 예: swap에서 복원, mmap된 파일의 첫 접근
   - 비용: ~1-10ms (디스크 I/O 발생, minor의 1,000~10,000배!)

3. Invalid Page Fault
   - 존재하지 않는 주소에 접근한 경우
   - 결과: SIGSEGV (segmentation fault) → 프로세스 종료
```

### Demand Paging 관찰하기

실제로 demand paging을 관찰할 수 있다:

```bash
# C 프로그램: 1GB를 malloc하고 점진적으로 접근
$ cat observe_demand_paging.c
#include <stdlib.h>
#include <string.h>
#include <unistd.h>

int main() {
    // 1GB malloc → 이 시점에서 RSS는 거의 0
    char *buf = malloc(1024 * 1024 * 1024);
    sleep(5);  // 이때 top에서 VIRT=1GB, RES≈0 확인 가능

    // 처음 256MB 접근 → minor page fault 발생, RSS 증가
    memset(buf, 0, 256 * 1024 * 1024);
    sleep(5);  // VIRT=1GB, RES≈256MB

    // 나머지 768MB 접근 → RSS가 1GB까지 증가
    memset(buf + 256*1024*1024, 0, 768 * 1024 * 1024);
    sleep(5);  // VIRT=1GB, RES≈1GB

    free(buf);
    return 0;
}
```

```bash
# 다른 터미널에서 관찰:
$ pidstat -r -p $(pgrep observe) 1
  Time    PID   minflt/s  majflt/s    VSZ      RSS    Command
  0s      1234  0         0           1048576  320    observe  # malloc 직후
  5s      1234  65536     0           1048576  262464 observe  # 256MB 접근
  10s     1234  196608    0           1048576  1048576 observe # 1GB 모두 접근
```

`minflt/s` 열이 minor page fault 발생률. 메모리에 처음 접근할 때 급증하는 것을 볼 수 있다.


### Copy-on-Write: Demand Paging의 확장

`fork()` 시스템 콜은 demand paging의 변형인 **Copy-on-Write (CoW)** 를 사용한다.
```
fork() 직후:

부모 프로세스                     자식 프로세스
페이지 테이블                     페이지 테이블
┌────────────┐                  ┌────────────┐
│ 페이지 0 ──┼────┐     ┌──────┼── 페이지 0  │
│ 페이지 1 ──┼──┐ │     │ ┌────┼── 페이지 1  │
│ 페이지 2 ──┼┐ │ │     │ │ ┌──┼── 페이지 2  │
└────────────┘│ │ │     │ │ │  └────────────┘
              │ │ │     │ │ │
              │ │ └─►┌──┴─┘ │   물리 메모리
              │ └──►┌┴──┘   │   (공유됨, 읽기 전용으로 표시)
              └───►┌┴──┘    │
                   │Frame 0 │
                   │Frame 1 │
                   │Frame 2 │
                   └────────┘

  → 물리 메모리 추가 소비: 0! (페이지 테이블 복사 비용만 발생)

자식이 페이지 1에 쓰기 시도:
  1. 페이지 폴트 발생 (읽기 전용이므로)
  2. 커널이 Frame 1을 복사하여 새 Frame 3 생성
  3. 자식의 페이지 1 → Frame 3으로 재매핑
  4. 부모는 여전히 Frame 1을 사용

  → 이제야 물리 메모리 1페이지분 추가 소비
```

**백엔드 개발자에게 중요한 이유:**

```python
# Python multiprocessing에서의 CoW 활용
import multiprocessing

# 부모 프로세스에서 대용량 데이터 로드
big_data = load_dataset()  # 10GB

# fork 기반 worker 생성
# → 이 시점에서 10GB가 복사되지 않음! (CoW)
pool = multiprocessing.Pool(4)

# 각 worker가 big_data를 읽기만 하면 → 물리 메모리 추가 소비 없음
# 각 worker가 big_data를 수정하면 → 수정된 페이지만 복사됨
results = pool.map(read_only_function, chunks)
```

## 리눅스 커널의 오버커밋 정책

### `/proc/sys/vm/overcommit_memory`

리눅스 커널은 세 가지 오버커밋 정책을 제공한다:

| 값 | 이름 | 동작 | 사용 사례 |
|----|------|------|----------|
| 0 | Heuristic | 커널이 휴리스틱으로 판단. "합리적인" 수준에서 오버커밋 허용 | 대부분의 리눅스 배포판 기본값 |
| 1 | Always | 무조건 허용. `malloc`/`mmap`이 절대 실패하지 않음 | Redis, 일부 과학 계산 |
| 2 | Never | 절대 오버커밋 안 함. 커밋 제한 = RAM + Swap × ratio | 금융 시스템, 실시간 시스템 |


```
물리 메모리: 8GB, Swap: 2GB, 현재 사용 중: 6GB

프로세스가 malloc(4GB) 호출:

모드 0 (Heuristic):
  → 커널이 판단: "여유 2GB + swap 2GB + page cache 회수 가능량..."
  → 상황에 따라 허용 또는 거부 (비결정적)

모드 1 (Always):
  → 무조건 허용
  → 실제 접근 시 물리 메모리 부족하면 OOM killer 발동

모드 2 (Never):
  → CommitLimit = 8 × 0.5 + 2 = 6GB
  → 현재 Committed_AS + 4GB > 6GB → 즉시 거부 (ENOMEM)
  → OOM killer 발동 없이 malloc에서 NULL 반환
```

## 오버커밋과 관련된 커널 메모리 회수 메커니즘

오버커밋이 실제로 동작하려면, 커널이 메모리 압박(memory pressure) 상황에서 메모리를 회수할 수 있어야 한다.

### Page Cache 회수

리눅스는 디스크에서 읽은 파일을 RAM에 캐싱한다 (Page Cache). 메모리가 부족하면 이 캐시를 버려서 공간을 확보한다.

```
메모리 회수 우선순위:

1. Clean Page Cache (디스크와 동일한 캐시)
   → 즉시 버릴 수 있음 (dirty 아님)
   → 비용: 거의 0

2. Dirty Page Cache (수정되었지만 디스크에 안 쓴 캐시)
   → 디스크에 먼저 써야 함 (writeback)
   → 비용: 디스크 I/O 발생

3. Anonymous Pages (힙, 스택 등 프로세스 메모리)
   → Swap 영역으로 내보내기 (swap out)
   → 비용: 디스크 I/O + 나중에 다시 읽어야 함 (swap in)

4. 회수 불가능 → OOM Killer 발동
```

```bash
# Page Cache 상태 확인
$ free -h
              total    used    free    shared  buff/cache  available
Mem:           62Gi    24Gi    1.2Gi    256Mi      37Gi       36Gi
Swap:          8.0Gi   0.5Gi   7.5Gi

# free: 1.2GB로 보이지만
# available: 36GB (buff/cache 37GB 중 대부분이 회수 가능)
# → 실제 가용 메모리는 36GB!
```

**백엔드 개발자가 흔히 겪는 착각:**

> "서버 메모리가 거의 다 찼어요! free가 1GB밖에 안 됩니다!"

이것은 대부분 Page Cache가 메모리를 차지하고 있을 뿐, 실제 가용 메모리는 `available` 열을 봐야 한다. `free`만 보고 메모리 부족으로 판단하면 안 된다.

> 참고: [linuxatemyram.com](https://www.linuxatemyram.com/) — "Linux ate my RAM!" 현상 설명

### Swappiness: 어떤 것을 먼저 회수할 것인가

```bash
$ cat /proc/sys/vm/swappiness
60  # 기본값
```

| swappiness 값 | 동작 |
|--------------|------|
| 0 | Anonymous page를 최대한 swap하지 않음. Page Cache 우선 회수 |
| 60 | 기본값. Page Cache와 anonymous page를 적절히 균형 |
| 100 | Anonymous page도 적극적으로 swap |

**데이터베이스 서버에서의 권장 설정:**

```bash
# PostgreSQL, MySQL 등 DB 서버:
vm.swappiness = 1   # 또는 10

# 이유:
# DB는 자체 버퍼 풀(shared_buffers)로 캐싱을 관리함
# 커널의 Page Cache보다 DB의 버퍼 풀이 더 효율적으로 캐싱
# → anonymous page(DB 프로세스)를 swap하면 쿼리 성능 급락
# → Page Cache를 우선 회수하는 것이 나음
```

> 참고: PostgreSQL Wiki — [Tuning Your PostgreSQL Server](https://wiki.postgresql.org/wiki/Tuning_Your_PostgreSQL_Server)

## OOM Killer — 오버커밋의 안전장치

오버커밋의 최대 위험은 "실사용량의 합이 물리 메모리를 초과하는 순간"이다.
이때 리눅스 커널의 OOM(Out-of-Memory) Killer가 발동한다.

### OOM Killer의 동작

```
물리 메모리 + Swap 모두 소진
    │
    ▼
커널: "더 이상 물리 페이지를 할당할 수 없다"
    │
    ▼
OOM Killer 발동: 프로세스별 oom_score 계산
    │
    ├─ 메모리를 많이 사용하는 프로세스 → 높은 점수
    ├─ root 소유 프로세스 → 점수 감점 (보호)
    ├─ oom_score_adj가 -1000인 프로세스 → 절대 죽이지 않음
    │
    ▼
가장 높은 점수의 프로세스에 SIGKILL 전송
```

### OOM Score 계산 로직

커널의 OOM score 계산을 간소화하면:

```
oom_score 계산 (kernel/mm/oom_kill.c → oom_badness):

기본 점수 = (프로세스의 RSS + swap 사용량 + 페이지 테이블) / 총 메모리
  → 0 ~ 1000 범위로 정규화

조정 = oom_score_adj (-1000 ~ +1000)
  → -1000: 절대 kill하지 않음 (예: init, systemd, sshd)
  → +1000: 최우선 kill 대상
  → 0: 기본값

최종 점수 = 기본 점수 + 조정값
```

```bash
# 프로세스별 OOM score 확인
$ cat /proc/$(pgrep python)/oom_score
245

$ cat /proc/$(pgrep python)/oom_score_adj
0

# 중요 프로세스를 OOM으로부터 보호
$ echo -1000 > /proc/$(pgrep sshd)/oom_score_adj
# → sshd는 절대 OOM kill되지 않음 (서버 접근 경로 보존)
```

### cgroup과 OOM의 관계

cgroup에 `memory.limit_in_bytes = 8GB`가 설정된 경우, 해당 cgroup 내 프로세스들의 메모리 합이 8GB를 초과하면 **cgroup 수준의 OOM**이 발생한다. 이 경우 호스트의 다른 프로세스에는 영향을 주지 않는다.

```
┌── cgroup (limit: 8GB) ──┐         ┌── cgroup (limit: 16GB) ──┐
│  프로세스 A: 5GB         │         │  프로세스 C: 10GB         │
│  프로세스 B: 4GB         │         │  프로세스 D: 3GB          │
│  합계: 9GB > 8GB         │         │  합계: 13GB < 16GB       │
│  → OOM 발생! B를 kill    │         │  → 정상                  │
└─────────────────────────┘         └──────────────────────────┘

호스트 전체 메모리: 충분 → 호스트는 안전
```

한 컨테이너의 메모리 폭주가 다른 컨테이너에 영향을 주지 않는다.

### OOM 디버깅: dmesg 로그 읽기

OOM kill이 발생하면 커널이 dmesg에 상세한 로그를 남긴다:

```bash
$ dmesg | grep -A 20 "Out of memory"

# 출력 예시:
Out of memory: Killed process 12345 (python) total-vm:8388608kB,
  anon-rss:7340032kB, file-rss:4096kB, shmem-rss:0kB,
  UID:1000 pgtables:14336kB oom_score_adj:0

# 해석:
# total-vm: 가상 메모리 8GB (malloc으로 예약한 총량)
# anon-rss: 실제 사용 중인 물리 메모리 7GB (anonymous page)
# file-rss: mmap된 파일의 물리 메모리 4KB
# oom_score_adj: 0 (기본값, 보호/우선 설정 없음)
```

## 보너스: PostgreSQL에서의 Huge Pages와 오버커밋

PostgreSQL은 대형 shared buffer를 사용하며, huge pages와 오버커밋 설정이 서로 영향을 미친다:

```
PostgreSQL shared_buffers = 32GB 설정 시:

일반 페이지 (4KB):
  32GB / 4KB = 8,388,608 개의 페이지 테이블 엔트리 필요
  페이지 테이블 크기: ~64MB (엔트리당 8바이트)
  TLB 미스 빈번 → 성능 저하

Huge Pages (2MB):
  32GB / 2MB = 16,384 개의 엔트리만 필요
  페이지 테이블 크기: ~128KB
  TLB 미스 대폭 감소 → 성능 향상

주의: Huge Pages는 오버커밋되지 않음!
  → 할당 시점에 물리 메모리가 반드시 확보됨
  → overcommit_memory = 2 (Never)와 함께 사용하면
     더 예측 가능한 메모리 동작 보장
```

```bash
# Huge Pages 설정
$ echo 16384 > /proc/sys/vm/nr_hugepages        # 32GB 분량의 2MB 페이지 예약
$ cat /proc/meminfo | grep -i huge
HugePages_Total:   16384
HugePages_Free:    16384
HugePages_Rsvd:        0
Hugepagesize:       2048 kB
```
