## CPU는 유휴 상태일 때 무엇을 할까?

![](files/Pasted%20image%2020260228142649.png)

- CPU는 쉴 때 뭘 할까?
- 작업을 원하는 프로그램들을 위해 대기열을 만들고, 스케줄링을 함
- 그러면 정말 실행을 원하는 프로세스가 없다면?
- 이런 예외를 다루지 않기 위해 대기열에서 항상 실행할 수 있는 프로세스를 찾도록 함. Sentinel Node
- Windows에서는 System Idle Process


- 정말 쉬고 싶다면, `halt` 명령을 실행함
- 일부 모듈을 절전 상태로 변경해 전력 소비를 크게 줄임
- 유휴 프로세스가 하는 일

![](files/Pasted%20image%2020260228142858.png)


- 리눅스 커널에서도 이런 비슷한 구현이 있는데, 더 복잡함
- Deep Sleep(C3), Deeper Sleep(C4) 등 다양한 상태가 있고, 커널이 유휴 시간을 예측해 결정한다


- 근데 이렇게 유휴 상태에 있다면, 어떻게 다른 프로그램에 대해서 응답하는 걸까?
- OS는 일정 시간마다 timer interrupt 생성하고 CPU는 이 신호를 감지해 인터럽트 처리를 수행한다.
- 여기서 프로세스가 실행할 준비가 되었는지 판단하고, 준비되었다면 중단되었던 프로세스를 실행
- 아니라면 프로세스 일시 중지하고 다른 프로세스 스케줄링


- 즉, 프로그램에 무한 루프가 있어도 타이머 인터럽트 통해서 스케줄링 제어

## CPU 코어 수와 스레드 수 사이의 관계는 무엇일까?

- 일반적으로 스레드 수는 코어 수와 일정한 선형 관계를 유지
- 스레드가 많아지면 context switching 비용이 증가하므로 성능이 떨어진다
- 순수 CPU intensive하다면 코어당 스레드 하나가 제일 좋다

## CPU 진화론(상): 복잡 명령어 집합의 탄생

- CPU마다 할 수 있는 일이 다르고, 이를 ISA라고 함(Instruction Set Architecture)
- 초기에는 CPU 명령이 매우 단순했음
- 서로 다른 형태의 CPU는 다른 유형의 명령어 집합을 가지고 있고, ISA는 CPU 하드웨어 설계에도 영향을 미침


- 과거에는 컴파일러가 아직 성숙하지 않아 직접 어셈블리어로 많이 짰음
- 따라서 ISA가 풍부하고, 명령어 자체 기능도 강력해야 한다고 여겼음
- 작업에 딱 대응되는 명령어가 있으면 합리적이고 편리했음

![](files/Pasted%20image%2020260228174938.png)


- 과거에는 메모리가 무척 작았다
- 코드도 메모리를 차지하고, 과거에는 이게 부담이었음
- 하나의 명령어가 더 많은 작업을 할 수 있으면 메모리 부담을 줄일 수 있었음
- 명령어 길이가 고정되어 있지 않고 가변적이라 프로그램 자체의 크기를 줄일 수 있었고
- 명령어는 공간을 절약하기 위해 고도로 인코딩되었음
- 이로 인해 복잡한 ISA가 등장함


- 이 시기 CPU ISA는 모두 hardwired 방식이었음
- 명령어 인출, 해독, 실행이 특정 조합 회로로 직접 제어됨
- 따라서 이는 명령어 실행에는 효율적이지만 유연성이 떨어져 ISA 변경에 대응하기 어려움
- 새로운 명령어가 추가되면 CPU 설계 및 디버깅이 복잡해짐


- 이는 결국 하드웨어 변경이 어려워서 발생하는 일
- 대부분의 명령어에 포함된 연산을 더 간단한 명령어로 구성된 작은 프로그램으로 정의하고, 이를 CPU에 저장하면 모든 기계 명령어에 대응하는 전용 하드웨어 회로를 설계할 필요가 없었음
- 이를 Microcode라고 함

![](files/Pasted%20image%2020260228175224.png)


- 마이크로코드에도 버그가 발생하기 시작함
- 마이크로코드 버그 수정은 일반 프로그램 수정보다 더 어렵고
- 마이크로코드 설계가 트랜지스터를 아주 많이 소모함
- 데이비드 패턴슨 마이크로코드 설계를 개선하려다가, 오히려 마이크로코드 자체를 해결해야 한다고 틀었음

## CPU 진화론(중): 축소 명령어 집합의 탄생

- 메모리가 커지고, 컴파일러가 발전하기 시작함
- 따라서 프로그램 부담이 사라지고 어셈블리어 작성의 의미가 없어짐


- 파레토는 80-20 법칙을 발견했고, 이는 기계 명령어에도 비슷하게 적용됨
	- CPU는 약 80% 시간 동안 ISA 중 20%를 실행하고
	- 복잡한 명령 중 일부는 자주 사용되지 않고, 
	- 컴파일러를 설계하는 프로그래머는 고급 언어를 더 간단한 기계 명령어 조합으로 변환하는 경향이 있음
- 마이크로코드를 제거하면 오히려 프로그램이 더 빠르게 실행되고, CPU를 구성하는 트랜지스터를 절약할 수 있다
- 또한 일부 복잡한 기계 명령이 같은 일을 하는 간단한 명령어 여러 개보다 느림


- 따라서 복잡한 명령어를 간단한 명령어로 대체하기 시작함
- 이를 RISC(Reduced Instruction Set Computer)라고 함


- 복잡한 명령어를 간단한 명령어 여러 개로 대체
- 또한 컴파일러가 더 강력한 제어권을 가짐
	- 마이크로코드와 달리 컴파일러가 할 수 있는 게 더 많으니 CPU가 이를 컴파일러에게 떠맡김
- Load + store 구조
	- 복잡한 명령어는 IF, EX, WB 한 번에 할 수 있음
	- 그러나 RISC에서는 레지스터 내 데이터만 처리할 수 있고, 메모리 내 데이터 처리를 할 수 없음
	- 대신 `LOAD`, `STORE` 명령어 제공


- CISC는
```
MULT A B
```
- 반면 RISC는
```
LOAD RA, A
LOAD RB, B
PROD RA, RB
STORE A, RA
```
- 기계 명령어는 많지만 CPU가 훨씬 단순해진다


- 이렇게 절약한 CPU의 트랜지스터는 CPU의 다른 기능에 활용할 수 있다
- 가장 중요한 점은 명령어가 매우 간단해 실행 시간이 모두 거의 동일하다는 것
	- 물론 메모리 명령은 좀 더 걸림
- 이를 이용하면 파이프라이닝을 더 효율화할 수 있음
- 더 높은 클럭을 가지는 것도 가능

## CPU 진화론(하): 절체절명의 위기에서 반격
	
- 그렇다고 다른 곳에서 이제 RISC를 도입하기에는 최신 CPU에서 이미 작성된 프로그램을 쓸 수 없게 됨
- 결국 복잡한 명령어 집합을 내부에서 RISC의 간단한 명령어로 변환하면 된다
- 이를 micro operation이라고 한다
- 이 또한 매우 간단하고 실행 시간도 같아 파이프라이닝 기술을 충분히 활용할 수 있다


- 또한 복잡 명령어 집합을 축소 명령어 집합처럼 보이게 하는 기술이 하나 더 있다
- 바로 하이퍼스레딩
- 하드웨어 스레드라고도 하는데, 기술적으론 이게 더 적합한 표현

![](files/Pasted%20image%2020260228180642.png)

- 기본적으로 CPU는 한 번에 한 가지 일만 할 수 있다
- 그러나 하이퍼스레딩을 쓰면 물리 CPU 코어 하나를 OS는 여러 개의 논리 코어로 인식
- 하이퍼스레딩 기술이 탑재된 CPU는 한 번에 스레드 두 개에 속하는 명령어 흐름을 처리할 수 있음


- 명령어 간 종속성으로 파이프라인이 완벽하게 채워지는 건 불가능
- 추가 명령어 흐름을 도입해 빈 공간을 채우면 CPU 리소스를 더 활용 가능
- 하드웨어 스레드는 CPU 하드웨어 기능으로 OS와 무관함. 기껏해야 OS가 더 많은 코어를 사용 가능하다고 인식하는 정도


- 하이퍼스레딩은 RISC에서도 도입이 가능하다
- 일부 고성능 RISC에서도 하이퍼스레딩이 쓰인다
- CISC는 점점 더 축소 명령어 집합에 가까워지고, 반대로 일부 고성능 RISC는 마이크로 명령어가 채용되어 있다
- 둘의 차이는 점점 줄어들고 있다


- 그러나 여전히 RISC에서 컴파일러는 중요한 역할을 하고 있고, 명령어 길이가 여전히 일정해 컴파일러 최적화 이점이 더 많다
- LOAD STORE 구조도 CISC에는 여전히 없다

## CPU, 스택과 함수 호출, 시스템 호출, 스레드 전환, 인터럽트 처리 통달하기

- 레지스터가 필요한 이유
	- 속도 때문
	- 메모리 접근 속도는 레지스터보다 100배 느림

- 스택 포인터
	- 스택은 쌓이므로 상단 주소가 제일 중요함
	- 스택 상단 정보는 스택 상단을 가리키는 스택 포인터에 저장된다

![](files/CleanShot%202026-03-06%20at%2021.26.32@2x.png)

- 명령어 주소 레지스터 : PC(Program Counter), IP(Instruction Pointer) 등등
- 프로그램이 실행되면 첫 번째로 실행할 기계 명령어 주소를 PC 레지스터에 저장한다
- 일반적으로 명령어는 순차적으로 실행되므로 PC 레지스터 값도 순차적으로 증가
- 단, if, 함수 호출 등에서는 새로운 주소를 저장


- 주소 레지스터 외에도 상태 레지스터가 있음
- x86에서는 FLAGS 레지스터라고 하고, ARM에서는 응용 프로그램 상태 레지스터라고 한다(application program status register)
- 산술 연산에서 carry, overflow 여부나 CPU 권한(상태)도 레지스터에 저장한다.
- 이를 통해 커널 모드를 판단


- 함수 호출, 시스템 호출, 스레드 전환, 인터럽트 처리 등에서 상태를 저장하고 복구한다
- 근데 이는 모두 중첩 구조 (함수 호출처럼 호출 순서가 있는)라서 스택으로 구현할 수 있음
- 소프트웨어로 구현할 수도 있고 하드웨어로 스택을 구현할 수도 있음

![](files/CleanShot%202026-03-06%20at%2021.37.12@2x.png)

- 함수 호출 시에는 반환 주소, 사용한 레지스터 정보 등을 저장하고 복원
- 이 저장 공간을 스택프레임이라고 함

![](files/CleanShot%202026-03-06%20at%2021.37.58@2x.png)

- 시스템 콜은 결국 운영체제의 함수를 호출하는 것
- OS의 함수를 위한 실행 시간 스택이 필요함
- 이는 커널 상태 스택(kernel mode stack)에 저장됨


- 시스템 호출은 특정 기계 명령어에 대응되며, 32비트 x86에서는 `INT` 명령에 대응됨
- 이 명령어를 실행하면 CPU는 커널 상태로 전환하고, 커널 상태 스택을 찾아서 복구하고 시스템 요청을 처리함

![](files/CleanShot%202026-03-06%20at%2021.38.45@2x.png)

![](files/CleanShot%202026-03-06%20at%2021.38.53@2x.png)


- 인터럽트는 CPU의 실행 흐름을 끊고 특정 인터럽트 처리 함수로 점프해, 함수를 실행하고 원래 위치로 다시 점프
- 이 또한 함수이므로 실행 시간 스택이 필요

![](files/CleanShot%202026-03-06%20at%2021.39.52@2x.png)

- 자체적인 실행 시간 스택이 없는 경우 커널 상태 스택을 이용
- 인터럽트 처리 함수(interrupt service routine) 스택, 즉 ISR 스택이라는 자체적인 실행 시간 스택이 있다

![](files/CleanShot%202026-03-06%20at%2021.43.01@2x.png)

- 인터럽트를 처리하는 것은 CPU이므로, CPU마다 자신만의 인터럽트 처리 함수를 가진다

![](files/CleanShot%202026-03-06%20at%2021.44.39@2x.png)

- 시스템 콜은 사용자 상태 프로그램이 직접 실행하지만 인터럽트는 외부 장치로 실행된다
- CPU가 사용자 상태에서 명령어를 실행할 때마다 인터럽트가 언제든지 발생할 수 있다


- 커널 상태를 스택으로 쓰는 경우
- 인터럽트가 발생하면
- CPU가 사용자 상태에서 커널 상태로 전환되고
- 사용자 상태 스레드에 대응되는 커널 상태 스택을 찾아 커널에 유저 스레드의 실행 상황 정보를 저장
- CPU는 인터럽트 처리 함수로 점프하고 이를 실행
- 실행이 끝나면 커널 상태 스택에 저장된 상황 정보에 따라 CPU 상태가 복원되고 유저 모드로 전환되어 다시 사용자 프로그램 실행된다

![](files/CleanShot%202026-03-06%20at%2021.47.26@2x.png)

- 그렇다면 스레드 전환은?
- A가 실행 중, B는 일시 중지된 상태
- 타이머 인터럽트 터지면, CPU는 인터럽트 프로그램에서 A에게 할당된 CPU Time Slice가 다 사용되었나 판단 -> 있다면 사용자 상태로 돌아가 실행을 계속하고, 아니라면 전환 시작
	- 주소 공간을 B로 전환하고
	- CPU 정보를 A에서 B로 전환
		- A의 CPU 상황을 저장하고
		- B의 CPU 상황을 복원

![](files/CleanShot%202026-03-06%20at%2021.49.45@2x.png)

- 스레드 B 입장에서는 타이머 인터럽트로 중지되었던 그 시점이므로, 아무 일이 없었던 것처럼 실행을 재개함
