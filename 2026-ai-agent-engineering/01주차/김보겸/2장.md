# 에이전트 시스템 설계

## 우리의 첫 번째 에이전트 시스템
에이전트의 작업 범위는 균형 잡혀야 함 
- 범위가 너무 좁으면 다른 요청을 놓침
- 범위가 너무 넓으면 엣지케이스 고민과 함께 성공/실패 기준을 세우기도 어려움
명확한 경계가 있는 작업에 집중하면 구체적인 입력, 구조화된 출력, 짧은 피드백 루프 확보 가능

## 에이전트 시스템의 핵심 구성 요소
### 모델
- 에이전트의 의사결정, 상호작용, 학습 역량 결정
- 에이전트가 입력을 해석하고 출력 생성, 환경에 적응하는 방식을 결정 -> 시스템의 성능, 확장성, 지연시간, 비용에 직접적 영향

### 도구
- 에이전트가 특정 작업을 수행하고 문제를 해결할 수 있도록 함
- 보통 에이전트가 해결하려는 작업에 맞춰 설계. 다양한 조건과 컨텍스트에 에이전트가 어떻게 작업을 수행할지 고려
- 각 도구는 필요에 따라 쉽게 통합하거나 교체할 수 있도록 독립적인 모듈로 설계해야 함

### 메모리
- 에이전트가 정보를 저장하고 검색할 수 있게 함
- 단기 메모리: 
    - 현재 작업이나 대화와 관련된 정보를 저장하고 관리하는 능력
    - 상호작용 중 컨텍스트를 유지하는 데 사용, 실시간으로 일관된 의사결정 내리는데 도움을 줌
    - 롤링 컨텍스트 윈도우 방식을 통해 구현
- 장기 메모리:
    - 오랜 기간에 걸쳐 지식과 경험을 저장하도록 하여 과거 정보를 바탕으로 향후 행동 결정에 도움
    - 시간이 지남에 따라 발전해야 하거나 개인화된 경험을 제공할 때 유용
    - 데이터베이스, 지식 그래프, 파인튜닝된 모델로 구현
- 메모리 관리 및 검색: 저장된 데이터를 체계적으로 구성하고 인덱싱하여 성능 향상. 때로는 오래되거나 불필요한 정보를 일부러 잊음

### 오케스트레이션
- 개별적인 기능을 end-to-end 솔루션으로 전환
- 여러 스킬을 배치하고 상황에 따라 실행, 전체 워크플로를 관리하고 조정하는 역할
- 도구나 스킬 호출의 가능한 순서를 평가, 결과 예측하여 가장 적절한 순서로 실행
- 진행 상황과 환경을 모니터링, 필요시 계획 수정, 도구 호출, 메모리 업데이트


### 엔지니어링적 확장
- 동적 GPU 할당
- 탄력적 GPU 프로비저닝
- 우선순위 큐잉, 작업 스케줄링
- 비동기 작업 실행
- 동적 로드 밸런싱

## 에이전트 시스템의 아키텍처
- 단일 에이전트 아키텍처
    - 하나의 에이전트가 시스템 내 모든 작업을 관리, 실행
    - 명확히 정의된 좁은 범위의 작업에 적합, 하나의 엔티니로 처리할 수 있는 업무에 이상적
- 다중 에이전트 아키텍처
    - 여러 에이전트가 공동의 목표를 달성하기 위해 협력
    - 작업을 전문화된 에이전트가 분담하거나 병렬 처리
