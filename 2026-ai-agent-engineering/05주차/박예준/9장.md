## **왜 측정이 핵심인가**

LLM 기반 에이전트는 **확률적**이기 때문에 "빠르게 배포하고 나중에 고친다" 접근이 통하지 않는다. 엄격한 측정과 평가 없이는 변경이 개선인지 회귀인지 알 수 없고, 팀은 사실은 뒷걸음질치면서도 전진하고 있다고 착각하기 쉽다.

**기존 지표 vs 에이전트 지표**

| 구분 | 기존 SW | 언어 기반 에이전트 |
| --- | --- | --- |
| 정답 형태 | 결정적, 단일 | 의미는 같되 표현 다양 |
| 주 지표 | Exact Match, 정확도
**→ 완전 일치** | 임베딩 거리, BERTScore, BLEU, ROUGE
**→ 의미론적 유사도 측정** |
| 실패 양상 | 크래시, 명시적 에러 | 유창하지만 오도하는 출력 |
| 재현성 | 동일 입력 → 동일 출력 | 확률적 변동 존재 |
- 명확한 목표 정의에서 시작
- 실행 가능한, 적절한 지표 선정 → 에이전트의 작업 수행 능력과 사용자 기대 충족 여부를 평가하는 벤치마크
    - 정량 - 정확성, 응답 시간, 견고성, 확장성, 정밀도, 재현율
    - 정성 - 사용자 만족도
- 실제 환경에서의 시스템 성능 평가 → 견고한 테스트 프레임워크 구축

**지표 과적합(metric overfitting)의 함정**: BLEU나 Exact Match 같은 단일 점수를 최적화하면 부자연스러운 정형 출력이 유도되어 사용자의 진짜 의도를 놓친다. 평가를 **고정된 체크리스트가 아니라 살아 있는 프로세스**로 다뤄야 한다.

### **평가 세트**

평가 세트는 **"살아 있는 명세서"**다. 수작업으로 고정된 테스트 스위트는 과적합되고 롱테일 실패 유형을 놓친다. 구조화된 JSON 포맷으로 입력 상태와 기대 최종 상태를 함께 정의해 자동 채점을 가능하게 한다.

```json
{
  "order": { "order_id": "A89268", "items": [...], ... },
  "conversation": [ { "role": "customer", "content": "..." }, ... ],
  "expected": {
    "final_state": {
      "tool_calls": [
        { "tool": "issue_refund",
          "params": { "order_id": "A89268", "amount": 19.99 } }
      ],
      "customer_msg_contains": ["처리되었습니다", "영업일"]
    }
  }
}
```

**평가 세트 확장 경로** → *목표 지향적인 다양한 시도를 통해 미묘한 오류를 발견하고 더욱 견고한 에이전트의 행동 검증 가능* 

| 경로 | 설명 | 장점 |
| --- | --- | --- |
| 수작업 추가 | 엔지니어·SME가 직접 작성 | 높은 품질, 의도 명확 |
| 프로덕션 로그 마이닝 | 실제 세션에서 추출 | 현실 분포 반영 |
| LLM 생성 | 파운데이션 모델로 변형·증강 | 빠른 확장, 엣지 케이스 |
| 적대적 프롬프팅 | 에이전트를 모순시키는 입력 | 견고성 테스트 |
| 반사실적 편집 | 단어 하나 바꿔 실패 유도 | 민감도 파악 |
| 분포 보간 | 두 의도를 섞어 모호하게 | 의도 해석 테스트 |

MMLU·BBH·HELM 같은 표준 벤치마크는 도메인 특화 에이전트엔 부족하지만 **추세 파악 용도로 병행**할 수 있다. 새 도메인에서는 엔지니어와 주제 전문가를 짝지어 커스텀 벤치마크를 구축하고, 실패 유형 태깅과 커버리지 추적용 메타데이터를 포함시킨다.

## **컴포넌트 평가 (단위 테스트)**

### **도구 평가**

- 같은 입력에 대해 출력이 **결정적**이어야 함 — 확률성이 설계에 포함되어 있다면 통계적 특성을 점검
- 출력 정확성뿐 아니라 **지연시간·리소스 사용량·오류 처리**까지 명시적으로 검증 → 부하나 장애 상황에서 우아하게 성능 저하되는지 확인
- 외부 의존성(API·DB)이 있는 도구는 **모킹·시뮬레이터**로 엣지 케이스 재현 — 운영 환경에선 드물지만 잘못 처리되면 치명적인 상황들 (다양성, 모호성, 엣지 케이스)
- 도구 수정 시마다 전체 회귀 테스트

### **계획 능력 평가**

계획 모듈은 사용자 의도를 도구 호출 시퀀스로 번역하는 단계. 기존 스크립트와 달리 **확률적이고 적응적**이어서 더 세심한 테스트가 필요하다.

| 지표 | 측정 내용 | 낮을 때의 의미 |
| --- | --- | --- |
| **도구 재현율** (tool recall) | 예상 도구를 모두 호출했는가 | 필수 행동 누락 |
| **도구 정밀도** (tool precision) | 불필요한 도구 호출은 없는가 | 의도 오해 |
| **파라미터 정확도** | 인자(order_id, 금액)가 올바른가 | 잘못된 컨텍스트 기반 작동 |
- 정형화된 워크를로부터 시작 : 사용자 의도와 그에 대해 검증된 에이전트의 응답 쌍으로 구성
- 플랜을 체계적으로 평가 : 에이전트를 E2E로 실행하고 그때 선택된 행동 추출

```python
def tool_metrics(pred_tools, expected_calls):
    expected_names = [c.get("tool") for c in expected_calls]
    pred_set, exp_set = set(pred_tools), set(expected_names)
    tp = len(exp_set & pred_set)
    return {
        "tool_recall": tp / len(exp_set),
        "tool_precision": tp / len(pred_set) if pred_set else 0.0
    }

def param_accuracy(pred_calls, expected_calls):
    if not expected_calls:
        return 1.0
    matched = 0
    for exp in expected_calls:
        for pred in pred_calls:
            if pred.get("tool") == exp.get("tool") \
                    and pred.get("params") == exp.get("params"):
                matched += 1
                break
    return matched / len(expected_calls)
```

**일관성 검증도 중요**: 결정적 시나리오에서는 동일 입력이 항상 동일 출력이어야 하고, 확률적 시나리오라도 허용 경계 안에 머물러야 한다. 엣지 케이스, 재현 가능성, 작은 입력 변화에 대한 민감도, 필드 누락이나 도구 실행 실패 시의 우아한 처리 여부를 점검한다.

→ 점점 전체 범위를 반영하는 계획 시나리오 코퍼스를 축적하게 됨 = 계획 능력에 대한 통합 테스트의 중추

### **메모리 평가**

메모리는 멀티 턴 대화·장시간 실행 워크플로·지속적 사용자 프로필에 필수. 네 축으로 평가한다.

| 평가 축 | 구체 점검 항목 |
| --- | --- |
| **정확성** | 기록된 데이터가 정확히 저장되고, 시간 경과·다른 작업 개입 후에도 정밀하게 조회되는가 · 최대 메모리 용량, 특이 데이터 타입, 매우 빠른 R/W 사이클 같은 경계 사례 · 잘못된 형식·중복·모호 항목 스트레스 |
| **관련성** | 오래되거나 무관한 정보를 끌어올리지 않는가 · 데이터 누출·인덱싱 오류로 잘못된 선호도 반환 방지 · 표현·의미만 비슷한 무관 데이터의 오조회 점검 |
| **효율성** | 메모리 부하 증가 시 조회 시간·리소스 벤치마킹 · 성능이 급격히 떨어지는 구간·병목 식별 · 벡터 검색·시맨틱 메모리는 임베딩·인덱싱 로직의 미묘한 오류를 잡기 위해 **'쉬운' 조회와 '어려운' 조회 시나리오 모두 테스트** |
| **복원력** | DB 사용 불가·데이터 손상·버전 마이그레이션 시뮬레이션 · 사용자 영향 최소화하며 우아하게 복구하거나 통제된 방식으로 실패하는지 |

주 지표는 Top-k 정확도:

```python
def evaluate_memory_retrieval(retrieve_fn, queries, expected_results, top_k=1):
    hits = 0
    for query, expect in zip(queries, expected_results):
        results = retrieve_fn(query, top_k)
        if set(results) & set(expect):
            hits += 1
    accuracy = hits / len(queries) if queries else 1.0
    return {f"retrieval_accuracy@{top_k}": accuracy}
```

### **학습 평가**

학습 컴포넌트는 확률성과 데이터 의존성 때문에 가장 까다롭다. 

[점검 포인트]

- 파라미터/캐시/규칙 갱신이 레이블·피드백·보상에 맞게 이뤄지는가
- **지도학습**: 대표 데이터셋 기대 정확도 도달 + 검증 데이터 일반화
- **강화학습**: 보상 극대화가 실제 행동 개선으로 이어지는가, 학습 정체 구간 감지 (조기 종료·동적 탐색)
- **일반화**: 홀드아웃 세트, 합성 예제, 취약한 휴리스틱·암기 응답을 겨냥한 적대적 테스트 케이스
- **적응력**: 새 사용자 입력, 이전에 보지 못한 도구 실패, 변화하는 보상 구조 같은 **분포 변화(distribution shift)** 시뮬레이션 → **치명적 망각(catastrophic forgetting)이나 성능 붕괴** 없이 적응하는가
- 지도·비지도·강화 학습 패러다임 간 상호작용이 미묘한 버그를 일으키지 않는가

## **총체적 평가 (통합 테스트)**

단위 테스트가 통과해도 조합에서 터지는 문제가 있다. E2E 시나리오에서는 다음 지표를 한꺼번에 계산한다:

- 도구 재현율, 도구 정밀도, 파라미터 정확도
- **문구 재현율** (phrase recall) – 필요한 문구가 최종 메시지에 포함됐는가
- **작업 성공률** (task success) – 전체적으로 목표를 달성했는가

(예시) `evaluate_single_instance(raw, graph)` 헬퍼로 이 평가를 운영화 — 구조화 입력으로 에이전트를 호출하고 AIMessage에서 도구 호출을 추출해 지표를 계산하는 패턴이다. 이 접근법의 **한계도 명시**되는데, 자동화된 테스트의 품질은 평가 세트와 지표의 품질에 좌우된다. 테스트 케이스가 좁거나 대표성 없으면 오프라인에선 통과하고 운영에선 실패한다.

총체적 평가가 다루는 네 가지 난제:

| 난제 | 핵심 질문 | 대응 기법 |
| --- | --- | --- |
| **일관성** (consistency) | 확률적 변동에도 정합성 유지?
(에이전트의 응답 <> 입력 간의 정합성) | 자동 검증, LLM 기반 평가(동일하거나 유사한 모델로 에이전트가 자신의 출력을 기대와 대조), **액터–크리틱 프레임워크** |
| **응집성** (coherence) | 장기 대화에서 컨텍스트 보존? (사용자가 같은 말을 반복하지 않아도 이전 정보 참조) | 긴 대화 시뮬레이션, 상태 이해 검증, 행동이 논리적이며 목표 지향적 순서를 따르는지 검증 |
| **할루시네이션** | 사실 데이터 기반인가? | RAG 그라운딩, 인간 개입 피드백, **비용 인식 평가** |
| **예기치 않은 입력** | 적대적·비정형 입력에서 우아한가? | 과거 사고·엣지 케이스 체계 주입 |

**액터–크리틱 평가**: '액터'가 응답을 생성하고 '크리틱'이 사전 정의된 정합성·관련성 기준으로 평가. 복잡하거나 동적 시나리오에선 이것만으론 부족하므로 LLM 기반 점검 + 사람 피드백과 결합한다.

**할루시네이션 비용 인식 평가**: 최근 확산 중. 추론 비용을 고려해 할루시네이션 감소와 비용을 균형 있게 맞춤 — '할루시네이션 비용'을 정량화해 정확도 향상과 연산 오버헤드를 저울질하는 지표로 사용한다. 콘텐츠 정확성을 보장하기 위해, 검증된 DB에 의존, 신뢰할 수 있는 정보원 기준으로 응답하는지 등을 교차 확인하도록 엄격히 테스트해야 한다.

- 최근에는 피드백 루프를 통해 에이전트의 예측과 실제 결과 불일치를 식별하고 모델이나 데이터 소스를 업데이트하도록 트리거해 신뢰성을 개선하는 것을 자동화할 수 있음
- 이는 하이브리느 사람-AI 피드백 루프를 중심으로 발전해옴

**예기치 않은 입력 처리**: 무작위 퍼징이 아니라 과거 사고·적대적 분석에서 얻은 엣지 케이스를 체계적 탐색. 안전이 중요한 응용에서는 **민감 정보 유출 방지, 정책 위반 방지, 하위 시스템 연쇄 실패 방지**를 명시적으로 검증한다.

> *효과적인 통합 테스트는 무작위 ‘퍼징’만이 아니라 과거 사거나 적대적 분석에서 얻은 통찰로 엣지 케이스를 체계적으로 탐색한다*
> 

## **배포 준비**

배포 기준은 단순 테스트 통과가 아니다. 충족해야 할 항목:

- **정량 임계값**: 환불·취소 시나리오에서 도구 재현율 95% 이상
- **부하·엣지 케이스 안정성**
- **워크플로 무결성**: 핵심 경로가 의도대로 작동
- **E2E 통합 테스트 통과**, 지연시간·가용성 목표 충족, 치명적 버그 없음

이를 강제하는 장치가 **게이트 메커니즘**이다:

> 모든 요구사항이 충족되기 전에 프로덕션 승격을 막는 자동/수동 점검
> 
- 최신 평가 스위트에서 회귀 1건이라도 감지되면 배포 차단 (예: modify_5 다중 턴 테스트 회귀)
- 파일럿·베타 성공 후에만 프로덕트 리드의 명시적 승인
- 자동 결과가 애매하면 사람에게 사례 전달
- 파일럿과 결합된 신속한 롤백 프로세스 필수
