# Ch3. 에이전트 시스템을 위한 UX 디자인

## 동기식 vs 비동기식 에이전트 경험

- 동기식 에이전트
  > 사용자와 실시간으로 상호작용하며 즉각적으로 대화
  - **_#즉시성 #명확성 #반응성_**
  - 즉각적 피드백, 실시간 의사결정이 필요한 작업
    - e.g. 채팅 인터페이스, 음성 대화, 실시간 협업 도구
  - 낮은 지연시간, 컨텍스트 인식이 필수
  - 응답에서는 명료함과 간결함을 우선해야 함
  - 오류 처리 및 회복이 중요 (사용자와의 실시간 상호작용에 영향을 줘서는 안 됨)
- 비동기식 에이전트
  > 사용자와 독립적으로 작동하고 시간 간격을 두며 소통
  - **_#유연성 #지속성 #장기적 명료성_**
  - 시간이 오래 걸리거나 백그라운드 처리 필요, 사용자의 지속적 주의를 요구하지 않는 작업
    - e.g. 이메일과 유사한 상호작용, 작업 알림, 처리 완료 후 전달되는 에이전트 생성 보고서
    - 알림, 요약, 구조화된 보고서 → 투명성을 유지하는 핵심 도구
  - 작업 상태와 결과를 명확히 전달해야 함

→ 작업의 성격, 사용자 기대, 운영 컨텍스트에 따라 적절한 설계가 필요하다

→ 능동성은 두 디자인 모두에서 중요하게 작용하며, 긴급 이슈 알림, 최적화 제안, 적시 리마인더 등에서 큰 도움이 됨

\*사용자는 알림 빈도, 채널, 전달 임계값을 제어해 필요에 맞게 에이전트 행동을 커스터마이징 할 수 있음

## 컨텍스트 유지와 연속성

> 다단계 워크플로 안내, 중단된 대화 이어나가기, 과거 상호작용에 따라 행동을 조정할 때 컨텍스트 유지 능력은 **_사용성, 효율, 신뢰_** 에 직접적인 영향을 주는 요소이다

### 메모리 저장 방식

1. **순수 클라이언트 측 컨텍스트** (e.g. 브라우저 메모리 저장)
   - 세션 내에서는 빠르게 느껴지지만, 기기나 로그인 간 연속성을 잃어 UX를 해침
2. **순수 서버 측 컨텍스트** (e.g. 사용자ID에 연결된 데이터베이스 저장)
   - 장기 메모리와 크로스 디바이스 경험이 가능하지만 지연시간, 프라이버스 고려사항을 초래할 수 있음
3. **하이브리드 접근** (e.g. 단기 컨텍스트-클라이언트/장기 컨텍스트-서버)
   - 대체로 최적의 UX 경험을 달성
   - 즉, 효과적인 컨텍스트 유지를 위해서는 단기/장기 메모리를 모두 적절히 관리해야 함
     - 단기 - 진행 중인 세션에서 방금 전 질문의 세부나 지침
     - 장기 - 여러 세션에 걸쳐 선호, 과거 상호작용, 더 넓은 사용자 패턴

### 상호작용 간 상태 유지

> 효과적인 상태 관리는 시스템이 **사용자나 세션을 어떻게 식별, 추적하느냐**에 달려 있음

[예시]

- 로그인 사용자 → 상태를 사용자 계정에 직접 연결해 기기-세션을 넘나들며 메모리 지속성 제공
- 익명 사용자 → 쿠키, 토큰 등의 세션 식별자로 대화 추적

- 에이전트 시스템 규모가 커질수록 메모리 외에 DB, 분산 캐시를 통한 상태 관리도 고려해야 함
  1. 사용자 기반 메모리 - 장기, 지속적, 개인화
  2. 세션 기반 메모리 - 단기, 휘발적, 세션 범위
- 명확한 세션 경계, 데이터 검증, fallback 메커니즘도 상태 유지를 위해 고려해야 함

### 개인화와 적응성

> 개인화는 단순한 컨텍스트 기억을 넘어 과거 상호작용과 선호를 사용해 에이전트의 행동, 응답, 추천을 사용자별로 조정하는 것이다

[개인화의 다양한 형태]

- **선호 보존** : 알림 선호, 자주 선택하는 옵션 등 사용자 설정 기억
- **행동 적응** : 관찰된 사용자 패턴에 기반해 응답 스타일, 상호작용 흐름 조정
- **능동적 지원** : 과거 행동을 바탕으로 사용자 필요를 예측하고 제안을 제공

→ 어떤 데이터를 저장하고 사용하는지 투명성이 신중하게 다뤄져야 하는 주제

## 에이전트 능력 커뮤니케이션

> 사용자가 **에이전트가 무엇을 할 수 있으며 어떻게 상호작용해야 하는지**를 명확하게 이해시키는 것은 효과적인 에이전트 경험을 설계하는 데 매우 중요하다!

- 백엔드 에이전트 설계가 지원 기능을 지원하지만,
- 그만큼 인터페이스 자체를 통해 발견 가능성, 직관성, 실사용성을 제공하야 함 \***_다양한 모달리티와 연관_**
  1. 채팅 기반 에이전트 - 제안 액션 버튼 제공
  2. 확장형 메뉴, 능력 카드로 제공 기능을 구조적으로 열거
  3. 개방형 텍스트 입력 - 스스로 능력을 분명하게 알리기 (사용자가 어떤 액션을 취할 수 있는지를 가이드처럼 미리 제공)
- 단, 한번에 너무 많은 옵션 제시X
- 효과적인 디자인은 **점진적 공개**를 우선하여 초기에는 핵심 기능을 보여주고 사용자가 익숙해질수록 고급 기능을 드러냄
- 사용자는 에이전트의 범위와 한계를 이해할수록 생산적으로 참여하고 출력에 신뢰를 가지며 워크플로에 통합할 가능성이 커질 것

### 신뢰도 & 불확실성 커뮤니케이션

- 에이전트는 종종 결정적 규칙이 아닌 통계 모델에 기반한 확률적 환경에서 작동
- 불확실성을 효과적으로 전달하는 일은 사용자 신뢰를 쌓고 사용자가 정보에 근거한 결정을 내리도록 돕는 데 필수적임
  1. **명시적 진술** : ‘정답일 확률 90%’
  2. **시각적 신호** : 아이콘, 색상 기반 경고 그래픽 인터페이스의 신뢰도 미터
  3. **행동 조정** : 신뢰도가 낮을 때는 단정적 권고 대신 제안을 제공
- 애매모호한 사용자 입력에 대해, 위험한 추정보다는 명확한 질문과 사용자 지침을 구할 시점을 아는 것이 좋음

### 효과적인 질문 방식

1. 명확하고 공손하며 컨텍스트 인식적
2. 로봇 같거나 반복적인 표현 피하기
3. 사용자가 이미 대화 일부에서 답변했다면, 처음부터 다시 묻지 않고 해당 컨텍스트 참조하기
4. 왜 명확화가 필요한지 투명하게 설명하기 (e.g. _정확하게 진행하려면 정보가 조금 더 필요합니다_ )
5. 한 번에 너무 많은 질문은 피하기
6. 논리적 순서로 질문 구성하여 가장 중요한 모호성부터 해소하기

### 우아한 실패

> 핵심은 **문제를 투명하게 인정하고 유용한 설명을 제공하며 실행 가능한 다음 단계를 제시**하는 것

- 잘못된 답이나 무의미한 응답 대신 안내하기
  - e.g. ‘찾으시는 정보를 확인하지 못했습니다.’ ‘사람 상담원으로 연결해드릴까요?’
- 흔한 실패 지점을 예측해 사전에 fallback 메커니즘 갖추기
  - e.g. 음성 기반 에이전트가 반복적으로 이해하지 못하면 → 텍스트 입력으로 전환하거나 질문을 직접 입력해달라고 명확히 설명
- 다단계 작업 - 실패 시 진행 상태 보존이 중요함
- 사과와 공감의 언어 (문제 발생 시 과도하게 기술적인 오류 메시지 지양)
- **⭐ 실패로부터 학습하기 - 실패 지점을 기록하고 반복 이슈를 분석해 개발 프로세스에 반영하여 유사 실패 빈도 줄이기**
  - → 시간이 지날수록 더 견고하고 신뢰성 높은 시스템으로 발전 가능

## 상호작용 설계에서의 신뢰

> 사용자가 에이전트를 명료하게 이해한다면, 확신 있는 생산적 협업이 가능하다!

- 투명성은 에이전트의 능력과 제약을 명확히 전달하는 데서 시작한다
  - 사용자는 에이전트의 설명에 대해, 해당 추론에 대한 가시성을 얻는다
  - 여기서 추론은 행동을 신뢰할 만큼의 적절한 인사이트 정도면 충분하다
- 예측 가능성은 다양한 상황에서의 일관된 행동을 통해 투명성을 보완한다
- 신뢰와 투명성을 우선하면 신뢰가능한 협력자가 된다
  - **사용자는 에이전트가 일관되게 행동하고 적절히 응답하며 오류를 우아하게 처리할 것이라 기대할 수 있어야 한다**
- 신뢰성은 기대를 일관되게 설정하고 충족하는 것이다
- 신뢰성의 또 다른 핵심은 시스템 복원력이다

### **에이전트가 지켜야 할 패턴 (공통)**

- **능력을 명확히 전달하기** - 온보딩, 제안, 버튼 등
- **모달리티를 신중히 결합하기** - 텍스트, GUI, 음성, 비디오 ↔ 사용자 컨텍스트에 결합
- **컨텍스트를 신중히 유지하기** - 과도한 기억, 프라이버시 침해 X
- **오류를 우아하게 처리하기** - 명확하고 공손한 fallback
- **신뢰 구축하기** - 투명하게 한계, 신뢰도 신호, 추론을 공유
