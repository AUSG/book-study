# Ch4. 도구

## AI 에이전트의 도구는 무슨 역할을 할까?

> 도구란, **_에이전트가 원하는 결과를 달성하기 위해 수행할 수 있는 특정 기능 또는 일련의 작업_** 이다

- AI 에이전트가 추가 정보와 컨텍스트를 검색하고 작업을 수행하며 환경과 의미 있게 상호작용하도록 한다
- 정보 제공 뿐만 아니라 실제 변화를 만들어내는 에이전트는 도구를 통해 구현 가능한 것!
  - 텍스트 생성 능력을 넘어 기능을 확장하도록 호출 가능한 외부 함수
- 각 도구는 **독립적으로 개발, 테스트, 최적화할 수 있는 모듈형 구성 요소**로 통합되어 복잡한 행동이 가능한 일관된 시스템을 이룬다

- 랭체인의 중심은 **채팅 모델**

# 도구의 종류

## 로컬 도구

- 보통 특정 작업에 맞춘 사전 정의 규칙과 로직을 따라 만들어짐
- 만들고 수정하는 것이 비교적 쉬우며, 에이전트와 함께 배포하는 형태
- 로컬 도구는 **정확성, 예측 가능성, 단순성**을 제공함 ← **_로직이 명시적으로 정의되기 때문_**
- 메타데이터(도구의 이름, 설명, 스키마)를 바탕으로 어떤 도구를 호출할지 결정됨
  ⚠️ 주의사항
  - `이름` 이름이 일반적이면, LLM이 불필요한 상황에도 도구를 호출할 수 있음
  - `설명` 여러 도구에 걸쳐 지나치게 넓거나 겹치는 설명은 혼란과 성능 저하를 초래함
  - `스키마` 스키마가 모호할수록 파운데이션 모델이 도구를 언제, 어떻게 사용할지 정확히 이해하지 못해 오작동을 초래함
- 반면, 확장성이 떨어지고 중복 구현이 많아질 수 있으며 유지보수가 까다로운 한계가 있음
- e.g. **수학 연산과 같은 단순한 수작업 도구** - 파운데이션 모델이 가진 약점ㅇㄹ 해결하는 데 특히 유용함
  - 실제로 실행이 필요한 도구만 파운데이션 모델에 바인딩 되도록 파라미터로 제어하여 더 많은 이익을 가져갈 수 있음

## API 기반 도구

- 자율 에이전트-외부 서비스 간의 상호작용으로 로컬에서 수행하기 어려운 데이터 처리나 액션 실행을 통해 기능을 확장할 수 있음
  - 에이전트의 기능을 동적으로 확장하고 확장성을 높이는 수단 제공
- API 기반 도구가 유용한 상황
  1. 에이전트가 여러 외부 시스템과 통합하는 경우
  2. 실시간 데이터를 가져오는 경우 (외부 소스로부터 항상 최신 데이터에 기반한 응답과 액션 제공 가능)
  3. 내부에서 처리하기에 자원 소모가 큰 복잡한 연산을 수행해야 하는 경우
- 작업의 범위를 크게 넓혀 전체적인 기능과 효율성을 높일 수 있음

## 플러그인 도구

- 모듈화되어 있으며 최소한의 커스터마이징만으로 AI 에이전트 프레임워크에 통합할 수 있는 구성 요소
  - 기존 라이브러리, API, 서드파티 서비스를 통해 에이전트의 기능 확장
- e.g.
  1. OpenAI
     - \*ChatGPT 제품 내에서만 사용 가능한 제약이 있을 수 있음
     - 1차 제공 플러그인 - Expedia, Zapier
       - OpenAI의 Completions, Chat 엔드포인트에서 호출 불가 → 랭체인으로 플러그인 기능을 모사하는 커스텀 함수 호출 계층 구현 필요
  2. Anthropic의 Claude
     - `tool use` 기능 전체를 Message API로 직접 제공
     - 사용자는 커스텀 툴 등록이나 도구를 사용해 클로드가 추론 시점에 이를 호출하게 할 수 있음
     - 콘텐츠 모더레이션, 편향 탐지, 도메인 특화 서비스 등을 클로드 기반 워크플로에 쉽게 통합 가능
  3. 구글 Gemini
     - Vertex AI API를 통해 함수 호출 지원
  4. Microsoft Phi
     - Azure의 AI 검색, 문서 처리, 데이터 시각화 API 등 다른 서비스와 통합 가능
- **`장점`** : **모델 실행 계층에서의 통합 가능성** - 복잡한 커스터마이징이나 추가 개발 없이 기능을 즉시 강화할 수 있는가
- **`한계점`** : 로컬/원격에서 제공되는 맞춤형 툴처럼 높은 수준의 커스터마이징과 적응성을 제공하지는 않음

- 오픈소스 파운데이션 모델에서도 플러그인 도구 생태계가 빠르게 형성되고 있음
  - e.g.
    1. Hugging Face의 Transformers 라이브러리 - NLP 작업을 위한 다양한 사전 학습 모델, 플러그인 도구 제공 → 텍스트 생성, 감정 분석, 언어 번역 등의 기능 쉽게 통합 가능
    2. Glama.ai, [mcp.so](http://mcp.so) - 수많은 MCP 서버를 통합해 간단한 유틸리티~복잡한 상태 기반 서비스를 탐색 및 활용하도록 지원
- 벡터 검색 도구 - 질의에 대한 관련 문서에 근거한 응답 생성
  - 금융 분야, 이상 탐지, 예측 분석, 교육 분야에서 인사이트 제공 및 의사결정 정확성을 높일 수 있음

## MCP 도구

> **_Model Context Protocol_**
>
> **모델에 독립적인 통합 표준을 제공해 LLM-외부 시스템을 연결하는 통일된 방식의 인터페이스 역할**
>
> 클라우드 저장소에서 문서를 읽고 비즈니스 애플리케이션에 데이터를 전송하며 내부 API를 호출하고 다른 에이전트와 협력하는 과정을 모두 지원

- 각 데이터 소스나 서비스별로 커스텀 어댑터를 작성하는 통합 방식의 한계를 극복
- 여러 에이전트에 걸친 **도구 재사용 문제**를 해결한다는 점에 의의가 있음
- 모듈식으로 재사용 가능한 아키텍처를 구성해 에이전트 개발을 더 쉽게 만듦

→ 대규모로 AI 에이전트를 설계, 배포, 운영하는 방식에 의미 있는 진전을 기여함

### 구성 요소

- MCP 서버 : 표준화된 [JSON-RPC 2.0](https://www.jsonrpc.org/specification) 인터페이스를 통해 데이터나 서비스를 노출하는 웹 서버
- MCP 클라이언트 : MCP를 사용하는 에이전트 / LLM 애플리케이션
</aside>

- MCP 사양을 구현하기만 하면 어떤 시스템이든 서버로 래핑할 수 있음
- 클라이언트가 JSON-RPC 요청을 전송하고 구조화된 JSON 응답을 받는 방식으로 통신
  - e.g. ‘이 /tmp 폴더에 모든 파일을 나열해주세요’, ‘id=1234로 getA() 함수를 실행해주세요.’
- 단일의 표준화된 [JSON-RPC 2.0](https://www.jsonrpc.org/specification) 인터페이스를 정의함으로써 MCP는 서비스 구현을 에이전트 로직에서 분리함
- 프로토콜이 일관되므로, 내부 작동 원리를 알 필요 없이 노출된 메서드만 알면 됨

### 보안 이슈

- 여러 에이전트가 MCP 엔드포인트를 공유하는 경우, **인증, 접근 제어, 잠재적인 공격 벡터**와 관련된 문제가 아직 완전히 해결되지 않은 상태
- 💡 아직 활발히 연구되고 있는 주제, 엔지니어링 과제
  1. 특정 메서드를 오직 인가된 에이전트만 호출하도록 보장하는 것
  2. 민감한 데이터에 대한 역할 기반(role-based) 접근 제어를 유지하는 것
  3. 악의적인 페이로드 주입을 방지하는 것
  4. Audit Log를 유지하는 것

## 상태 유지 도구

- 로컬 스크립트, 외부 API, MCP에서 배포된 서비스 모두에 존재 가능한 형태
- **`한계`** 파운데이션 모델에 지속적인 상태를 직접 조작할 권한을 부여할 시, 모델이 파괴적인 실수를 하거나 악성 사용자가 악용할 수 있다
  - e.g. AI 에이전트가 DB 성능을 최적화한다며 프로덕션 테이블의 절반 이상 row를 삭제한 사건도 있었음
    - → 임의 SQL 실행 엔ㄴ드포인트를 노출하는 대신 범위가 좁게 정의된 작업만 도구로 등록하는 식으로 대응 필요
- 읽기 전용 접근만 필요한 에이전트는 데이터를 삭제/수정할 권한을 절대 가져서는 안 됨
- 만약 자유형 쿼리가 반드시 필요한 상황이라면, 철저한 데이터 sanitization과 접근 제어를 구현해야 함
  - e.g. OWASP의 GenAI Security Project https://genai.owasp.org/
- **최소 권한 원칙**이 설계의 기준이 되어야 함
  1. 입력 검증을 통해 `DROP`, `ALTER`와 같은 패턴이 포함된 문장 거부
  2. 파라미터 바인딩, prepared statement를 사용해 SQL 인젝션 방지
  3. 에이전트가 사용하는 DB 계정 분리 및 쿼리 실행 권한 최소한만 부여

# 도구 개발 자동화

## Code Generation

1. **파운데이션 모델 활용**
   - LLM에 **API 명세나 샘플 입력값**을 제공하면 초기 래퍼, 헬퍼 함수, 상위 수준의 원자적 연산을 자동 생성할 수 있음
   - 모델이 안전한 샌드박스 환경에서 코드를 작성하고 실행하여 스스로 결과를 평가하는 방식
     - → 일일이 래퍼를 만들지 않아도 에이전트가 직접 호출할 수 있는 검증된 소규모 집합을 얻게 됨
   - 복잡한 API 환경을 다룰 때 특히 유용함
2. **실시간 코드 생성**
   - AI 에이전트가 작동 중 필요에 따라 코드를 작성하고 실행하는 기능
   - 특정 작업을 수행하기 위해 기존 도구를 수정하거나 새로운 도구를 생성할 수 있고, 결과적으로 매우 높은 적응력을 갖추게 됨
   - 실시간 데이터 분석이나 복잡한 SW 통합 등 동적 무제 해결과 유연성이 요구되는 애플리케이션에서 매우 중요
   - **`문제점`** 품질 관리, 재현성 문제, 리소스 소비

# 도구 사용 설정

- 파운데이션 모델 API (OpenAI, Anthropic, Gemini)
  - 파라미터 **`tool-choice`**를 통해 도구 사용 방식을 명시적으로 제어할 수 있다
    **[Mode]** https://developers.openai.com/api/docs/guides/function-calling#tool-choice
    1. `auto` 모델이 컨텍스트에 따라 도구 호출 여부를 스스로 결정
    2. `any`, `required` 모델이 최소 1개의 도구를 반드시 호출하도록 강제 → 도구의 출력이 필수적인 경우에 이상적
    3. `none` 모든 도구 호출이 차단됨 → 제어된 출력이나 테스트 환경에서 활용
    4. 특정 도구 고정 → 예측 가능하고 반복 가능한 흐름 보장
  - 파운데이션 모델 중심의 유연한 호출 방식 → **결정적 기능**으로 전환 가능
- 에이전트가 도구 호출 중에 예상치 못한 오작동을 보일 수 있으므로, 반드시 신뢰가능한 fallback 메커니즘, Preprocessing 체계를 마련해야 함
  **[교정 절차]**
  1. 스키마 검증 - jsonschema, Pydantic 등으로 누락된 필드나 잘못된 구조 탐지
  2. 지능적 재시도 - 지수 백오프와 같은 구조화된 재시도 로직 적용, 문제된 부분만 재생성
  3. 안정적인 fallback - 재시도 실패 시 백업 모델이나 서비스 사용, 캐시된 데이터 활용으로 기본값 반환
  4. 로깅 - 프롬프트, 도구 호출, 검증 오류, 재시도, 폴백 등 모든 단계를 기록해 관측 가능성 확보 및 디버깅과 개선에 활용
