# 07. 크레이그리스트 설계

## 7.1 사용자 스토리와 요구사항
- 10억 명 이상, 지리적 분할


- 게시물에는 다음 정보 포함
	- 제목
	- 설명 문단 몇 개
	- 가격. 단일 통화 가정하고 통화 변환 무시
	- 위치
	- 최대 10장의 사진(각 1MB)
	- 동영상. 이후에 추가될 수 있움


- 게시자는 7일마다 게시물 갱신할 수 있음
- 게시자는 갱신할 수 있는 클릭 링크가 담긴 이메일 알림을 받음


- 조회자는 다음 할 수 있음
	- 지난 7일 동안 특정 도시에서 작성된 모든 게시물 보거나 검색할 수 있다. 결과 목록 끊임 없이 스크롤
	- 결과에 필터 적용
	- 개별 게시물 클릭해 세부 정보 본다
	- 이메일 등으로 게시자에게 연락
	- 사기성, 오해의 소지가 있는 게시물을 신고. (게시물에 낮은 가격 명시하고, 설명에는 더 높은 가격을 쓴다거나 등등)


- 비기능적 요구사항
	- 확장성 : 단일 도시에서 최대 1,000만 명의 사용자 수용
	- 높은 가용성 : 99.9% 가동 시간
	- 높은 성능 : 게시물 작성 후 몇 초 내에 볼 수 있어야 함. 검색과 게시물 조회의 P99가 1초
	- 보안 : 게시물 작성 전에 로그인 필요


- 필요한 스토리지 대부분은 게시물
	- 사용자에게는 해당 지역의 게시물만 표시
		- 이는 데이터 센터가 전체 게시물 중 일부만 저장하면 된다는 뜻
		- 게시물은 자동이 아니라 수동 생성이므로 스토리지 증가 속도가 느림
		- 자동으로 생성되는 데이터는 처리하지 않음
		- 게시물은 1주일 후 자동 삭제될 수 있음


- 스토리지 요구사항이 낮다는 건 모든 데이터를 단일 호스트에 저장할 수 있음
- 즉, 분산 스토리지 솔루션이 필요 없음
- 게시물이 평균 1,000자, 1KB 텍스트 포함 가정
- 대도시에 1,000만 명 인구가 있고 10%가 게시자고, 하루 평균 10개의 게시물을 쓴다면 10GB/일. SQL DB로 충분히 몇 달치 커버 가능

## 7.2 API
- 게시물 관리, 사용자 관리 나누기


- CRUD 게시물
	- GET, DELETE /post/{id}
	- GET /post?search={search_string}
		- 모든 게시물 GET하는 엔드포인트
		- 게시물 내용 검색을 위해 `search` 쿼리 매개변수 가질 수 있음
		- 페이지네이션 쿼리 매개변수 포함할 수도 있음
	- POST, PUT /post
	- POST /contact
	- POST /report
	- DELETE /old_posts
- 사용자 관리
	- POST /signup
	- POST /login
	- DELETE /user
- 기타
	- Health Check
		- 200 반환하거나 P99, 다양한 엔드포인트 가용성 통계 등


- 다양한 필터가 있을 수 있음
- 단순화를 위해 고정된 필터 세트 가정
- 필터는 프론트엔드, 백엔드 모두에서 구현 가능
	- 동네 정보(enum)
	- 최소 가격, 최대 가격
	- 물품 상태(enum)

## 7.3 SQL 데이터베이스 스키마
- Post : 비정규화해 JOIN 쿼리 없이 운영
- Images : post_id와 image_address 가짐


- 이미지 저장 : 객체 스토리지에 저장.


- 사용자 쿼리에 낮은 지연 시간이 필요할 때, SQL이나 레디스와 같은 낮은 지연 시간의 인메모리 DB를 사용한다.
- HDFS와 같은 분산 파일 시스템을 사용하는 NoSQL DB는 대규모 데이터 처리 작업용이다.

## 7.5 모놀리스 아키텍처
- 첫 제안 설계는 직관적이지 않다
- 실무에서 이런 모놀리스를 쓸 일은 적다
- 그러나 모든 설계 결정은 트레이드오프에 관한 것임을 명심하고, 이런 설계를 제안하고 트레이드오프를 설명하는 것을 두려워하지 말자


- UI, 백엔드 모든 기능 포함하는 모놀리스 애플리케이션 구현하고, 전체 웹 페이지를 객체 스토리지에 저장할 수도 있다
- 핵심 설계 결정은 게시물의 사진 포함해 게시물 웹페이지 전체를 객체 스토리지에 저장할 수 있는 것
- Post 테이블의 많은 열을 사용하지 않을 수 있다는 의미


- 이 방식은 구현, 유지보수가 간단하지만 주요 트레이드오프는 다음과 같다
	1. HTML, CSS, JS 모든 게시물에서 중복
	2. 네이티브 모바일 앱 개발하면 브라우저 앱과 백엔드를 공유할 수 없다. 가능한 해결책은 프로그레시브 웹 앱을 개발하는 것
	3. 게시물 분석은 HTML을 파싱해야 한다. 물론 이런 유틸리티 만들어도 된다.


- 첫 번째 트레이드오프 단점은 중복된 페이지 구성 요소 저장하기 위해 추가 저장 공간 필요
- 또 다른 단점은 새로운 기능이나 필드를 오래된 게시물에 적용할 수 없다
- 그러나 게시물은 일주일 후에 삭제되므로 때에 따라 허용될 수도 있다


- 브라우저 앱을 반응형으로 작성하고, 모바일 앱을 웹뷰를 쓰면 두 번째 트레이드 오프를 어느 정도는 해소할 수 있다.


- 백엔드 서비스, 객체 스토리지 서비스 인증을 위해서 타사 사용자 인증 서비스를 사용하거나 자체적으로 유지보수할 수도 있다.

## 7.6 SQL 데이터베이스와 객체 스토리지 사용
![](files/Pasted%20image%2020260520211435.png)

- UI 프론트엔드 서비스가 있고, 백엔드 서비스는 SQL 서비스에 요청을 보낸다
- 객체 스토리지는 이미지 파일 용. SQL 데이터베이스가 게시물 나머지 내용 저장


- 이미지도 DB에 넣어 객체 스토리지를 아예 쓰지 않을 수도 있다.
- 그러나 이는 매번 이미지를 백엔드 서비스를 통해 다운받아야 하므로 백엔드 호스트에 추가 부담을 주고, 지연 시간을 늘리고, 다운로드 실패 가능성을 높인다
- 초기 구현을 단순하게 유지하고 싶으면 게시물에 이미지 없이 시작하고, 나중에 객체 스토리지와 함께 기능을 추가하는 것도 방법이다.


- 만약 이미지 수가 제한되고, 파일 사이즈가 작다면 SQL에 이미지를 저장할 수도 있다고 제안할 수는 있다.
- 이 설계의 장점은 단순성이다

## 7.7 마이그레이션은 번거롭다
- SQL에 이미지 저장하는 다른 문제점은 이후 객체 스토리지로 마이그레이션해야 한다는 것
- 한 데이터 스토리지에서 다른 데이터 스토리지로 마이그레이션하는 건 언제나 까다로운 일이다


- 다음을 가정하고 최대한 간단한 마이그레이션 프로세스를 논의하자
	- 두 데이터 스토리지를 단일 엔티티 취급할 수 있다. 복제는 추상화 되어있고, 지연 시간이나 가용성 등 비기능적 요구사항 최적화하기 위해 데이터 센터에 데이터가 어떻게 분산되어있는지 고려할 필요가 없다
	- 다운 타임이 허용된다. 마이그레이션 중 애플리케이션 쓰기를 비활성화할 수 있다.
	- 다운타임이 시작될 때 진행 중인 요청을 끊거나 종료할 수 있다.  다운 타임을 푸시나 공지로 사전 통지할 수도 있다.

![](files/Pasted%20image%2020260520212036.png)

- 개발자 노트북에서 마이그레이션하는 스크립트 작성할 수도 있다.
- 한 번 돌리면 끝이므로 스크립트 개선에 큰 가치는 없다
- 단, 버그나 네트워크 문제로 중단될 수도 있으므로 재실행 가능해야하며, 멱등성을 가져야한다.
- 이런 체크포인트 메커니즘이 동작할 때 같은 순서로 매번 레코드를 읽어야 한다
	- 전체 리스트를 가져와 정렬해서 읽거나
	- 타임스탬프나 위치 등 순서를 정할 수 있는 필드가 있다면 이걸 활용한다


- 데이터가 너무 크다면 데이터 센터 내에서 스크립트 실행해야 한다
- 쓰기 오류 등이 생기면 이를 로깅하고, 계속 실행해야 한다.


- 데이터 마이그레이션은 복잡하고 비용이 많이 드는 작업이므로 가능하면 피해야 한다.
- 한 시스템에 사용할 데이터 스토리지를 선택할 때 실험적 모델로 소량의 데이터만 처리하는 시스템을 구현하는 경우가 아니라면 처음부터 적절한 데이터 스토리지를 구축해야 한다.

## 7.8 게시물 작성과 읽기
![](files/Pasted%20image%2020260520212415.png)
- 이 접근 법에서는 백엔드는 이미지 파일이 객체 스토리지에 성공적으로 업로드되었는지 알지 못한 채 성공을 반환한다
- 백엔드가 전체 게시물이 성공적으로 업로드되었는지 확인하려면 백엔드 자체에서 이미지를 업로드해야한다.
![](files/Pasted%20image%2020260520212505.png)
- 그러나 여기에도 트레이드오프가 있다
- 전자
	- 장점
		- 이미지 업로드를 클라가 하므로 리소스 소모가 적다
		- 이미지 파일이 추가 호스트를 거치지 않아 전반적인 지연 시간이 감소한다. CDN이 있다면 CDN 엣지를 활용할 수 있어 차이가 더 커진다
- 후자
	- 장점
		- 객체 스토리지에 인증, 권한 부여 메커니즘 구현하고 유지보수할 필요가 없다. 객체 스토리지가 외부에 노출되지 않는다
		- 조회자가 게시물의 모든 이미지를 볼 수 있음이 보장된다. 이것이 타협 가능한지는 면접관과 논의할 수 있다


- 두 접근법의 장점을 대부분 취하는 한 가지 방법은 클라가 이미지 파일을 백엔드에서 쓰지만 CDN에서 읽기


- 클라가 정말 이미지 파일을 별도 요청으로 업로드해야 할까?
- 이런 복잡성은 불필요할 수도 있다.
- 쓰기 요청 최대 크기는 10MB 약간 넘을 거고, 몇 초 내에 업로드할 수 있을 만큼 작다.
- 그러나 이는 재시도 또한 더 비용이 많이 든다는 것이다.


- 이미지 가져오기 요청은 병렬로 이뤄질 수 있다
- 파일은 보통 여러 스토리지 호스트에 복제되므로 병렬로 빠르게 다운로드할 수 있다.

## 7.9 기능적 분할
- 확장의 첫 단계는 도시와 같은 지리상 지역별로 기능적 분할을 사용하는 것
- 이를 지리적 위치 라우팅이라고 하며, 예를 들어 DNS 쿼리의 위치를 기반으로 서비스한다
- 따라서 데이터센터의 DB에는 서비스하는 도시의 데이터만 포함된다.
- 바이너리로그 복제로 DB간 데이터 복제를 할 수도 있다.


- 크레이그리스트는 각 도시에 서브도메인 할당해 이런 지리적 분할 수행
![](files/Pasted%20image%2020260520213114.png)

- 서브도메인 사용하는 대신 UI 드롭다운 메뉴에서 도시를 선택할 수도 있따
- 도시를 선택해 적절한 데이터 센터에 요청을 보내고, 해당 도시의 게시물 볼 수도 있다

## 7.10 캐싱
- 지연시간 SLA 준수하고 오류율 줄이기 위해 인기있는 게시물 캐싱할 수 있다
- Redis로 LRU 캐시할 수 있다
- 키는 게시물 ID, 값은 게시물 전체 HTML


- 게시물의 정적 특성으로 게시자가 글 업데이트할 수 있지만, 캐시가 Stale되기 쉽지는 않다

## 7.11 CDN
![](files/Pasted%20image%2020260520213236.png)
- CDN 사용을 고려할 수는 있지만, 이 서비스는 모든 사용자에게 표시되는 정적 미디어, 즉 이미지와 동영상이 매우 적다

- 읽기 작업 확장이 필요하다면 SQL 복제 등을 활용할 수 있다

## 7.13 쓰기 처리량 확장
- 일반적으론, 읽기 위주겠지만
- 게시물 생성을 프로그래밍 방식으로 허용하고, 공개 API를 제공한다고 해보자.
- INSERT, UPDATE 트래픽 급증이 있으면 처리량을 초과할 수도 있다.
- DB에 따라 빠른 INSERT 구현을 사용하거나, 배치 커밋하거나 할 수 있다

### 카프카와 같은 메시지 브로커 사용
![](files/Pasted%20image%2020260520213552.png)
- 쓰기 트래픽 급증 처리를 위해 카프카와 같은 스트리밍 솔루션을 쓸 수도 있다
- SQL 서비스 앞에 카프카를 배치


- 새 게시물 제출하거나, 기존 게시물 업데이트할 때 Post 토픽에 쓴다
- 이 서비스는 Stateless이며, 수평적 확장 가능
- PostWriter 만들어서 Post 토픽에서 지속적으로 소비하고 SQL 서비스에 쓸 수 있다


- 주요 트레이드오프
	- 복잡성, 최종 일관성
	- 이미 카프카가 있을 확률이 높으니 복잡성이 어느 정도 상쇄된다
	- 쓰기가 SQL 팔로워에 도달하는 데 시간이 더 오래 걸리므로 최종 일관성 지속 시간이 늘어난다


- 필요한 쓰기 처리량이 단일 SQL 호스트 넘어선다면 SQL를 기능적 분할하고, 쓰기 트래픽이 많은 카테고리에 전용 SQL 클러스터를 둘 수도 있다
- 이 해결책은 이상적이지 않은데, 애플리케이션 로직이 카테고리 따라 DB 선택해야 한다
- 쿼리 로직이 애플리케이션에도 존재하게 된다. 유지보수가 어려워진다.


- 더 높은 쓰기 처리량이 필요하다면 카산드라나 HDFS가 있는 카프카와 같은 NoSQL DB를 사용할 수도 있다.


- DDoS 방지를 위해 앞단에 Rate Limit을 달 수도 있다.

## 7.14 이메일 서비스
- 일주일 지나서 갱신 알림 보내는 건 SQL로 7일된 게시물 쿼리하고, 각 게시물에 대해서 이메일 전송 요청 보내는 배치 ETL 작업 구현할 수 있다
- 다른 앱의 알림 서비스는 트래픽 급증 처리, 낮은 지연 시간, 짧은 시간 내 알림 전달 등의 요구사항이 있을 것이다

## 7.16 오래된 게시물 제거
- 크론 작업이나 Airflow로 `DELETE /old_posts` 매일 호출하는 방식으로 구현할 수 있다


- 이 작업은 간단하고 중요도가 낮다
- 삭제 예정이던 게시물이 며칠 동안 계속 접근 가능한 상태로 남아있어도 괜찮아 크론 작업으로도 충분하고, Airflow 쓰면 오히려 불필요한 복잡성이 생길 수 있다
- 단지 예정된 시간보다 일찍 삭제하지 않도록 주의하고, 이 기능에 대한 모든 변경사항은 배포 전에 철저히 테스트해야 한다.
- Airflow 등의 복잡한 플랫폼 대신 단순성 선택하면 유지보수성이 향상된다.


- 오래된 게시물 제거하거나 전반적으로 오래된 데이터 삭제하는 것의 
	- 장점
		- 스토리지 프로비저닝, 유지보수 비용 절감
		- 읽기 및 인덱싱 속도 향상
		- 모든 데이터 복사, 마이그레이션 등 할 때 속도가 빨라지고 복잡성이 낮아지고 비용이 줄어든다.
		- 프라이버시 보호 우려가 줄고, 데이터 유출 영향이 제한된다. 그러나 이는 공개된 데이터라 이러한 이점이 크게 느껴지지 않는다.
	- 단점
		- 데이터 보관함으로써 얻을 수 있는 분석, 인사이트 손실
		- 정부 규정으로 인해 특정 기간 데이터를 보관해야 할 수 있음
		- 삭제된 게시물의 URL이 기존 게시물에 활용될 가능성은 낮지만 존재하며, 조회자가 옛 게시물 보고 있다고 착각할 수도 있다. 링크 단축 서비스 사용하면 이런 일이 발생할 확률은 더욱 더 높다. 그러나 매우 낮고, 영향도 작아 허용할만 하다. 민감한 개인 데이터가 노출될 수 있다면 허용할 수 없다.


- 비용 문제고, 오래된 데이터 접근이 빈번하지 않다면 압축 후 테이프 같은 저비용 아카이브 하드웨어나 AWS Glacier, Azure Archive Storage 등의 서비스를 쓸 수도 있다.

## 7.19 기타 논의 가능한 주제

### 7.19.1 게시물 신고
- 다음 내용을 충족하는 시스템 설계가 포함될 수 있따
	- 일정 수의 신고가 접수되면 게시물이 삭제되고 게시자에게 이메일 알림이 전송됨
	- 게시자 계정, 이메일이 자동으로 차단되어 로그인하거나 게시물을 작성할 수 없다. 그러나 로그인하지 않고 게시물을 계속 볼 수 있고, 다른 게시자에게 이메일을 보낼 수 있다
	- 게시자는 관리자에게 연락해 이 결정의 이의를 제기할 수 있어야 한다. 이런 상호작용과 결정을 추적하고 기록하는 시스템이 필요한지 면접관과 논의할 수 있다
	- 게시자가 이메일 차단하고 싶다면 자신의 이메일 계정에서 발신자의 이메일 주소를 차단하게 설정해야 한다.

### 7.19.2 점진적 성능 저하
- 각 구성 요소의 장애는 어떻게 처리? 장애를 일으킬 수 있는 가능한 예외 상황은 무엇이고 어떻게 처리할 수 있을까?

### 7.19.3 복잡성
- 이 서비스는 소규모 팀이 유지보수하기 쉽도록 단순성을 최우선으로 설계된 간단한 분류 광고 앱이다
- 기능 세트는 의도적으로 제한되고 잘 정의되어 있으며, 새로운 기능은 거의 도입되지 않는다.
- 이를 달성하기 위한 전략을 설명할 수 있다
	- 종속성 최소화
		- 라이브러리 버전이 너무 오래되거나, 취약점으로 인해 업데이트해야 할 수도 있다.
		- 종속성을 최소화하면 디버깅, 문제 해결, 유지보수가 간단해진다
		- 이 접근 방식은 각 지역이나 고객층별로 광범위한 맞춤화가 필요하지 않은 최소한의 유용한 기능 세트를 제공하는 데 중점을 두는 적절한 기업 문화를 필요로 한다.
		- 예를 들어 결제 기능을 제공하지 않는 이유는 결제를 처리하는 로직이 각 도시마다 다를 수도 있다
	- 클라우드 서비스 사용
	- 전체 웹 페이지를 HTML 문서로 저장
		- 크레이그리스트의 기능은 굉장히 간단해 앞에서 본 것처럼 하나의 HTML 파일로도 충분하다.
		- 대신, 모든 항목에 중복된 HTML이 포함되나 저장 공간을 트레이드오프한다.
		- 이 접근 방식은 필드 추가나 제거도 덜 복잡하게 만든다. DB 마이그레이션 없이 필드를 변경할 수 있다.
		- 보존 기간이 있고, 자동을 삭제될 오래된 게시물의 필드를 수정할 필요가 없다
	- 관찰 가능성 
		- 관찰 가능성의 중요성을 강조해야 한다
		- 로깅, 모니터링, 알림, 자동화된 테스트에 투자
		- 좋은 모니터링 대시보드, 런북, 디버깅 자동화 등 우수한 SRE 사례 채택

### 7.19.7 구독과 저장된 검색
- 문자 제한이 있는 검색어 저장하고, 일치하는 새 게시물에 관한 알림을 받을 수도 있다
- 이는 그 자체로 복잡한 시스템이 될 수 있다
- 사용자는 모든 저장된 검색을 포함하는 단일 일일 알림을 받아야 한다


- 사용자에게 저장된 검색에 대한 새로운 결과를 보내기 위해 일일 배치 ETL 작업을 구현할 수 있따
- 이 작업을 구현하는 방법으로 적어도 두 가지를 제안할 수 있다
	- 검색 서비스에 중복 요청을 허용하는 더 단순한 방법
	- 이를 피하는 더 복잡한 방법

### 7.19.8 검색 서비스 중복 요청 허용
- ES는 빈번한 검색 요청을 캐싱하므로 동일한 검색어로 빈번한 요청을 해도 많은 리소스를 낭비하지 않는다
- 배치 ETL 작업에서 사용자와 개별 저장 검색어를 한 번에 하나씩 처리할 수 있다
- 각 처리는 사용자의 검색어를 검색 서비스에 요청 보내고, 결과를 통합 후 알림 서비스로 보낸다

### 7.19.9 검색 서비스에 중복 요청 피하기
- 검색어 중복을 제거해 각 용어에 한 번만 검색한다. 검색은 대소문자 구분하지 않을 수 있으니 검색어를 소문자로 바꿀 수도 있다
- 검색어에 대해 요청 보내고, 해당 검색어와 관련된 사용자를 DB에서 찾아서 알림을 보낸다
	- 이게 실패한다면 어떻게 될까? 중복 전송을 어떻게 피할 수 있을까?
	- 알림이 이미 표시되었는지, 닫혔는지 등은 클라에서 구현할 수 있다
	- 브라우저나 모바일 앱 등 일부 클라는 가능하지만 이메일, 문자 등에서는 불가능하다

### 7.19.10 속도 제한
- 서비스 요청은 모두 rate limit을 거쳐야 한다

### 7.19.11 대량의 게시물
- 위치에 관계없이 모든 목록이 누구나 접근할 수 있는 단일 URL 제공하고 싶다면?
- Post 테이블이 단일 호스트에 비해 너무 클 수 있고, ES 인덱스도 그럴 수 있따
- 그러나 단일 호스트에서 여전히 검색 쿼리를 제공해야 한다. 여러 호스트에서 쿼리 조회하고, 결과 집계하는 설계는 지연 시간이 길고 비용이 많이 든다
- 그 대신 아래 정도 가능성이 있다
	- 게시물 만료를 일주일로 정하고, 만료된 게시물 삭제하는 일일 배치를 만든다. 짧은 보존 기간은 검색하고 캐시할 데이터가 적다는 뜻이므로 시스템 비용, 복잡성 줄인다
	- 게시물에 저장되는 데이터 양을 줄인다
	- 게시물 카테고리에 기능적 분할을 수행한다. 그러나 애플리케이션이 적절한 맵핑 로직을 구현해야 한다
	- 압축된 데이터 검색은 비용이 너무 많이 들므로 압축을 고려하지 않을 수도 있다.

### 7.19.12 지역 규정
- 국가나 지역에 따라 자체 규정을 가질 수도 있다
	- 허용되는 제품이나 서비스 유형이 관할 구역마다 다르다면?
	- 고객 데이터, 프라이버시 규정으로 인해 데이터 국외 반출이 불가능하다면?
		- 물론 이런 고려사항은 면접 범위를 벗어날 가능성이 높다


- 정확한 요구사항을 설명해야 한다
	- 사용자 위치에 따라 제품이나 서비스 구역을 선택적으로 표시해도 괜찮은지
	- 혹은 사용자가 금지된 제품이나 서비스 보는 것도 방지해야 하는지


- 사용자 IP 주소 국가 따라 필터링 로직을 클라에 추가한다
- 더 나아가 이런 규정이 많거나 자주 바뀐다면 관리자가 규정을 구성할 수 있는 서비스를 만들고, 클라가 이 서비스에 요청해서 결정해야 한다
- 이 서비스는 읽기 트래픽이 많고 쓰기 트래픽이 적으므로 CQRS 기법을 적용할 수 있다


- 금지된 콘텐츠 게시를 막아야 한다면 금지된 단어나 문구를 감지하는 시스템 혹은 머신러닝 접근법을 설명해야 할 수도 있다.


- 또한 국가별로 목록 맞춤화를 시도하지 않았는데, 이런 접근법의 트레이드오프를 논의할 수도 있다.
