# 10. 데이터베이스 배치 감사 서비스 설계
- 데이터 품질은 여러 차원이 있다
	- 정확성(Accuracy) : 측정 값이 실제 값에 얼마나 가까운지
	- 완전성(Completeness) : 데이터에 우리의 목적에 필요한 모든 값이 있는지
	- 일관성(Consistency) : 다른 위치의 데이터가 동일한 값을 가지며, 다른 위치에서 동시에 동일한 데이터 변경 사항을 제공하기 시작하는지
	- 유효성(Validity) : 데이터가 올바르게 형식화되고 값이 적절한 범위 내에 있는지
	- 고유성(Uniqueness) : 중복되거나 겹치는 데이터가 없는지
	- 적시성(Timeliness) : 필요한 때 데이터를 사용할 수 있는지


- 데이터 품질 검증은 이상 감지, 수동으로 정의된 유효성 검사 등이 있다
- 여기서는 수동으로 정의된 유효성 검사만 설명
- 특정 테이블이 매시간 업데이트되고 때때로 몇 시간 동안 업데이트가 없을 수 있지만, 두 업데이트 사이의 간격이 24시간을 초과하는 것은 매우 드문 일이다.
- 이때 유효성 검사 조건은 최근 타임스탬프가 24시간 미만인 경우다.


- 수동 유효성 검사를 통한 배치 감사는 일반적인 요구사항이다. 트랜잭션 감독자는 많은 가능한 사용 사례 중 하나이지만 이는 데이터 유효성만 확인하는 게 아니라 비교 대상인 여러 서비스, DB 간의 데이터 차이와 일관성을 복원하기 위해 필요한 작업도 반환한다.

## 10.1 감사는 왜 필요한가?
- 트랜잭션 감독자를 제외하면 배치 감사가 나쁜 관행을 조장할 수 있다고 주장할 수도 있다
- 복제되거나 백업되지 않은 DB에서 데이터 손실로 인해 데이터가 유효하지 않은 상황이라면 데이터 잃는 대신 복제나 백업을 구현해야 한다
- 그러나 복제나 백업은 몇 초 이상 걸릴 수 있으며, 데이터가 성공적으로 복제되거나 백업되기 전에 리더 호스트가 실패할 수 있다.


- 데이터 손실 방지
	- 늦은 복제로 인한 데이터 손실을 방지하기 위한 한 가지 기술은 정족수 일관성, 즉 클라에게 성공 응답을 반환하기 전에 클러스터 과반수의 호스트, 노드에 쓰기다.
	- 카산드라는 성공 응답 반환하기 전에 여러 노드의 Memtable이라는 메모리 내 데이터 구조에 쓰기가 복제된다.
	- 메모리에 쓰는 건 디스크보다 훨씬 빠르다
	- Memtable은 주기적으로 특정 크기에 도달하면 SSTable 디스크로 플러시된다.


- 리더 호스트가 복구되면 데이터가 복구되어 다른 호스트로 복제될 수 있다
- 그러나 구성에 따라 일관성 유지하기 위해 의도적으로 리더 호스트 데이터를 잃을 수 있는 MonoDB 같은 DB에서는 작동하지 않는다


- 서비스가 데이터를 받을 때 유효성을 검증해야 하고, 이미 DB에 저장 후 검증해서는 안된다고 주장할 수도 있다.
- 예를 들어 유효하지 않은 데이터를 받으면 4xx 에러 응답하고, 이 데이터를 유지해서는 안된다


- 감사에 반대하는 또 다른 입장은 유효성 검사를 외부 감사 프로세스가 아니라 데이터베이스와 애플리케이션에서 수행해야 한다는 것
- 애플리케이션은 DB보다 훨씬 빠르게 변하므로 가능한 한 데이터베이스 제약 조건을 사용하는 것을 고려할 수 있다
- 스키마보다 애플리케이션 코드 바꾸는 게 훨씬 쉽다.
- 데이터베이스 제약 조건이 해롭거나, 조기 최적화이거나, 모든 데이터 무결성 요구사항을 포착하지 못한다거나, 요구사항 변화에 맞추기 어렵다는 주장도 있다.
- 일부 회사는 외래 키 제약 조건을 금지한다.


- 위 내용으로도 감지되지 않는 버그나 오류가 있을 수 있다
	- 애플리케이션 코드가 이를 확인할 일이 없어 감지되지 않는다거나 등
	- DB를 외부에서 감사했다면 이런 문제가 없었을 것


- 배치 감사에 대한 일반적인 사용 사례는 큰 파일, 특히 생성 방식을 통제할 수 없는 조직 외부 파일 검증
- 유효하지 않은 값이 발견되어도 데이터가 없는 것보다는 불완전한 데이터라도 있는 게 낫다고 판단해 이를 DB에 저장할 수도 있다


- 중복되거나 누락된 데이터와 같이 배치 감사로만 찾을 수 있는 특정 문제도 있다
- 일부 데이터 유효성 검사는 이전에 수집된 데이터가 필요할 수도 있다. 이상 감지 알고리즘이라든지


## 10.2 SQL 쿼리 결과에 대한 조건문으로 유효성 검사 정의
- 특정 쿠폰 코드가 특정 날짜에 만료된다면, 거래 테이블에 대해 해당 날짜 이후에 해당 코드 ID가 나타난다면 경보를 발생시키기
```sql
SELECT COUNT(*) AS cnt
FROM Transactions
WHERE code_id = @code_id 
AND Date(timestamp) > @date 
AND Date(timestamp) = Curdate() INTERVAL 1 DAY
```
- 하루에 5회 이상 구매 불가 감사
```sql
SELECT user_id, count(*) AS cnt
FROM Transactions
WHERE Date(timestamp) = Curdate() - INTERVAL 1 DAY
GROUP BY user_id
```
- 혹은
```sql
SELECT *
FROM (
  SELECT user_id, count(*) AS cnt
  FROM Transactions
  WHERE Date(timestamp) = Curdate() - INTERVAL 1 DAY
  GROUP BY user_id
) AS yesterday_user_counts
WHERE cnt > 5;
```
- 특정 쿠폰 코드 총 판매 수가 하루에 100 초과할 수 없다
```sql
SELECT count(*) AS cnt
FROM Transactions
WHERE Date(timestamp) = Curdate() - INTERVAL 1 DAY 
AND coupon_code = @coupon_code
```
- 북미 지역 판매를 기록하는 `country_code` 열이 있는 `sales_na`라는 팩트 테이블이 있으면, 각 지역의 국가 코드 목록이 있는 `country_codes`라는 차원 테이블을 만들 수 있다
- 모든 새로운 행이 북미 내 국가의 `country_code` 가지는지 확인하는 주기적 유효성 검사 정의
```sql
SELECT *
FROM sales_na S JOIN country_codes C ON S.country_code = C.id
WHERE C.region != 'NA';
```
- 하루 판매 수가 지난주 같은 요일에 비해 10% 이상 변경되면 경보를 발생시키기
```sql
SELECT COUNT(*)
FROM sales
WHERE Date(timestamp) = Curdate()

SELECT COUNT(*)
FROM sales
WHERE Date(timestamp) = Curdate() - INTERVAL 7 DAY
```

- 아래와 같은 케이스도 상상해볼 수 있다
	- 테이블에 매시간 최소한의 새로운 행이 기록되어야 한다.
	- 특정 문자열 열은 null 값을 포함할 수 없고, 길이는 1~255 사이
	- 특정 문자열 값은 특정 정규 표현식과 일치하는 값을 가져야 한다
	- 특정 정수 열은 음수가 아니어야 한다.

## 10.4 요구사항
- 사용자가 SQL, Hybe, Trino 쿼리를 정의해 주기적으로 DB 테이블의 배치 감사를 수행할 수 있는 시스템을 설계한다
- 요구사항
	- 감사 작업의 CRUD.
		- 분, 시간, 일 등 지정 간격
		- 소유자
		- SQL, HQL, Trino 등 쿼리
		- SQL 쿼리 결과 조건문
	- 실패한 작업은 경보를 발생시켜야 함
	- 과거와 실행 중인 작업의 로그를 볼 수 있어야 하고, 오류 발생 여부와 조건문의 결과를 포함한다. 사용자는 또한 발생한 경보의 상태, 발생한 시간, 해결 여부와 해결된 시간 기록을 볼 수 있어야 한다
	- 감사 작업은 최대 6시간 내에 완료되어야 한다.
	- DB 쿼리는 15분 이내에 완료되어야 한다. 시스템은 장기간 실행되는 쿼리가 있는 작업을 허용하지 않아야 한다.
- 비기능적 요구사항
	- 스케일(Scale) : 10,000개 미만의 작업이나 10,000개의 DB 문이 있을 거라 예상. 작업과 그 로그는 UI를 통해서만 읽히므로 트래픽이 낮다
	- 가용성(Availability) : 다른 시스템이 직접 의존하지 않는 내부 시스템. 고가용성은 필요하지 않다.
	- 보안(Security) : 작업에는 접근 제어가 있다. 작업은 소유자만이 CRUD할 수 있다.
	- 정확성(Accuracy) : 감사 작업 결과는 작업 구성에서 정의한대로 정확해야 한다.

## 10.5 고수준 아키텍처

- 배치 ETL 서비스가 에어플로 서비스거나 비슷하게 작동한다고 가정
- 배치 작업의 파이썬 파일을 저장하고, 정의된 일정에 따라 실행하며, 이러한 작업의 상태와 기록을 저장하고, 감사 조건이 참인지 거짓인지 나타내는 부울 값을 반환한다.
- 사용자는 백엔드를 통해 요청하는 UI와 상호작용한다.
	- 서비스에 요청하여 작업 상태 및 이력 확인 등 배치 감사 작업 CRUD 한다
	- 공유 배치 ETL 서비스는 경보 서비스가 아니므로 경보를 트리거하거나 트리거된 경보의 상태와 기록을 보기 위한 API 엔드포인트가 없다. 사용자는 UI와 백엔드를 통해 경보 서비스에 요청해 이 정보를 볼 수 있다.

- 배치 감사 작업 생성 요청 제출하면 다음 단계가 발생
	1. 입력 값을 템플릿에 대입해 `validation.py` 파일 생성. 템플릿은 짧은 문자열이므로 호스트 메모리에 저장해도 충분하다
	2. 백엔드 서비스는 이 파일과 함께 배치 ETL 서비스에 요청 보낸다. 배치 ETL 서비스는 배치 ETL 작업 생성하고, 이 파일을 저장 후 백엔드 서비스에 200 응답


- 배치 감사 서비스는 본질적으로 공유 배치 ETL 서비스 래퍼
- 감사 작업에는 작업 소유자, 크론 표현식, 데이터베이스 유형, 실행할 쿼리 등의 필드가 있다
- `job_config`에 이런 정보를 저장한다.
- 작업을 소유자에 매핑하는 `owner` 테이블을 만들 수 있고, `job_id`와 `owner_id` 열이 있다.


- 유효성 검사 쿼리는 다양한 유사 SQL 언어로 정의될 수 있다.
- 작업에 실패하거나 실패한 감사가 있으면 경보를 보내기 위해 공유 경보 서비스에 요청

### 10.5.1 배치 감사 작업 실행
- 감사 작업은 구성된 시간 간격으로 주기적으로 실행되고, 두 가지 단계가 있다
	1. 쿼리를 실행
	2. 쿼리 결과로 조건문을 실행


- 주요 확장성 문제는 배치 ETL 서비스와 경보 서비스에서 발생하므로 확장성 논의는 확장 가능한 배치 ETL 서비스와 확장 가능한 경보 서비스 설계하는 것에 관한 것


- 사용자 감사 작업이 주로 SQL 실행하는 유효성 검사 함수로 정의되므로, FaaS 플랫폼 사용하고 내장된 확장성을 활용하는 것도 제안한다.
- 쿼리 실행 시간을 15분으로 제한하거나 결과가 유효하지 않을 때 작업을 중단하는 등 비정상적인 누커리의 안전장치를 만들 수 있다.
- 각 작업 실행 결과는 SQL DB에 저장되므로 사용자는 UI를 통해 이를 접근할 수 있다.

### 10.5.2 경보 처리
- 실패한 감사 경보는 배치 ETL 서비스에서 트리거해야 하는가, 아니면 백엔드에서 트리거해야 하는가?
- 감사 작업을 실행하는 건 ETL 서비스이므로,여기서 한다고 해보자
	- 경보 트리거 요청은 배치 ETL 서비스가 한다
	- 경보 상태 및 기록 조회 요청은 백엔드 서비스에서 한다.
- 즉, 경보 서비스 관련 부담이 두 서비스에 나눠지며 이는 유지보수 부담으로 이어진다.


- 따라서 경보 서비스와의 모든 상호작용은 백엔드 서비스에서 이뤄줘야 한다고 결정할 수 있다
- 배치 ETL 작업은 조건만 확인하고, 이 부울 값을 백엔드 서비스에 보낸다.
- 값이 거짓이면 백엔드 서비스에서 경보를 트리거한다.
- 그러나 이런 접근 방식은 잠재적인 버그를 유발할 수 있다.
- 경보 요청을 생성하고 만드는 백엔드 서비스가 충돌하거나 사용할 수 없게 되면 경보가 보내지지 않을 수 있다
- 이를 방지할 수 있는 방법은 여러 가지가 있다
	- 배치 ETL 서비스에서 백엔드 서비스로의 요청을 차단할 수 있고, 백엔드 서비스는 경보 요청을 성공적으로 보낸 후에만 200을 반환한다. 경보 요청이 이루어지게 Airflow 등 배치 ETL 서비스의 재시도 메커니즘에 의존할 수도 있다. 그러나 이 접근 방식은 배치 ETL 서비스가 본질적으로 여전히 경보 요청을 하고 있고, 두 서비스가 긴밀하게 결합한다는 것을 의미한다
	- 배치 ETL 서비스는 카프카 토픽을 생성할 수 있고, 백엔드 서비스는 이러한 파티션에서 소비하고 각 파티션에서 체크포인트를 수행할 수 있다. 단, 중복 알림이 발생할 수 있으므로 경보 서비스는 경보 중복을 제거할 수 있어야 한다.


- 현재 아키텍처는 로깅과 모니터링을 모두 수행한다.
- 감사 결과를 SQL에 기록한다.
- 이러한 감사 작업을 모니터링하며, 작업이 실패하면 배치 감사 서비스가 경보를 트리거한다.
- 경보만 공유 서비스에서 수행된다.


- 대안 접근 방식은 감사 작업 경과를 SQL과 공유 로깅 서비스 모두에 기록하는 것
- 몇 개의 결과마다 체크포인트를 위해 다른 SQL 테이블을 사용할 수 있다
	- 각 처리 결과를 저장하기 위해 여러 테이블을 사용한다??


![](files/Pasted%20image%2020260607181050.png)

- SQL은 `INSERT INTO IF NOT EXISTS` 등으로 중복을 방지할 수 있다
- 그러나 로깅 서비스는 중복이 발생할 수 있다
- 이에 대한 해결책은 다음과 같다.
	- 중복은 사소하다고 가정하고 로깅 서비스에 작성한다
	- 로깅 서비스가 중복을 처리해야 한다
	- 로깅 서비스에 쓰기 전에 결과가 존재하는지 확인하기 위해 로깅 서비스를 쿼리한다. 이는 로깅 서비스 트래픽을 두 배로 늘린다.

![](files/Pasted%20image%2020260607181056.png)
- 로깅, 경보가 배치 ETL 서비스에서 분리됨
- 배치 ETL 서비스는 경보 서비스 변경을 걸정할 필요가 없고 그 반대도 마찬가지
- 배치 ETL 서비스는 경보 서비스에 요청을 보내게 구성할 필요도 없음

## 10.6 데이터베이스 쿼리 제약
- 다음 이유로 배치 ETL 서비스는 실행할 수 있는 쿼리의 속도와 기간에 제약을 받아야 한다
	- 다양한 데이터베이스 서비스가 공유 서비스다. 길고 비용이 많이 드는 쿼리를 실행하는 사용자는 다른 사용자의 쿼리를 처리할 수 있는 서비스의 남은 용량을 크게 감소시키고 전반적인 지연 시간을 증가시킨다. 호스트의 CPU, 메모리를 쓰고 스레드를 소비한다.
	- 일부 DB는 사용량에 따라 요금을 청구하며, 비싸고 오래 실행되는 쿼리는 비용이 많이 들 것이다.
	- 배치 ETL 서비스는 실행할 쿼리 일정이 있따. 모든 쿼리가 해당 기간 내에 실행될 수 있게 해야 한다. 예를 들어 시간별 쿼리는 1시간 이내에 완료되어야 한다.


- 쿼리 작성하고 이를 백엔드에 제출하면 사용자의 쿼리 정의를 구문 분석하는 기술을 구현할 수도 있다

### 10.6.1 쿼리 실행 시간 제한
- 비용이 많이 드는 쿼리를 제한하는 가장 간단한 방법은 작업 구성을 생성하거나 편집할 때는 쿼리 시간을 10분으로 제한하고, 실행 중일 때는 15분으로 제한하는 것
- 즉, 작업을 만들 때는 사용자가 반드시 10분 이내에 쿼리가 완료되는지 확인해야만 한다.


- 대안으로 비차단/비동기 경험을 제공할 수도 있다
- 일단 저장 및 실행을 허용하고 비동기로 10분 이내에 성공했는지 여부를 알려준다
- 이 UX의 단점은 소유자가 쿼리를 변경하지 않을 수도 있다는 것이다.


- 쿼리 실행이 15분을 초과하면 쿼리를 종료하고, 소유자가 쿼리를 편집하고 검증할 때까지 작업을 비활성화하며, 소유자에게 높은 긴급도의 경보를 트리거한다.
- 쿼리 실행이 10분을 초과하면 낮은 긴급도의 경보를 트리거해 향후 15분을 초과할 수 있음을 경보한다.

### 10.6.2 제출 전 쿼리 문자열 확인
- 쿼리 실행을 기다리고 저장을 허용하면 UX 적으로 불편하다.
- 즉각적으로 유효하지 않거나 비용이 많이 드는 쿼리를 거부하면 편할 것이다
- 이런 유효성 검사에는 다음이 포함될 수 있다
	- 전체 테이블 스캔을 허용하지 않는다. 파티션 키가 포함된 테이블에서만 쿼리를 허용하고, 쿼리에는 파티션 키 필터가 포함되어야 한다. 더 나아가 파티션 키 값의 수를 제한할 수도 있다.
	- 비용이 상당시 많이 들 수 있는 JOIN 쿼리는 허용하지 않는다
	- 사용자가 쿼리 정의 후, 실행 계획을 표시해 사용자가 실행 시간을 최소화하게 쿼리를 조정할 수도 있다.

### 10.6.3 초기 사용자 훈련
- 사용자는 이런 제약 사항을 초기에 교육받아야 한다. 그래야 제약 사항에 적응하는 법을 배울 수 있다
- 사용자에게 이런 제약조건을 안내하는 좋은 UX와 유익한 문서를 제공해야 한다.
- 이런 제약 조건은 첫 출시 후 몇 달 뒤에 추가하는 것보단 초기 릴리즈에서 정의하고 설정하는 게 좋다.
- 초기에 이런 쿼리를 허용했다면 이런 제약에 저항하고 반발할 수 있다.

## 10.7 과도한 동시 쿼리 방지
![](files/Pasted%20image%2020260607181103.png)
- 배치 ETL이 동시에 실행할 수 있는 쿼리 수를 제한해야 한다
- 동시 쿼리 수가 제한에 근접하면 경보를 트리거할 수 있다
- 각 쿼리가 시작하기 전 대기 시간을 모니털이하고, 대기 시간이 30분이나 그 외 결정한 기준값을 초과하면 낮은 긴급도의 경보를 트리거할 수 있다
- 용량을 추정하기 위해 부하테스트를 할 수도 있다.


- 데이터베이스는 공유 서비스이므로 동시 쿼리 수에 대한 구성 제한 같은 공통 관심사는 데이터베이스 쿼리 서비스에 저장해야 한다.
- 또 다른 최적화는 배치 ETL 서비스가 데이터베이스 쿼리를 실행하기 전에 백엔드 서비스를 통해 경보 서비스에 쿼리해 해결되지 않은 경보가 있는지 확인하는 것
- 그렇다면 감사 작업을 진행할 필요가 없다.

## 10.8 데이터베이스 스키마 메타데이터의 사용자
- 사용자의 쿼리 작성을 돕기 위해 스키마 메타데이터에서 작업 구성을 자동으로 도출할 수 있다
- WHERE 필터는 보통 파티션 열에 정의되므로 UI는 사용자에게 이러한 열을 제안하는 쿼리 템플릿을 제시하거나 최신 파티션만 테스트하는 쿼리를 작성하게 제안할 수 있다
- 기본적으로 새 파티션이 감사를 통과하면 서비스는 해당 파티션에 대해 추가 감사를 예약하지 않아야 한다
- 단, 감사 자체에 버그가 있었을 수 있으므로 사용자가 수동으로 감사를 실행할 수는 있어야 한다.


- 테이블에 새로운 행이 얼마나 자주 추가되는지에 대한 갱신 주기 SLA가 있을 수 있다.
- 이는 데이터의 최신성과 관련된 개념이다.
- 데이터가 준비되기 전에 테이블 감사를 수행해서는 안 된다.


- 테이블과 관련된 사건을 기록할 수도 있다
- 테이블 소유자나 서비스는 특정 테이블에 문제가 발생하고 있음을 데이터베이스 메타데이터 플랫폼에 업데이트할 수 있다.
- 쿼리 서비스는 이 테이블을 쿼리하는 모든 사람이나 서비스에 경보할 수 있다
- 사용자는 이런 메타데이터 변경사항을 구독하거나 경보를 받을 수도 있다.


- 배치 ETL 서비스는 스키마 변경을 모니터링하고, 그에 따라 대응할 수도 있다
- 열 이름이 변경되면 해당 영 이름을 포함하는 감사 작업 구성 쿼리 문자열에서 열 이름을 업데이트해야 한다.
- 열이 삭제되면 관련된 모든 작업을 비활성화하고 소유자에게 알려야 한다.

## 10.9 데이터 파이프라인 감사
![](files/Pasted%20image%2020260607181108.png)

- 위는 Airflow DAG 데이터 파이프라인이다.
- 파이프라인 이름과 수준 필드가 포함될 수 있고, 이는 `job_config` 테이블에 추가 열로 넣을 수 있다.


- 특정 감사 작업이 실패하면 서비스는 다음을 수행해야 한다.
	- 상위 작업이 실패하면 감사 작업 실행하는 것이 무의미하므로 하위 감사를 비활성화해 리소스를 절약한다.
	- 이 테이블 쿼리를 포함하는 다른 작업과 그 하위 작업도 비활성화한다
	- 모든 비활성화된 작업의 소유자와 그 모든 하위 작업 소유자에게 높은 긴급도의 경보를 트리거한다.


- 또한 이 테이블에 문제가 있음을 메타데이터 플랫폼에 업데이트해야 한다.
- Airflow는 이미 사용자가 트리거 규칙을 구성해 모든 종속성이나 최소한 하나의 종속성이 성공적으로 실행을 완료할 때 각 작업이 실행되게 허용한다.
- 새로운 배치 ETL 서비스 기능은 Airflow와 다른 워크플로우 관리 플랫폼의 향상된 기능이다.


- 이 모든 것은 배치 ETL 서비스를 공유 서비스로 일반화해 조직 전체의 배치 ETL 작업에 이 기능을 제공할 수 있음을 시사한다.

## 10.10 로깅, 모니터링, 경보
- 다음 로그는 사용자에게 유용할 수 있고, UI에 표시할 수 있다
	- 시작, 진행 중, 성공, 실패 등의 현재 작업 상태와 이 상태가 기록된 시간
	- 실패한 배치 ETL 서비스 데이터베이스 쿼리. 경보에는 쿼리 시간 초과나 쿼리 실행 오류 등 실패 이유도 포함
	- 쿼리 실행 시간이 설정된 기준을 초과한 경우
	- 상위 작업이 실패한 경우
	- 백엔드 엔드포인트의 1초 P99, 4xx, 5xx 응답
	- 외부 서비스 요청의 1초 P99, 4xx, 5xx 응답
	- 부하 테스트를 통해 결정된 부하 한도보다 높은 요청률로 정의되는 높은 트래픽
	- 높은 CPU, 메모리, I/O 사용률
	- 공유 서비스를 사용하지 않고 자체 SQL 서비스 관리할 때 SQL 서비스의 저장소 사용률

### 10.11.1 데이터 센터 간 일관성 감사
- 동일한 데이터를 여러 데이터센터에 저장할 수 있다
- 데이터 센터 간 데이터 일관성을 보장하기 위해 데이터베이스 배치 감사 서비스는 데이터를 비교하는 샘플링 테스트를 실행할 수 있는 기능을 제공

### 10.11.2 업스트림과 다운스트림 데이터 비교
- 한 테이블에서 다른 테이블로 복사하는 경우 데이터 일관성 보장 감사 작업

## 10.12 기타 논의 가능한 주제
- 확장 가능한 배치 ETL 서비스나 확장 가능한 경보 서비스를 설계. 둘 다 카프카와 같은 분산 이벤트 스트리밍 플랫폼 필요
- `validation.py.template` 처럼 템플릿에서 Airflow 파이썬 작업을 생성하는 함수를 코딩. 각 쿼리에 별도의 Airflow 작업을 만든다. 단, 이는 코딩 테스트 질문이다
- 감사 작업 경보는 문제를 알릴 수는 있으나 문제의 원인을 어떻게 발견하고 해결할 수 있는지는 별도로 설명되지 않았다. 테이블 소유자는 이 문제를 어떻게 해결할 수 있는가? 감사 서비스를 개선하거나 도움을 줄 수 있는 다른 방법은 무엇인가?
- 특정 감사 작업은 한 번 실행하면 실패했다가 소유자가 디버깅을 위해 동일한 쿼리를 실행하면 해결될 수도 있다. 소유자는 이런 작업을 어떻게 해결할 수 있고, 서비스는 이를 지원하기 위해 어떤 로깅이나 기능을 제공?
- 동일하거나 유사한 감사 작업 어떻게 찾고 중복 제거?
- 배치 서비스는 다수의 경보를 보내는데, 어떻게 하면 경보 중복을 제거? 이 로직의 어느 부분을 배치 서비스에 구현하고 어느 부분을 공유 경보 시스템에 구현?
- 크론이 아니라 특정 이벤트에 의해 테스트가 트리거되도록 할 수도 있다. 각 쿼리 후 변경된 행 수를 추적하고 이 수를 합산한 뒤 지정된 수의 행이 변경된 후에 테스트를 실행. 테스트를 트리거할 수 있는 가능한 이벤트와 그 시스템 설계에 대해 논의할 수도 있다.
