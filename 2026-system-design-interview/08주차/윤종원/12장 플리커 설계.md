# 12. 플리커 설계
- 이미지 공유 서비스 설계
- 파일, 이미지 공유 외에도 접근 제어, 댓글, 즐겨찾기 등 메타데이터를 다른 사용자에게 추가할 수 있음


- 이미지와 비디오 공유하고 상호 작용하는 건 모든 소셜 애플리케이션의 기본 기능이자 일반적인 면접 주제

## 12.1 사용자 스토리와 기능 요구사항
- 사용자는 다른 사용자가 공유한 사진을 볼 수 있다. 이 사용자를 뷰어라고 한다
- 앱은 너비 50px 썸네일 생성하고 표시한다.
- 사용자는 사진을 업로드할 수 있다. 이 유저를 공유자라고 한다.
- 공유자는 자기의 사진에 대한 접근 제어 설정 가능. 개별 사진 단위 혹은 뷰어 단위. 여기서는 단순화를 위해 뷰어 단위라 가정
- 사진에는 미리 정의된 메타데이터가 있다. 위치, 태그 등
- 동적 메타데이터의 예로는 파일 읽기 권한이 있는 뷰어 목록 등
- 사진에 댓글 달 수 있고, 사진에 댓글 기능 on off 할 수 있다. 새 댓글 알림 받을 수 있다
- 사용자는 사진에 즐겨찾기 할 수 있다
- 사용자는 사진 제목과 설명으로 검색 가능
- 사용자는 API로 사진 다운로드 가능
- 개인화 간단히 논의

## 12.2 비기능적 요구사항
- 예상되는 사용자 수, API 다운로드 수?
	- 시스템은 확장 가능해야 함
	- 전 세계에 분산된 10억 사용자 지원
	- 1%가 매일 10개의 고해상도 이미지 업로드 가정 -> 매일 1PB 업로드, 10년 동안 3.65EB
	- 평균 트래픽은 초당 1GB 이상, 급증 고려하면 초당 10GB
- 업로드 직후 사진 이용 가능? 삭제는? 개인정보 설정 변경은?
	- 사진이 전체 사용자에게 표시되는 데는 몇 분 걸릴 수 있다. 댓글도 마찬가지. 최종 일관성 허용
	- 개인정보 설정은 그보다 빠르게 적용되어야 한다. 사진이 바로 삭제될 필요는 없지만 몇 분 내에 모든 사용자가 접근할 수 없어야 한다
- 고해상도 사진은 높은 네트워크 속도가 필요. 비용이 많이 든다. 비용 통제는?
	- 한 번에 하나의 고해상도 사진만 다운로드 가능. 여러 저해상도 썸네일은 동시에 다운로드 가능.
	- 파일 업로드는 한 번에 하나씩

- 다른 나머지
	- 99.999% 높은 고가용성. 업로드 및 다운에 중단이 없어야 한다
	- 썸네일 다운로드에는 P99 1초의 높은 성능, 낮은 지연 시간 필요. 고해상도 사진에는 이것이 필요 없다
	- 업로드에는 높은 성능이 필요하지 않다


- 썸네일에 css img 태그로 크기 조정할 수 있지만 이는 원본 크기는 그대로라 확장성이 없다
- 썸네일 생성은 두 가지 가능
	- on demand 생성
		- 썸네일 생성 비용이 낮다면 확장성이 있다
		- 그러나 전체 이미지 파일은 수십 MB이고, 단일 요청에서 10개 이상의 썸네일 그리드 요청 -> 1초 P99 유지하기 위해 1초보다 훨씬 더 짧은 시간 내에 100MB 처리 필요
		- 뷰어가 썸네일 스크롤하면서 이런 요청을 여러 번 할 수 있다
		- 이는 비용이 매우 많이 들고, 실현 가능하지 않다
		- 매초 처리, 저장 서비스 간 수 GB 전송은 네트워크 지연으로 인해 P99 1초 달성이 어렵다.
	- 미리 생성
		- 업로드 직후 썸네일 생성하고 저장 후 썸네일 제공
		- 몇 KB 크기에 불과하므로 저장 비용이 낮고, 클라 캐시도 가능하다

## 12.3 고수준 아키텍처
![](files/Pasted%20image%2020260621152652.png)
- 이미지 파일, 메타데이터를 위한 CDN
	- 대부분 서드파티
- 이미지 파일 업로드를 위한 별도의 분산 파일 저장 서비스가 필요
	- 이를 파일 저장 서비스라고 한다.


- CDN SLA에 따라 다양한 데이터센터에 이미지 복제하는 데 최대 몇 시간 걸릴 수 있다
- 그 사이 많은 뷰어가 다운로드 하려고 하면 이미지 다운 속도가 느려진다
- CDN 이미지 업로드 속도가 허용할 수 없을 정도로 느릴 수 있다. 여러 공유자가 동시에 이미지 업로드하는 걸 CDN이 지원하지 못할 수도 있다. 필요에 따라 확장해 높은 업로드, 쓰기 트래픽 처리해야 한다
- CDN 업로드 후 파일 저장소에서 지우거나 백업할 수도 있다. CDN SLA 완전히 신뢰할 수 없고, CDN 중단이 발생하면 파일 저장소 서비스를 백업으로 사용하고 싶을 수도 있기 때문이다. 보통 CDN에는 보존 기간이 있고, CDN과 계약을 끊어야하는 등 다양한 상황이 있을 수 있다.

## 12.4 SQL 스키마
```sql
CREATE TABLE Image (
    cdn_path VARCHAR(255) PRIMARY KEY COMMENT 'CDN의 이미지 파일 경로.',
    cdn_photo_key VARCHAR(255) NOT NULL UNIQUE COMMENT 'CDN이 할당한 ID.',
    file_key VARCHAR(255) NOT NULL UNIQUE COMMENT '파일 저장 서비스가 할당한 ID. CDN에 업로드한 후 파일 저장 서비스에서 이미지를 삭제한다면 이 열이 필요하지 않을 수 있다.',
    resolution ENUM('thumbnail', 'hd') COMMENT '이미지가 썸네일인지 고해상도인지 구분한다.',
    owner_id VARCHAR(255) NOT NULL COMMENT '이미지 소유자의 ID.',
    is_public BOOLEAN NOT NULL DEFAULT 1 COMMENT '이미지가 공개인지 비공개인지 나타낸다.',
    INDEX thumbnail (resolution, owner_id) COMMENT '특정 사용자에 속한 썸네일이나 고해상도 이미지를 빠르게 찾을 수 있게 해상도와 사용자 ID 복합 인덱스.'
) COMMENT = '이미지 메타데이터.';

CREATE TABLE ImageDir (
    cdn_dir VARCHAR(255) PRIMARY KEY COMMENT '사용자에게 할당된 CDN 디렉터리.',
    user_id INTEGER NOT NULL COMMENT '사용자 ID.'
) COMMENT = '각 공유자에게 할당된 CDN 디렉터리를 기록한다.';
```

- Image 테이블에는 이미지 메타데이터 포함
- 각 공유자에게 고유한 CDN 디렉터리 할당할 수 있고, 이는 Image Dir 테이블로 추적한다.

- 공유자가 뷰어에게 자신의 사진을 볼 수 있는 권한을 부여하거나 뷰어가 사진을 즐겨찾기 하게 할 수 있다
- 두 개의 테이블 사용하는 대신 Share 테이블에 `is_favorite`라는 Boolean 칼럼 정의하는 방법도 있지만 이는 불필요한 저장 공간을 사용하는 희박한 칼럼이 된다.

```sql
CREATE TABLE Share (
  id INT PRIMARY KEY,
  cdn_photo_key VARCHAR(255),
  user_id VARCHAR(255)
);

CREATE TABLE Favorite (
  id INT PRIMARY KEY,
```

## 12.5 CDN에서 디렉터리와 파일 구성하기
- 디렉터리 계층 구조는 사용자 -> 앨범 -> 해상도 -> 파일이 될 수 있다
- 최근 파일에 더 관심이 있을 수 있으므로 날짜도 고려할 수 있다
- 각 사용자는 자신의 CDN 디렉터리를 가진다.
- 앨범을 만들 수 있고, 앨범에는 0개 이상의 사진이 있다. 1:N
	- 즉, 각 사진은 하나의 앨범에만 속함
- 앨범이 없는 사진을 default라는 앨범에 배치할 수 있다.


- 앨범 디렉터리는 다양한 해상도의 여러 이미지 파일을 각각 고유한 디렉터리에 저장하고, JSON 이미지 메타데이터 파일도 저장할 수 있다.
- CDN Path 값 템플릿은 `<album_name>/<resolution>/<image_name.extension>` 이다.

### 12.6.1 클라이언트에서 썸네일 생성하기
- 백엔드 리소스 절약. 네트워크 트래픽 영향 거의 x
- 업로드 전에 클라는 썸네일이 이미 CDN에 업로드 되었는지 확인할 수 있다.

![](files/Pasted%20image%2020260621153900.png)
1. 썸네일 생성
2. 두 파일을 폴더에 넣고 Gzip이나 Brotli같은 인코딩으로 압축. 네트워크 트래픽 절약하지만 디렉터리 압축 풀기 위해 CPU, 메모리 사용
3. POST 요청으로 압축된 파일을 CDN에 업로드. 요청 본문은 이미지 수, 해상도 설명하는 JSON
4. CDN에서 필요한 디렉터리 만들고, 압축 파일 해제하고 디스크에 기록. 그러고 다른 데이터센터에 복제


- 단점
	- 클라 기기 제어할 수 없고, 버그 재현도 어렵다
	- 서버에 비해 클라에서 발생할 수 있는 더 많은 실패 시나리오 예상해야 함
	- 클라 디스크 용량 부족, 네트워크 연결 끊김 등
	- 클라에서 발생하는 이런 상황 대부분은 제어할 수 없다
	- 많은 요인이 영향을 미칠 수 있어 로깅 결정도 어렵다
	- 또 그 외에 다양한 요인
		- 다양한 클라 대응 필요
		- 디바이스 문제로 썸네일 생성이 너무 느릴 수 있음
		- 클라 OS의 버그나 보안 문제로 손상된 파일을 업로드할 수도 있다거나 등.
		- 클라가 보안 모범 사례를 따르고 있다고 보장할 수 없다
		- 악의적인 활동 방지가 어렵다
		- 차단된 포트, 호스트, VPN 등이 방해
		- 네트워크 연결 불안정
		- 썸네일 저장할 기기 저장 공간 부족
		- 썸네일 생성으로 인해 로컬 쓰기 권한 등이 필요


- 이런 문제는 대부분 소수의 사용자에게만 영향을 미치므로, 이를 해결하기 위해 자원을 투자할 가치가 없다고 느낄 수 있지만 이런 문제가 누적되면 상당수에게 영향을 줄 수도 있다.

![](files/Pasted%20image%2020260621162948.png)

- 지루하고 긴 소프트웨어 릴리스 수명 주기
	- 클라 사이드 처리는 버그 확률이 높고, 해결 비용이 높다. 따라서 배포 전 많은 테스트를 거쳐야하고 이는 개발 속도를 늦춘다
	- 이는 릴리스가 느리고 지루해지며 각 릴리스에는 많은 코드 변경이 포함된다.

![](files/Pasted%20image%2020260621163702.png)

- 릴리스가 느린 상황에서 새 릴리스의 버그를 방지하는 또 다른 접근 방식은 이전 코드를 제거하지 않고 새 코드를 포함하는 것
	1. 새 함수를 추가. 동일한 입력으로 새 함수 실행하지만 새 함수 대신 이전 함수의 출력을 계속 사용. 새 함수의 사용을 try catch로 감싸 예외가 애플리케이션을 중단하지 않게 한다. catch 문에서 예외를 로깅하고 로그를 로깅 서비스로 보내 문제를 해결하고 디버그할 수 있게 한다.
	2. 함수를 디버그하고 더 이상 버그가 관찰되지 않을 때까지 새 버전을 릴리스한다.
	3. 이전 함수에서 새 함수를 사용하게 코드를 전환. 예외가 발생하면 백업으로 이전 함수를 사용. 이 버전을 릴리스하고 문제를 관찰. 문제가 있다면 롤백
	4. 새로운 함수가 충분히 안정화 되었다고 판단되면 기존 함수를 코드에서 삭제. 이런 개선은 코드가독성 + 유지보수성 높이기 위함


- 이 접근 방식의 한계는 이전 버전과 호환되지 않는 코드 도입이 어렵다는 것
- 또 다른 단점은 코드베이스가 더 커지고 유지보수가 어려워진다는 것
- 개발팀이 이 프로세스를 따른다는 규율이 필요하다.
- 추가로, 앱의 크기를 증가시킨다.

### 12.6.2 백엔드에서 썸네일 생성하기
- 서버에서 생성하는 건 많은 엔지니어링 노력 + 리소스가 필요하지만 다른 서비스와 동일한 언어 + 도구로 만들 수 있다
- 전자의 비용이 후자보다 더 크다고 판단할 수 있다.


- 크게 3단계
	1. 업로드 전 중복 업로드 확인
	2. 파일을 파일 저장 서비스 + CDN에 저장
	3. 썸네일을 생성하고 파일 저장 서비스와 CDN에 업로드


- 백엔드에 업로드되면 파일을 저장 서비스와 CDN에 기록 후 썸네일 생성하는 스트리밍 작업 트리거
- 파일 저장 서비스의 주 목적은 CDN 업로드 버퍼 역할. 데이터센터 내 호스트 간 복제는 할 수도 있지만 데이터센터 간 복제는 구현하지 않는다. 데이터 센터 중단 시 급하다면 CDN 파일을 복구에 사용할 수도 있다
- 확장 가능한 이미지 파일 업로드는 파일 업로드 단계를 비동기적으로 처리할 수 있으므로 사가 패턴 접근 방식 사용할 수 있다.

### 12.6.3 서버 사이드와 클라이언트 사이드 생성 모두 구현하기
- 서버 사이드 생성 구현 후, 모든 클라 썸네일을 생성할 수 있다
- 각 클라 유형에 클라 사이드 생성을 구현해 클라 사이드 생성 이점을 실현한다
- 클라 생성을 시도 후 실패하면 서버 생성을 고려한다
- 모든 실패 시나리오를 고려하지 않아도 되고, 클라 사이드 생성을 반복적으로 개선할 수 있다.


- 이는 서버만 하는 것보다 더 복잡하고 비용도 많이 들지만, 더 저렴하고 쉽다.

## 12.7 이미지와 데이터 다운로드하기
- 공유자의 썸네일에 대한 뷰어 요청은 아래처럼 처리된다
	1. Share 테이블 쿼리해 뷰어가 이미지 볼 수 있게 허용한 공유자 목록 가져온다
	2. Image 테이블 쿼리해 썸네일 해상도 이미지의 CDN Path 얻는다. CDN path와 CDN에서 읽기 위한 임시 OAuth2 토큰 반환한다
	3. 클라가 CDN에서 썸네일을 반환한다. 다운 권한 확인을 위해 CDN은 토큰 인증 메커니즘을 사용할 수 있다.

- 동적 콘텐츠는 업데이트되거나 삭제될 수 있으므로 CDN이 아니라 SQL에 저장
	- 사진 댓글, 프로필 정보, 사용자 설정 등
- 인기 있는 썸네일과 전체 해상도 이미지는 캐싱할 수도 있다.
- 이미지 즐겨찾기하는 경우 이미지 불변성 활용해 썸네일, 전체 해상도 이미지를 클라 캐싱할 수도 있다.
### 12.7.1 썸네일 페이지 다운로드하기
- 썸네일을 페이지네이션한다고 했을 때
	- 각 페이지에 1~10번, 11~20번 있을 때
	- 두 번째 페이지로 갔을 때 어떻게 10~19번 대신 11~20번 썸네일을 보여줄 수 있도록 보장할 수 있을까?
	- 한 가지 방법은 `GET thumbnails?page=<page>&page_version=<page_version>` 같이 페이지네이션에 버전 부여하는 것. 생략되면 최신 버전으로 대체
	- 이 요청 응답은 `page_version` 포함하므로 사용자는 필요에 따라 후속 요청에도 동일한 버전을 사용할 수 있다.
	- 1페이지로 돌아가 `page_version` 생략하면 최신 1페이지의 썸네일 표시된다


- 단, 이는 썸네일 목록이 시작 부분에 추가되거나 삭제될 때만 작동
- 중간에 썸네일 추가되거나 삭제되면 새 썸네일을 못 보거나 삭제된 썸네일을 계속 보게된다
- 더 나은 기법은 클라가 첫 번째 항목이나 마지막 항목을 백엔드에 전달

## 12.9 기타 서비스

### 12.9.1 프리미엄 기능
- 공유자는 사진에 저작권이 있으면 다운 + 다른 곳 사용할 때 비용 지불해야 한다고 명시하거나
- 다른 사용자에게 사진을 판매할 수 있는 비즈니스나
	- 판매를 기록하고 사진 소유권 추적 필요
- 더 나은 비즈니스 결정을 위해 판매 지표, 대시보드, 분석 기능 등을 제공
- 판매자에게 어떤 종류의 사진 판매하고 어떻게 가격을 책정할 수 있는지 추천 시스템 제공 등


- 무료 계정에는 1,000장 사진 제한, 다양한 구독 플랜에는 더 큰 용량 제공한다거나 등

### 12.9.2 결제와 세금 서비스
- 거래 + 결제 관리하기 위해선 결제 + 세금 서비스 필요
- 세금도 판매세, 소득세, 법인세 등 다양함

### 12.9.3 검열/콘텐츠 조정
- 콘텐츠 조정을 위한 시스템 설계가 필요
- 수동 + 자동 수행 가능
- 수동 방법에는 신고하면 운영팀이 처리
- 휴리스틱 혹은 머신러닝 접근 방식도 고려할 수 있음

### 12.9.4 광고
- 클라에게 광고를 표시할 수도 있다
- 일반적인 방법은 서드파티 광고 SDK 붙이기
- 또 다른 방법은 클라 내부에 공유자를 위한 광고 표시 시스템 설계
- 사진 판매 늘리기 위해 광고 표시하고 싶을 수도 있음
- 혹은 광고 제거 패키지 제공

### 12.9.5 개인화
- 사용자 데이터 기반으로 개인화된 광고, 검색, 콘텐츠 추천 등
