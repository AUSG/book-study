# 시스템 설계 개념 둘러보기
## 시스템의 다양한 서비스 확장 방식
1. 소규모 초기 배포(프론트 - 백엔드 - DB)
2. GeoDNS로 클라를 가장 가까운 데이터 센터로 라우팅(IP는 ISP, OS, 브라우저 등 다양한 수준에서 캐시될 수 있음)
3. 캐싱 서비스 추가(Redis)
4. CDN을 통한 빠른 콘텐츠 다운로드
- CI/CD/IaC를 활용한 개발 작업 자동화를 사용할 수 있음
- 공유 서비스
    - 검색, 모니터링, 로깅 등의 다양한 서비스 간 공유하는 기능을 제공해야 하는 경우들이 발생
    - API Gateway를 사용하여 공유 서비스의 기능을 제공하면서 적절한 서비스로 라우팅하거나 
    - 서비스 메시를 사용(정책을 구성하는 사이드카를 띄워 모든 서비스 요청과 응답이 사이드카를 통해 왔다갔다 하게 구성 가능)
- 배치 및 스트리밍 ETL
- 클라우드 vs 베어메탈
- 서버리스

# 일반적인 시스템 설계 면접 흐름
1. 요구사항 명확화와 트레이드 오프 논의
- 사용자 범주/역할 고려
- 사용자 범주를 바탕으로 확장성 요구사항을 명확히 하기
- 어떤 사용자 어떤 데이터에 접근 가능?
- 사용자, 역할마다의 유저 스토리 논의하기
2. API 명세 초안 작성
3. 사용자와 데이터 간의 연결과 처리
4. 데이터 모델 설계
- 데이터베이스 공유/비공유
- 동시 업데이트 충돌 방지
5. 로깅, 모니터링, 경보
6. 검색창
7. 기타
- 애플리케이션 유지보수와 확장
- 다른 유형 사용자 지원
- 대안적 아키텍쳐 결정
- 사용성과 피드백
- 엣지 케이스와 새로운 제약 조건
8. 면접 후 회고와 평가

# 비기능적 요구사항
비기능적 요구사항: 시스템의 입출력 이외의 요구사항
1. 확장성
- 시스템이 부하를 비용 효율적으로 지원할 때 하드웨어 리소스 사용을 쉽고 번거롭지 않게 조정할 수 있는 능력
- 수평 확장 vs 수직 확장
- 상태 비저장 서비스와 상태 저장 서비스
- 로드 밸런싱
    - 4계층 vs 7계층
    - 고정 세션
    - 세션 복제
    - 로드 밸런싱 vs 리버스 프록시

2. 가용성
- 시스템이 요청을 수학하고 원하는 응답을 반환할 수 있는 시간의 백분율
- 바로 높은 가용성을 전제하지 말고 진짜 필요한지를 먼저 논의
    - 즉각적인 응답 필요시 동시 프로토콜
    - 높은 가용성이 필요 없는 상황의 예시로 캐싱 서비스가 있음

3. 내결함성
- 일부 구성 요소가 실패해도 시스템이 계속 작동할 수 있는 능력 / 다운타임 발생 시 영구적인 데이터 손실 등을 방지하는 능력
- 복제와 중복: 중복 인스턴스/복사본 3개 이상
- 서킷 브레이커: 실패할 가능성이 높은 작업을 반복적으로 시도하는 것을 막는 매커니즘 - 리소스 낭비를 줄이고 외부 의존성 과부하를 막음
- 지수 백오프와 재시도
- 다른 서비스의 응답 캐싱(에러를 내는 대신 빈값이나 기본값으로 반환)
- 체크포인팅: 어느 데이터부터 집계하면 되는지 등에 대해 체크포인트를 항상 작성
- 데드 레터 큐: 실패 요청을 큐에 넣고 다시 시도
- 벌크 헤드: 
    - 시스템의 자원을 여러 격리된 구역으로 나눠 한 부분의 장애가 다른 곳으로 전파되지 않게 함
    - 스레드 풀, 커넥션 풀, 프로세스 , 큐, 컨테이너 등의 자원을 서비스 별로 분리하여 관리할 수 있게 함

4. 성능/지연 시간과 처리량
5. 일관성
- ACID는 네트워크 분할일 발생하면 사용 불가
- 전체 메시
    - 모든 노드가 다른 모든 노드와 직접 연결되는 구조
    - 단순하고 직접적
    - 연결 관리 비용이 기하 급수적으로 늘어남, 장애 감지, 재연결 로직 복잡
- 조정 서비스
    - 클러스터의 메타데이터와 중요한 결정을 관리
    - 리더 선출, 분산 락, 서비스 디스커버리, 설정 관리 등에 필요
- 분산 캐시
    - 주로 읽기 성능과 원본 저장소 보호를 위해 사용
- 가십 프로토콜
    - 노드들이 서로 무작위로 정보를 주고받으며 클러스터 전체에 상태를 퍼뜨리는 방식
    - 중앙 서버 하나가 모든 노드에게 상태를 알려주는 대신, 노드들이 서로 조금씩 정보를 교환
    - 중앙 조정자 없이 동작할 수 있음 / 노드 수가 많아도 비교적 잘 확장
    - 정보가 즉시 모든 노드에 반영되지는 않음 / 전파 시점을 예측하기 힘듦
- 무작위 리더 선택

6. 정확성: 데이터 값이 정확하고 근사치가 아님을 의미
