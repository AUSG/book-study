# 2장. 일반적인 시스템 설계 면접 흐름

### 주요 평가 요소

- 관련 세부사항을 고려하고 평가하는 능력
- 고수준 아키텍처와의 관계
- 모든 구성 요소의 저수준 구현 세부 사항

### 일반적인 진행 순서

1. 요구사항을 명확히 한다. 트레이드오프를 논의한다.
2. API 명세 초안을 작성한다.
3. 데이터 모델을 설계한다. 가능한 **분석**을 논의한다.
4. 실패 설계, 성능 저하, 모니터링, 경보를 논의한다. — 병목 현상, 부하 분산, 단일 장애 지점 제거, 고가용성, 재해 복구, 캐싱 등의 주제 포함
5. 복잡성과 트레이드오프, 유지보수와 폐기 프로세스, 비용을 논의한다.

\*_동적으로 위 순서는 변경될 가능성이 높음_

## 1. 요구사항 명확화, 트레이드오프 논의

- 30초~1분 정도 소요
- 시스템의 전반젇인 목적과 큰 그림의 비즈니스 요구사항에 어떻게 부합하는지 논의
- 빠르게 브레인스토밍하여 기능적 요구사항 목록을 작성하고 논의할 것

### 공통 기능적 요구사항

1. 사용자 범주/역할
   1. 시스템 사용 대상, 다양한 사용자 범주의 조합
   2. 서비스의 성격이 기술적 or 비기술적인지
   3. 사용자 역할 나열
   4. 수치에 주의를 기울일 것
   5. 사용자(-운영자) 간 커뮤니케이션이 있는지
   6. 국제화(i18n), 현지화(L10n) 지원, 국가/지역 언어, 우편 주소, 가격(다중 통화 지원)이 필요한지
2. 사용자 범주 → 확장성 요구사항
   - DAU → 일/시간당 요청 빈도 추정
3. 인증, 인가 역할과 메커니즘 / 데이터 특성
   - 어떤 사용자가 어떤 데이터에 접근할 수 있는가
   - API 응답의 본문 내용
   - 데이터 검색 주기 (싥시간, 월간 보고서, …)
4. 검색에 관해 가능한 사용 사례
5. 분석, 실험, 머신러닝 요구사항
6. 의사코드 함수 시그니처 작성 → 사용자 스토리와 일치시키는 과정
   - 어떤 요구사항이 필요하고,
   - 어떤 것이 범위를 벗어나는지

## 2. API 명세 초안 작성

- 5분 미만 내
- 엔드포인트 작성 전에 미리 기능적 요구사항을 명확히 해야 함
- 공통 API 엔드포인트
  - health check
  - 회원가입, 로그인 (일반적으로 OIDC 인증 프로토콜이 사용됨)
  - 사용자 정보 CRUD, 부적절 컨텐츠 신고/리포트 채널 제공

## 3. 사용자-데이터 간의 연결과 처리

시스템을 다양한 추상화 수준으로 분해하는 시스템 아키텍처 다이어그램 기법인 **C4 모델**을 적용해볼 수 있다

Step1. 각 사용자 유형, 시스템을 그리고, 이들 간의 연결을 그린다.

Step2. 요청 처리와 저장소를 분리하고, 비기능적 요구사항을 기반으로 한 다양한 설계를 만든다. 이때 공유 서비스를 고려한다.

Step3. 시스템을 구성 요소(라이브러리, 서비스 등)로 구분하고, 연결을 그린다. 이때 로깅, 모니터링, 경보, 보안을 고려한다.

Step4. 전반적인 시스템 설계를 요약하고, 새로운 추가 요구사항을 제공한다. 또한 각 구성 요소에 대해 내결함성을 분석하고 개선 방안을 고려한다.

## 4. 데이터 모델 설계

- 서비스 간 **공유 DB** 구조는 일반적으로 안티패턴에 속한다
  [발생 가능한 문제상황]
  1. 같은 테이블에 대해, 다양한 서비스의 쿼리 리소스 간 경쟁
  2. 스키마 마이그레이션에 대한 복잡도 상승 (모든 서비스의 소스코드를 최신화해야 하는 리소스)
  3. 특정 DB 기술(e.g. MySQL, HDFS, Cassandra, Kafka)에 종속적이게 되어 유연성 저하
- https://vladmihalcea.com/jpa-entity-version-property-hibernate/
  - 버전 추적 관리를 통해 (a) 다수 서비스의 동일 테이블 접근 문제를 SQL 잠금 메커니즘으로 해결할 수 있다
  - 이때, 스키마 정규화/비정규화 수준에 대해 원하는 정도를 질문으로 알아내고 lock에 관한 타임스탬프와 최종 수정 정보에 대한 컬럼을 추가하는 작업이 필요하다

## 5. 로깅, 모니터링, 경보

- 모니터링은 매우 중요하다
  - 현재와 미래의 가능한 기능적/비기능적 요구사항의 시스템 능력의 버그, 성능 저하, 예상치 못한 이벤트, 다른 약점들을 식별해야 한다
  - 이는 고객 경험에 대한 가시성 제공과 직결된다
- 실패에 대해 얼마나 긴급한지, 빠른 대응이 필요한지 분류해야 한다
  - 긴급도가 높을수록 즉시 처리해야 하는 대상
  - 타 서비스에 의존도가 높을수록 쉽게 성능 저하를 조사하고 질문에 답할 수 있는 로깅/모니터링 설정을 갖추는 것이 좋다
- **관찰 가능성** : 얼마나 측정이 잘 되어 있고 내부에서 무슨 일이 일어나고 있는지 얼마나 쉽게 알 수 있는가
  - 로깅, 메트릭, 추적을 통해 달성할 수 있는 지표
- 모니터링의 중요 신호 4가지
  1. 지연 시간(Latency) - SLA 초과 시 알럿
  2. 트래픽(Traffic) - 초당 HTTP 요청 수로 측정. 부하 테스트의 부하 한계를 기반으로 적절한 수치 설정
  3. 오류(Errors) - 즉시 해결해야 하는 4xx/5xx 응답 코드의 높은 긴급성 알럿
  4. 포화(Saturation) - 시스템의 제약이 CPU, 메모리, I/O인지에 따라 초과해서는 안 되는 사용률 목표 설정, 스토리지 사용률 등 → 초과 시 알럿
     fyi. https://sre.google/sre-book/monitoring-distributed-systems/#xref_monitoring_golden-signals
- 모니터링과 경보 툴 3가지
  1. Metric
  2. Dashboard
  3. Alert
     → 이들은 로그 데이터를 통해 채워진다
- 로깅의 일반적인 고려사항
  1. 로그 항목은 툴과 자동화로 쉽게 파싱될 수 있도록 구조화되어야 한다
  2. 각 항목은 서비스 간 요청을 추적하고, 사용자-개발자 간 공유 가능한 고유 식별자를 포함해야 한다
  3. 로그 항목은 작고, 읽기 쉬우며, 유용해야 한다
  4. 타임스탬프의 시간대, 포맷은 통일한다
  5. 로그 항목의 레벨을 분류한다 (DEBUG, ERROR, INFO, WARN, ..)
  6. 민감 정보는 로깅 대상에서 제외한다 — PII
- 경보 목록, 발생 가능한 원인, 원인 분석 및 해결 절차가 포함된 Runbook을 준비해야 한다
  - 문제 해결 방안이 쉽게 복사/붙여넣기 가능한 명령이라면, 로깅과 함께 애플리케이션에서 자동화되어야 한다
    - 이런 케이스에서 자동화된 실패 복구를 구현하지 않는다면 오히려 런북을 남용하는 것이다
  - 런북에 특정 메트릭을 보는 명령이 포함된다면, 바로 대시보드에서 확인 가능하도록 구성해야 한다
- 장애 발생 상황에서는 사후 분석(Postmoterm)을 작성해 문제에 대한 근본 원인, 해결 방안, 재발 방지를 식별해야 한다
  - 조치 패턴을 식별하면, 문제의 해소를 자동화하는 방법을 파악해 시스템에 자가 치유(self-healing) 특성을 도입할 수 있다

### 애플리케이션 레벨의 로깅 툴

    - Nagios

  - +) Splunk, Datadog, NewRelic, Nagios

  - 다운샘플링을 구성해 저장 공간 절약
  - 정의된 간격 평균을 계산해 오래된 데이터 롤업 (평균값만 저장. 원본 데이터 삭제)
  - e.g. Graphite (OS 메트릭 로깅에 주로 사용됨), Prometheus, OpenTSDB (HBase 기반), InfluxDB

### 데이터 품질

> 데이터가 지칭하는 실제 구성으로, **의도한 목적으로 사용될 수 있는지 확인**하는 것

- 스트리미, 배치 ETL 작업을 통해 지속적, 주기적으로 데이터 품질 문제를 감사할 수 있다
- 이는 조용한 오류를 탐지하는 데 유용하다

### 데이터 이상 탐지

> 비정상적인 데이터 포인트를 탐지하기 위한 머신러닝 개념

- 데이터 품질을 보장하고 분석 인사이트를 도출하는 데 유용
- 특정 메트릭의 비정상적인 상승/하락은 데이터 처리 문제나 시장 조건의 변화를 나타낼 수 있다
- 기본 형태 - 연속적인 데이터 스트림을 이상 탐지 알고리즘에 입력하는 것으로 구성

## 6. 검색창

면접에서 사용자 스토리와 기능 요구사항을 논의할 때 검색을 언급하는 것은 고객 중심적인 접근 자세를 보여주는 것이다

주로 다루는 수준 - 인덱스 생성, 수집과 인덱싱, 검색 쿼리 수행, 결과 처리

### 검색 구현 기술

1. SQL LIKE 연산자, 패턴 매칭을 사용한 쿼리

   → 한계가 많기 때문에 임시 방편으로만 사용해야 함
   - 검색 쿼리를 사용자 정의하기 어려움
   - 부스팅, 가중치, 퍼지 검색, 어간 추출, 토큰화 같은 텍스트 전처리의 정교화 기능이 없음

2. https://github.com/kentcdodds/match-sorter - 레코드 매칭과 정렬을 수행하는 JS 라이브러리
3. Elasticsearch와 같은 검색 엔진 사용 (PB 단위의 데이터까지 처리 가능)

### Elasticsearch

- 인덱스 생성은 사용자가 검색창에서 검색 쿼리를 제출할 때 검색할 문서를 수집 → 인덱싱 수행하는 것으로 구성된다
- 인덱스 매핑 변경하기
  1. 주기적/이벤트 트리거 방식의 인덱싱, Bulk API를 통한 삭제 요청으로 인덱스를 최신화할 수 있다
  2. Elasticsearch의 Reindexing 작업 사용

- SQL 개념과 대응시킨다면
  | **SQL** | **일래스틱서치** |
  | --------- | -------------------------- |
  | Database | Index |
  | Partition | Shard |
  | Table | Type (대체 없이 지원 중단) |
  | Column | Field |
  | Row | Document |
  | Schema | Mapping |
  | Index | 모든 것이 인덱싱됨 |

## 7. 기타 논의 가능한 주제

- **애플리케이션 유지보수와 확장**
  - 어떤 시스템 구성 요소가 기술(e.g. 소프트웨어 패키지)에 의존하며, 어떤 것이 가장 빠르게 개발되고 가장 많은 유지보수 작업이 필요한가?
  - 모든 구성 요소에서 호환성을 꺠는 변경을 도입하는 업그레이드를 어떻게 처리할 것인가?
  - 향후 개발해야 할 수 있는 기능과 시스템 설계
  - 향후 필요하지 않을 수 있는 기능은 무엇이고 이를 어떻게 점진적으로 폐기하고 해체할 것인가?
    - 이 과정에서 적절한 수준의 사용자 지원은 무엇이며 이를 어떻게 잘 제공할 수 있을까?
- **다른 유형의 사용자 지원**
  - 기존 서비스 확장 VS 새로운 서비스 구축
- **대안적 아키텍처 결정**
- **사용성과 피드백**
  - 사용성 지표를 정의 → 사용자 인터페이스의 사용용이성 평가
  - 이를 위해 필요한 데이터 로깅 및 계산, 대시보드에 표시하는 부분까지 고려할 수 있다
  - 순추천지수(Net Promoter Score, NPS) : 일반적인 사용성 설문조사 지표
- **엣지 케이스와 새로운 제약 조건**
  - 트레이드오프할 수 있는가, 아키텍처 재설계할 수 있는가
