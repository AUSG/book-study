# 11장. 자동 완성/타입어헤드

> 목적 - 지속적으로 데이터를 수집하고 처리해 특정 목적으로 사용자가 쿼리할 수 있는 작은(수 MB) 데이터 구조로 만드는 분산 시스템을 설계하는 능력 테스트

- 전 세계 수십업 명의 사용자가 입력한 문자열에서 데이터를 수집
- 이 데이터를 **가중 트라이(weighted trie)**로 처리하고, 사용자가 문자열 입력 시 가중 트라이가 자동 완성 제안을 제공하는 구조
  - 가중치가 부여된 트라이 구조
  - 각 노드나 엣지에 가중치 정보가 추가되어 데이터의 중요도나 빈도를 표현하는 자료구조
- 개인화, 머신러닝 요소가 추가될 수 있음

## 자동 완성의 가능한 사용 사례

- 검색 서비스를 보완 - 사용자가 검색 쿼리의 키 입력마다 자동 완성 제안 목록을 반환
  - e.g. Google, Bing, Baidu, Yandex, Wikipedia
- 워드 프로세서 - 퍼지 매칭 기술을 사용해 사용자가 입력한 접두사와 근접(정확히 일치X)한 단어를 제안하는 맞춤법 검사 기능을 제공할 수 있음
- IDE - 프로젝트 디렉터리 내 변수명, 상수값을 기록해 선언 시마다 이를 제안 목록으로 제공 (정확한 일치로 동작)

## 검색(Search) vs 자동완성(Autocomplete)

> 검색 = "사용자가 원하는 문서를 찾는 문제"
>
> 자동완성 = "사용자가 다음에 입력할 문자열을 예측하는 문제"

- 유사점
  - 사용자의 검색 문자열을 기반으로 사용자의 의도를 파악하고, 의도와 가장 일치할 가능성이 높은 순서로 정렬된 결과 목록 반환
  - 부적절한 콘텐츠 노출을 방지하기 위해 결과를 전처리해야 함
  - 사용자 입력(e.g. 클릭 결과)을 기록하고 이를 통해 제안/결과를 개선할 수 있음
- 차이점
  | **항목**  | **검색**                | **자동완성**             |
  | --------- | ----------------------- | ------------------------ |
  | 목적      | 관련 문서 찾기          | 입력 중 다음 문자열 추천 |
  | 응답속도  | 수백 ms ~ 수초 허용     | 약 100ms 이하 필요       |
  | 결과 형태 | 문서, URL, 객체 등 다양 | 문자열 후보만            |
  | 랭킹      | 관련성 점수 매우 중요   | 단순 정렬도 가능         |
  | 정확도    | 매우 중요               | 상대적으로 덜 중요       |
  | 처리 범위 | 전체 데이터셋 대상      | 일부 샘플·근사 탐색 가능 |
  | 결과 수   | 수십~수백 개            | 보통 5~10개              |
  | 피드백    | 클릭·체류시간 등 풍부   | 추천 선택 여부 정도      |
  - 시스템 설계 관점에서는?
    ```
    검색 엔진
    → 정확도 우선
    → 대규모 인덱스
    → 복잡한 랭킹
    → 수백 ms 허용

    자동완성
    → 지연시간 우선
    → 메모리 기반 조회
    → Prefix Trie / FST 등 사용
    → 100ms 이하 필수
    ```

## 기능 요구사항

- 자동 완성 서비스의 범위
  - 지원해야 할 사용 사례 - 일반 검색 서비스 / 워드 프로세서 / IDE / ..
  - 지원 언어 - 영어만 지원
  - 지원해야 할 단어 수 - 최대 10만 개 (사전 + 사전에 없는 인기 단어) → 5MB의 저장공간
- UX 세부 사항
  - 자동 완성 대산 - 문장 단위 / 개별 단어 단위
  - 제안을 표시하기 위해 입력할 최소 문자 수 - 3글자
  - 제안의 최소 길이 - 6글자 이상의 단어
  - 숫자, 특수문자 고려 여부 - 문자 only
  - 한 번에 보여질 자동 완선 제안 목록 수 - 10개. 빈도 수로 정렬
    - API를 통해 우선순위 정렬 & 개인화 등 적용 가능
- 검색 기록 고려
  - 사용자의 현재 입력 기반 / 검색 기록, 다른 데이터 소스 기반
  - 미리 인덱싱된 자동완성 전용 인덱스에 대해, 새로운 단어/구문이 추가될 때마다 리인덱싱할 것인가
    - e.g. 문서 1억개 → 자동완성 색인 생성 → "karina" 추가 → 문서 1억개 다시 스캔 → 색인 재생성
  - 모든 사용자 데이터 기반 / 개인화된 현재 사용자 데이터 기반
  - 제안 시 사용할 기간 - 1년 이상 된 데이터는 제거
- 콘텐츠 조정과 공정성
  - 부적절한 제안 신고 메커니즘
  - 검색횟수 집계 방식 - 한 사용자가 너무 많이 검색한 경우에도 1회로 카운팅 (`score(query) = unique_user_count(query)`) / 검색 문자열 자체의 실제 검색 횟수만을 고려 (`score(query) = count(query)` \*Query Frequency 기반 모델)

## 비기능적 요구사항

- 글로벌 사용자 기반이 사용할 수 있도록 확장 가능해야 함
- 높은 가용성/내결함성 X
- 높은 성능, 처리량 O — 0.5s 내에 자동 완성 제안을 반환
- 일관성 X — 새로운 사용자의 입력을 즉시 추천 리스트로 업데이트하지 않아도 됨
- 프라이버시, 보안을 위해 인증/승인까지는 아니더라도 사용자 데이터는 비공개로 유지
- 정확도
  - 검색 문자열의 빈도 계산 (근사치 → 정확도 지표 정의 포함하는 방식으로 점차 보장)
  - 맞춤법 오류, 혼합 언어 쿼리 고려X
  - 잠재적으로 부적절한 단어와 구문은 사전 단어 집합으로 수동 관리가 가능해야 함
  - 1일까지 제안의 최신성을 느슨하게 적용

## 전체 아키텍처

> 가중 트라이 접근법과 초기 고수준 아키텍처





1. 데이터 수집
   - 사용자는 자신의 검색 쿼리(문자열)를 수집 시스템에 제출 → DB에 저장
   - 트래픽 급증을 처리해야 함 \*_수집-처리 시스템을 분리한 CQRS 설계_
   - 수집 시스템은 검색 서비스의 로깅 서비스로서 동작할 수도 있음
     - 검색 서비스에서 사용자가 쿼리를 전송하면, 해당 쿼리는 로깅 서비스에 기록되므로
     - 자동 완성 제안에 활용가능하다면 다른 서비스의 로그도 쿼리할 수 있음 ⇒ **공유 로깅 서비스로서 동작**
2. 데이터 처리
   - 리소스 집약적
   - 수집한 사용자의 검색 문자열에 대해 주기적으로 가중치 트라이를 생성해 자동 완성 서비스에 전달
     !“all”, “apes”, “bat”, “bay” 단어의 가중치 트라이 구조
     “all”, “apes”, “bat”, “bay” 단어의 가중치 트라이 구조
   - 결과 목록은 입력 문자열과 대조된 문자열의 자식 노드로 생성되며, 가중치가 감소하는 순서로 정렬됨
3. 처리된 데이터를 쿼리해 자동 완성 추천을 얻는다
   - 사용자는 검색 문자열 입력 시, DB에서 자동 완성 제안을 받음

## 가중치 트라이 구조의 상세 구현

> 가중치 트라이 생성기 = 일일 배치 ETL 파이프라인 / 스트라밍 파이프라인

- 별도의 독립적인 ETL 파이프라인으로 구현 시, DAG 구성
  > **각 단계는 독립적인 작업이어야 한다**
  
  1. 로깅 서비스의 검색 토픽에서 관련 로그를 조회 → 임시 저장소에 저장
  2. 검색 문자열을 단어로 분리
  3. 부적절한 단어 필터링
     1. 적절한 단어, 부적절한 단어 목록 관리하기
     2. 검색어 목록을 적절/부적절한 단어 목록과 대조해 필터링하기
        1. 부적절한 단어 요청 → 이 목록을 HDFS 파일에 기록
        2. 부적절한 단어가 포함된 HDFS 파일이 있으면 → `LOAD DATA`로 하이브 테이블을 등록
        3. 쿼리로 필터링 & 출력을 다른 HDFS 파일에 기록
        4. Spark 등의 분산 분석 엔진으로 어떤 검색 문자열이 부적절한 단어인지를 판단할 수도 있음

        \*map join, broadcast hash join, ..

     3. 단어를 세기 전 사용자의 검색어에서 철자 오류 수정하기
        1. Fuzzy matching 알고리즘 - 근사 문자열 매칭. 패턴과 대략적으로 일치하는 문자열을 찾는 기술
        2. 균등하게 크기가 나눠진 하위 목록으로 나눈 단어 목록에 이 함수를 병렬로 실행
  4. 단어 수를 센 값을 단어 수 테이블에 기록 \*count-min sketch 등 알고리즘 활용
     1. 입력 HDFS 파일의 단어를 (String, Int) 쌓인 counts로 매핑
     2. counts의 내림차순으로 정렬
     3. 다른 HDFS 파일로 저장
  5. 적절한 단어 필터링 & 잘 알려지지 않은 인기 단어(신조어) 기록
     - 단어 수 테이블을 통해 필터링할 단어 수를 크게 줄여야 함
       - \*hive 명령, map join, broadcast hash join, ..
     - 신조어를 운영 팀에서 수동으로 추가할 수 있도록 UI 기능과 엔드포인트로 제공
  6. 단어 수 테이블에서 가중치 트라이 생성
     - 이제까지 구성된 적절한 상위 단어 목록은 몇 MB에 불과하므로 단일 호스트에서 처리 가능
     - JSON으로 직렬화
     - 객체 저장소(AWS S3), 문서 데이터베이스(Amazon DocumentDB, MongoDB)에 기록
       - \*jitter를 사용해 동시에 쿼리하게끔 구성 가능 (무작위 시간 쿼리나 동시 부하 방지)
       - 데이터 규모가 클 경우 CDN도 고려해볼 수 있음
     - 이 트라이의 조회는 클라이언트 사이드에서 전체를 다운로드 받아두고 로드되므로, 백엔드 요청 횟수를 줄일 수 있음
  7. 가중치 트라이를 백엔드 호스트로 전송
- 원시 검색 로그
  - 일별로 분할된 ElasticSearch 인덱스
  - HDFS = 맵리듀스 프로그래밍 모델의 일반적인 저장 시스템
  - ⇒ 여러 노드에 걸쳐 데이터 처리를 병렬화

## 샘플링 접근 방식

> 자동 완성에 높은 정확도가 필요하지 않은 경우

대부분의 단계에서 수행 가능

- 로깅 서비스에서 검색 문자열 샘플링하기
- 검색 문자열을 개별 단어로 분할하고, 6자 이상 길이의 단어를 필터링하여 단어 샘플링하기
- 적절한 단어를 필터링한 후, 단어 샘플링하기

## 저장소 요구사항 처리하기

- 근사화, 샘플링 기법으로 정확도를 트레이드오프하여 저장 비용을 줄일 수 있다
- 다양한 기간으로 데이터를 집계해 롤업하는 배치 ETL 작업
  
  - 가장 빈번한 문자열의 상위 10~20%만 유지
  - 가중치 트라이 생성기가 사용하는 최종 단어 수 계산에 필요한 처리 데이터양도 줄일 수 있음
  - 확장성 개선까지 가능한 방식
- 일일 가중치 트라이 생성기 배치 ETL 작업
  \***_주별/월별로 롤업된 데이터를 수행하는 아키텍처_**
  

## 단어 대신 구문 처리하기

- 최대 길이를 결정해야 한다. 트라이 자체가 커지고, 더 많은 하드웨어 리소스를 사용하게 되며, 로깅 및 처리에도 구문이 더 부담이 크기 때문이다.
  - 사용자 검색 문자열 길이의 90번째 백분위수를 찾아 이를 최대 길이로 사용하는 ETL을 구현하는 방법이 있다
  - 중앙값, 백분위수 계산 로직 수반
- 부적절한 단어 뿐만 아니라 부적절한 구문도 필터링 해야 한다.
  - 부적절한 단어가 구문에 하나라도 포함되어 있으면 전체 구문을 필터링한다.
  - 맞춤법 틀린 단어 수정 X
  - 휴리스틱, 머신러닝의 조합을 사용해 부적절한 구문 범위의 방대함을 대응하는 방법도 있다
