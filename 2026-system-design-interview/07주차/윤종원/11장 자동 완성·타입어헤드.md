# 11. 자동 완성/타입어헤드

## 11.1 자동 완성의 가능한 사용 사례
- 사용 사례
	- 검색 서비스 보완
	- 워드 프로세서가 자동 완성 제안 제공
	- IDE에서의 자동 완성
- 각 사용 사례에서 자동 완성 서비스는 서로 다른 데이터 소스와 아키텍처 사용
- 면접에서 실수할 수 있는 건 성급한 결론을 내리고 검색 서비스를 위한 것이라 가정하는 것
- 면접관이 검색 앱용 자동 완성 제공 시스템을 설계하라고 얘기해도 잠깐 동안 자동 완성 기능의 다른 용도에 대해 논의할 수도 있다.
## 11.2 검색 vs. 자동 완성
- 두 서비스는 유사해보이지만 다르다
	- 두 서비스 모두 검색 문자열을 기반으로 의도를 파악해 가장 일치할 가능성이 높은 순서로 정렬된 결과 목록을 반환
	- 부적절한 콘텐츠가 반환되는 것을 막기 위해 전처리해야할 수도 있다
	- 사용자 입력을 기록하고 이를 사용해 개선할 수도 있다


- 간단하게 1분 이내로 차이를 설명할 수도 있다. 너무 길지 않게

|**검색**|**자동 완성**|
|---|---|
|결과는 일반적으로 웹페이지 URL이나 문서 목록이다. 이러한 문서는 전처리 돼 색인을 생성한다. 검색 쿼리 중에 검색 문자열을 색인과 대조해 관련 문서를 검색한다.|결과는 사용자 검색 문자열을 기반으로 생성된 문자열 목록이다.|
|몇 초의 P99 지연 시간이 허용될 수 있다. 특정 상황에서는 최대 1분의 더 긴 지연 시간도 허용될 수 있다.|우수한 사용자 경험을 위해 P99 기준 약 100ms의 낮은 지연 시간이 필요하다. 사용자는 각 문자를 입력한 후 거의 즉시 제안받기를 기대한다.|
|문자열, 복잡한 객체, 파일이나 미디어를 포함한 다양한 결과 데이터 유형이 가능하다.|결과 데이터 유형은 문자열뿐이다.|
|각 결과에 관련성 점수가 부여된다.|항상 관련성 점수가 있는 것은 아니다. 예를 들어, IDE의 자동 완성 결과 목록은 사전 순으로 정렬될 수 있다.|
|사용자가 인식하는 정확도인 관련성 점수를 가능한 한 정확하게 계산하기 위해 많은 노력을 기울인다.|정확도 요구사항(예: 사용자가 나중 제안이 아닌 처음 몇 개의 제안 중 하나를 클릭함)이 검색만큼 엄격하지 않을 수 있다. 이는 비즈니스 요구사항에 따라 크게 달라지며, 특정 사용 사례에서는 높은 정확도가 요구될 수 있다.|
|검색 결과는 입력 문서 중 어느 것이든 반환할 수 있다. 이는 모든 문서를 처리하고, 색인화하고, 검색 결과로 반환할 수 있어야 함을 의미한다. 복잡성을 낮추기 위해 문서 내용을 샘플링할 수 있지만 모든 단일 문서를 처리해야 한다.|높은 정확도가 필요하지 않으면 복잡성을 낮추기 위해 샘플링과 근사 알고리즘과 같은 기술을 사용할 수 있다.|
|수백 개의 결과를 반환할 수 있다.|일반적으로 5~10개의 결과를 반환한다.|
|사용자는 ‘뒤로’ 버튼을 클릭한 다음 다른 결과를 클릭해 여러 결과를 클릭할 수 있다. 이는 다양한 추론을 도출할 수 있는 피드백 메커니즘이다.|다른 피드백 메커니즘이 있다. 자동 완성 제안 중 일치하는 것이 없으면 사용자는 검색 문자열 입력을 마치고 제출한다.|

## 11.3 기능 요구사항

### 11.3.1 자동 완성 서비스의 범위
- 어느 서비스 용도인지?
- 언어 지원은?
- 지원해야하는 단어 수는?
	- 웹스터 영어 사전에는 47만개. 옥스퍼드 영어 사전에는 17만개 이상.
	- 글자수 6개 이상이 얼마나 되는진 모르므로 가정은 하지 않음
	- 사전에 없는 단어도 지원하고 싶으므로 최대 10만개 단어 지원한다고 가정
	- 영어 단어 평균 길이가 4.7이고 1자당 1바이트라 하면 필요한 저장 공간은 5MB에 불과하다.
	- 수동 방식으로 단어 및 구문 추가 허용해도 저장 공간 요구사항은 무시할 만한 수준으로 증가

### 11.3.2 UX 세부 사항
- 문장 단위인지 개별 단어 단위인지, 몇 자를 입력해야 제안이 표시되는지 등이 필요
- 여기서는
	- 단어 대상. 시간이 있다면 구문이나 문장 확장 고려
	- 최소 문자 수 3글자
	- 제안의 최소 길이 6글자 이상. 3글자를 입력한 후 4~5글자 제안은 1~2글자만 더 입력하면 되므로 유효하지 않다
	- 숫자나 특수문자 고려? -> 문자만 고려
	- 한 번에 몇개의 자동 완성 보여줘야 하고, 순서는? -> 한번에 10개 제안 표시. 빈도 순으로 정렬. 문자열을 받아 우선순위가 높은 순서로 정렬된 10개 단어 반환하는 API 만든 후 나중에 사용자 ID도 받아 개인화된 제안 반환하게 확장

### 11.3.3 검색 기록 고려
- 사용자의 현재 입력 기반? 아니면 검색 기록과 다른 데이터 소스를 기반?
- 제안을 단어 집합으로 제한한다는 건
	- 단어 집합으로 제한하는 건 사용자가 제출한 검색 문자열을 처리해야 함을 의미
	- 수동으로 추가 및 제거된 단어, 구문을 포함하기 위해 이전에 처리된 데이터를 다시 처리해야 하나?
	- 과거 데이터가 상당히 많을 수 있음을 면접관과 논의
	- 수동으로 추가되는 이유는 뭔가? 과거 사용자 검색 문자열에 대한 분석을 기반으로 한다면 ETL이나 파이프라인 등을 논의할 수 있음
- 제안에 대한 데이터 소스는? 사용자 인구 통계 등의 다른 데이터 소스는?
	- 다른 데이터 소스도 고려하는 건 좋은 생각.
	- 제출된 쿼리만 사용하고 향후에 확장
- 모든 사용자 데이터를 기반으로 제안 표시? 아니면 개인화?
	- 모든 사용자 데이터로 시작하고 이후에 개인화 고려
- 제안에 어느 기간을 사용?
	- 모든 시간 고려 후 1년 이상 된 데이터를 제거하는 걸 고려할 수 있음

### 11.3.4 콘텐츠 조정과 공정성
- 부적절한 제안 신고할 수 있는 메커니즘?
- 소수의 사용자가 대부분의 검색을 제출할 경우 이를 고려해야 하는가? 자동 완성 서비스가 사용자당 동일한 수의 검색을 처리해 대다수 사용자에게 서비스를 제공해야 하는가?
	- 검색 문자열 자체만 고려하자. 어떤 사용자가 검색했는지는 고려하지 않는다.

## 11.4 비기능적 요구사항
- 가용성 대 성능 등 트레이드 오프를 논의할 수 있다
	- 글로벌 사용자 기반이 사용할 수 있게 확장 가능해야 한다.
	- 높은 가용성은 필요하지 않다. 중요한 기능이 아니므로 내결함성은 양보할 수 있다.
	- 높은 성능과 처리량이 필요하다. 사용자는 0.5초 이내에 볼 수 있어야 한다.
	- 일관성은 필요하지 않다. 추천이 과거 내용이어도 무관하며 즉시 추천을 업데이트하지 않아도 된다
	- 프라이버시와 보안을 위해 자동 완성 기능을 사용하는 데 인증이나 승인이 필요하진 않고, 사용자 데이터는 비공개로 유지해야 한다.
	- 정확도에 대해서는 아래와 같이 추론할 수 있다
		- 검색 빈도에 따라 제안을 반환하고자 할 수 있으므로 검색 문자열의 빈도를 계산할 수 있다. 첫 설계 단계에서는 이런 계산이 정확할 필요가 없으므로 근사치로 충분하다고 결정할 수 있다. 시간이 있다면 정확도 지표 정의를 포함해 더 나은 정확도를 고려할 수 있다.
		- 맞춤법 오류나 혼합 언어 쿼리는 고려 X
		- 잠재적으로 부적절한 단어나 구문에 관해서는 제안을 단어 집합으로 제한해 부적절한 단어를 방지할 수는 있지만 구문은 방지할 수 없다.
		- 제안이 얼마나 최신이어야 하는지에 대해서는 1일의 느슨한 요건을 적용할 수 있다.

## 11.6 가중치 트라이(Trie) 접근법과 초기 고수준 아키텍처
![](files/Pasted%20image%2020260614165940.png)
- 사용자는 자동 완성 서비스와만 상호작용하고 시스템의 나머지 부분과는 상호작용하지 않음
- 나머지 부분은 사용자의 검색 문자열을 수집하고 주기적으로 가중치 트라이를 생성해 자동 완성 서비스에 전달


- 공유 로깅 서비스는 자동 완성 서비스가 사용자에게 제공하는 자동 완성 추천을 도출하는 원시 데이터 소스
- 사용자가 검색 서비스를 이용해 쿼리를 전송하면, 해당 쿼리는 로깅 서비스에 기록된다.
	- 다른 서비스도 이 공유 서비스에 로그를 남김
	- 자동 완성 제안을 개선하는 데 이런 로그가 도움이 된다면 이 또한 사용해도 됨


- 공유 로깅 서비스는 주제와 타임스탬프를 기반으로 로그 메시지를 가져오는 API를 갖춰야 함

![](files/Pasted%20image%2020260614165946.png)

- 자동 완성 추천은 가중치 트라이로 생성됨
- 가중치 내림차순으로 top k 찾을 수 있음
- 시간 복잡도 `O(P + M log K)`
	- P : 접두사 길이
	- M : 그 접두사로 시작하는 단어 후보 수
	- K : 보여줄 상위 개수
- 즉, 입력 길이 + 후보 수에만 의존
	- 전체 단어 수에 의존하지 않음
	- + 새로운 노드 삽입 비용이 O(L). 삽입하는 단어 길이에만 의존
		- 이분 탐색 등을 위한 정렬된 배열 방식은 삽입 비용이 크다


## 11.7 상세 구현
- 일 배치 ETL나 실시간 스트리밍 등으로 처리할 수 있다
- 이 파이프라인은 단어 처리 ETL 포함


- 단어 처리 ETL과 트라이 생성기가 별도로 되어있는 이유는 단어 처리 ETL이 다른 여러 목적 서비스에 유용할 수 있고, 별도의 단계로 구성하면 각각을 독립적으로 구현, 테스트, 유지보수 및 확장할 수 있기 때문

![](files/Pasted%20image%2020260614165953.png)

1. 로깅 서비스에서 관련 로그를 가져와 임시 저장소에 저장
2. 검색 문자열을 단어로 분리
3. 부적절한 단어 제거
4. 단어 수를 세어 단어 수 테이블에 기록. 필요한 정확도에 따라 모든 단어를 세거나, Count-Min Sketch 같은 근사 알고리즘 사용
5. 적절한 단어를 필터링하고 잘 알려지지 않은 인기 단어를 기록
6. 단어 수 테이블에서 가중치 트라이를 생성
7. 가중치 트라이를 백엔드 호스트로 전송


- 검색 로그에 대해서는 다음 DB 기술 고려 가능
	- ELK 스택. 일별로 분할된 ES 인덱스. 보존 기간 있음
	- 각 일자의 로그는 HDFS 파일(일별 분할).
		- 이건 이해를 못했습니다.. 훔

### 11.7.1 각 단계는 독립적인 작업이어야 한다
- 왜 각 단계를 분리할까?
- 처음에는 트라이 생성을 단일 작업으로 구현하고 모든 함수를 연결할 수 있다. 이는 간단하지만 유지보수가 어렵다
	- 보통 복잡성과 유지보수성은 반대로 움직이지만 여기는 트레이드오프가 있다


- 가중치 트라이 생성에서 오류가 발생하면 처음부터 다시 시작해야 한다
- ETL 작업은 계산 집약적이고, 완료하기까지 몇 시간이나 걸릴 수 있으므로 이런 접근은 성능이 낮다
- 이런 단계를 별도 작업으로 구현하고, Airflow 같은 스케줄러를 쓰는 게 더 좋다.

### 11.7.5 퍼지 매칭과 철자 교정
- 단어 세기 전 마지막 처리 단계는 철자 오류 수정

- 맞춤법 수정 단계는 여러 알고리즘이나 라이브러리, 서비스 등 아무거나 선택하면 된다
- 쉽게 전환할 수도 있긴 하다
- 라이브러리를 쓰면 변화하는 트렌드나 인기 있는 신조어 잡기 위해서 라이브러리 업데이트가 필요할 수 있다.

### 11.7.9 가중치 트라이 생성 및 전달하기
- 가중치 트라이는 몇 MB 수준이므로, 단일 호스트에서 생성할 수 있다.
- 이 알고리즘은 설계 문제에서는 벗어나며 코딩 테스트 문제로는 출제될 수 있다


- 가중치 트라이는 JSON으로 직렬화 가능하다
- 트라이 크기는 몇 MB로, 모든 클라가 다운 받기에는 너무 크지만 모든 호스트에 복제하기에는 충분하다.
	- 클라에 넣어서 클라 사이드 필터링을 구현해도 된다
	- 이렇게 하면 서버 부담이 줄어든다
		- 네트워크 불안정할 때도 사용자 경험을 유지하고
		- 트라이가 업데이트될 때 사용자가 변화를 느낄 수 있다. 바로바로 업데이트
- 백엔드 호스트는 매일 객체 저장소에 쿼리해 업데이트된 JSON 문자열을 가져오게 구성할 수 있다.


- 공유 객체가 GB 단위로 크다면 CDN 배치를 고려

## 11.8 샘플링 접근 방식
- 자동 완성에 높은 정확도가 필요 없다면 샘플링을 고려
- 그렇게 하면 가중치 트라이 생성의 대부분을 단일 호스트에서 수행할 수 있어 아래 이점이 있다
	- 트라이 생성이 빨라짐
	- 배포 및 테스트도 쉬워짐
	- 리소스 소모도 줄어든다


- 샘플링은 대부분의 단계에서 수행 가능
	- 로깅 서비스에서 검색 문자열 샘플링. 정확도가 가장 낮지만 복잡성도 가장 낮음. 통계적으로 유의미한 단어 수를 얻으려면 샘플이 커야할 수 있음
	- 단어 분할 후 필터링 후 샘플링. 적절한 단어를 필터링하는 계산 비용을 피할 수 있음.
	- 적절한 단어 필터링 후 샘플링. 정확도가 가장 높지만 구현 복잡성도 크다.

## 11.9 저장소 요구사항 처리하기
- 위 아키텍처 기반으로 아래 테이블을 만들 수 있음
	- 타임스탬프, 사용자 ID, 검색 문자열이 있는 원시 검색 요청
	- 원시 검색 문자열을 분할 후 개별 단어를 날짜와 단어를 테이블에 추가
	- 어떤 검색 문자열이 사전 단어인지 판단하고 날짜, 사용자 ID, 사전 단어를 포함하는 테이블 생성
	- 사전 단어를 단어 수 테이블로 집계
	- 자동 완성 제공을 위한 가중치 트라이 생성


- 저장 공간은 얼마나 필요할까?
- 10억명 사용자 하루에 10번. 검색당 평균 20자
- -> 매일 약 1B * 10 * 20 = 200GB 검색 문자열 생김
- 한 달에 한 번 오래된 데이터 삭제 -> 최대 12개월 데이터 보유 -> 73TB 필요


- 저장 공간을 줄이는 다양한 방법
	- 샘플링, 근사화 등
![](files/Pasted%20image%2020260615211317.png)
- 혹은 저장양을 줄이기 위해 다양한 기간으로 데이터 집계 후 롤업하는 배치 ETL 작성
- 최대 하루치의 원시 데이터, 주별로 롤업된 4주치 데이터, 월별로 롤업된 11개월치 데이터 보유
- 각 롤업 작업에서 가장 빈번한 문자열 상위 10, 20%만 유지해 용량을 더 줄일 수도 있다.


- 이는 확장성도 개선한다
- 롤업 작업 없이는 ETL 작업이 73TB 데이터 처리해야 해 시간도 많이 들고 비용도 많이 든다
- 롤업 작업은 가중치 트라이 생성기가 사용해야 하는 최종 단어 수를 줄인다.


- 로깅 서비스 보존 기간을 14일, 30일 등으로 짧게 설정하면 저장소 요구사항이 2.8 ~ 6TB에 불과하다

## 11.10 단일 단어 대신 구문 처리하기
### 11.10.1 자동 완성 추천의 최대 길이
- 최대 길이가 길수록 사용자에게 유리하지만 비용 문제가 있다
- 이는 언어나 문화에 따라 달라질 수 있다.


- 한 가지 해결책은 배치 ETL에서 사용자 검색 문자열 길이의 p90 값을 사용하는 것
- 단, 이걸 계산하는 것도 부하가 크므로 샘플링 후 90번째 백분위 수를 계산할 수도 있다.
- 혹은 이 엔지니어링 비용도 줄이기 위해 간단한 휴리스틱 적용할 수도 있다
- 적당히 30자 하고 사용자 피드백이나 비용, 성능 문제를 살펴본다.

### 11.10.2 부적절한 단어 방지하기
- 구문에 부적절한 단어가 하나라도 있으면 전체 구문 필터링
- 더 이상 적절한 단어를 필터링하지 않고 모든 단어, 구문을 자동 완성으로 제공


- 이 문제의 어려움은 부적절한 구문을 필터링해야 한다는 것
- 이는 구글도 완전한 해결책을 찾지 못했다
- 예를 들면
	- 종교, 성별, 다른 집단에 대한 차별이나 부정적 고정관념
	- 허위 음모론, 허위 정보 등
	- 유명 인사 비방이나 판결이 나지 않은 피고인 비방 등


- 한 가지 해결책은 휴리스틱 + 머신러닝 조합 사용

## 11.12 기타 논의 가능한 주제
- 세 글자보다 긴 흔한 단어도 많다. then, make, know, take 등
	- 이런 단어 중 일부는 지속적으로 가장 인기 있는 단어 목록에 항상 있다.
	- 이를 항상 세는 건 리소스 낭비일 수 있다
	- 이런 단어 목록을 유지하고 어떤 단어 반환할지 근사화 기법을 적용할 수 있을까?
- 사용자 로그는 추천 시스템 등 다양한 곳에 제공할 수 있다
- 분산 로깅 서비스 설계
- 부적절한 검색어 필터링, 콘텐츠 필터링
- 개인화된 자동 완성 기능 만들기 위해 다른 데이터 입력 및 처리 고려
- 람다 아키텍처 고려. 빠른 파이프라인으로 빠르게 가중치 트라이를 업데이트 해 몇 초, 몇 분 내에 빠르게 자동완성을 최신화할 수 있다.
- 업스트림 컴포넌트 다운 시 오래된 추천 반환하는 단계적 기능 저하 방식
- DDoS 공격 방지
- 맞춤법 틀린 단어 입력하면 단어 제안 받는 철자 추천 서비스.
