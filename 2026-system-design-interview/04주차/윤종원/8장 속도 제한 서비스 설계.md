# 08. 속도 제한 서비스 설계
- rate limit이 막지 못하는 봇 공격도 있다.
- 속도 제한은 라이브러리로 구현하거나, 프론트, API Gateway, 서비스 메시가 호출하는 별도 서비스로 구현할 수 있다.
- 기능적 분할의 장점을 얻기 위해 서비스로 구현한다.

## 8.1 속도 제한 서비스의 대안과 그것이 실현 불가능한 이유
- 왜 서비스를 확장하지 않고 제한하는가?
	- 트래픽 급증이 발생했을 때 새 호스트 추가가 너무 느릴 수 있다
	- LB는 각 호스트로 보내는 요청 수를 제한할 수 있다. 호스트에 과부하가 걸리지 않게 보장하면서 여유 용량이 없을 때 요청을 삭제하는 건 어떤가?
		- 악의적인 요청을 처리해서는 안 된다. 보통 IP 주소를 감지하고 버림으로써 이를 방어한다
		- 429를 반환할 수 있으나 특정 요청이 악의적이라는 것을 확신한다면 아래 옵션도 가능
			- 요청을 버리고 응답 반환하지 않아 공격자가 서비스에 장애가 발생했다고 생각하게 한다
			- 사용자를 섀도우 밴해서 200 반환하되, 빈 응답이나 오해의 소지가 있는 응답을 보낸다


- 왜 별도의 서비스? 각 호스트가 독립적으로 처리하면 안되나?
	- 특정 요청이 그 외 요청보다 비용이 더 많이 든다
	- 특정 사용자는 더 무거운 요청을 할 수도 있다
	- 호스트는 특정 클라이언트의 비용이 많이 드는 요청을 처리하느라 느려질 수 있다
- L4 LB는 요청의 내용을 처리할 수 없다. 사용자의 요청을 같은 호스트로 라우팅하려면 L7 LB가 필요한데 이는 비용과 복잡성을 늘린다
- 이미 L7 LB를 쓰던 게 아니라면 이를 위해서 쓰는 건 좀 가치가 없을 수 있고, 전용 공유 속도 제한 서비스가 나을 수 있다.

| 속도 제한 | 새 호스트 추가 | 7계층 로드 밸런서 사용 |
|---|---|---|
| 높은 요청 속도를 보이는 사용자에게 429 Too Many Requests 응답을 반환해 트래픽 급증을 처리 | 새 호스트 추가가 트래픽 급증에 대응하기에 너무 느릴 수 있음. 새 호스트가 트래픽을 처리할 준비가 됐을 때 서비스가 이미 중단됐을 수 있음 | 트래픽 급증을 처리하기 위한 해결책이 아님 |
| 오해의 소지가 있는 응답을 제공해 DoS 공격 처리 | 처리하지 말아야 할 악의적인 요청을 처리함 | 해결책이 아님 |
| 비용이 많이 드는 요청을 하는 사용자의 속도 제한 가능 | 비용이 많이 드는 요청을 처리하는 비용 발생 | 비용이 많이 드는 요청을 거부할 수 있지만 독립 솔루션으로는 비용과 복잡성이 너무 높을 수 있음 

## 8.2 속도 제한을 하지 말아야 할 때
- 속도 제한이 모든 종류의 클라이언트에게 적절한 해결책은 아닐 수도 있다
- SNS에서 너무 많은 구독 요청을 보내는 경우 429를 보내는 대신, 최근 몇 분 동안 너무 많은 구독 요청을 했다고 응답하는 것도 방법이다
- 또 다른 예는 특정 요청 속도의 구독 요금을 부과하는 서비스
	- 클라가 특정 시간 간격의 할당량을 초과하면 다음 시간 간격까지 추가 요청을 처리해서는 안된다
	- 공유 속도 제한 서비스는 클라가 구독을 초과하는 것을 방지하기 위한 적절한 해결책이 아니다
	- 각 클라에게 다른 속도 제한을 주는 것과 같은 복잡한 사용 사례가 아닌 단순한 사용 사례 지원에 국한되어야 한다.

## 8.3 기능적 요구사항
- 최대 요청 속도를 설정할 수 있어야 하고, 이를 초과하는 추가 요청은 지연되거나 429 응답이 나가야 한다
- 간격은 10초나 60초라고 가정할 수 있다.
	- 예를 들어 10초에 최대 10개의 요청 처리 가능


- 또 다른 기능적 요구사항
	- 각 사용자 서비스가 호스트 전체에서 요청자 속도를 제한해야 하지만, 서비스 간에 동일한 사용자의 속도를 제한할 필요는 없다. 속도 제한은 각 사용자 서비스에서 독립이다
	- 사용자는 엔드포인트당 하나씩 여러 개의 속도 제한을 설정할 수 있다. 특정 요청자, 사용자에게 다른 속도 제한을 허용하는 것과 같은 복잡한 구성은 필요하지 않다
	- 어떤 사용자가 속도 제한되었는지, 이런 속도 제한 이벤트가 언제 시작하고 끝났는지 볼 수 있어야 한다. 이를 위한 엔드포인트를 제공해야 한다.
	- 모든 요청을 로깅해야하는지 논의할 수 있다. 필요하다면 비용을 줄이기 위한 저장 공간 절약 기법을 설명한다
	- 수동 후속 조치와 분석을 위해서는 속도가 제한된 요청자를 로깅해야 한다.

## 8.4 비기능적 요구사항
- 확장 가능하고, 높은 성능, 가능한 단순하고 안전하고 프라이버시 보장
- 가용성에 필수적이지는 않으므로 고가용성과 내결함성을 트레이드오프할 수 있다
- 정확성과 일관성이 중요하지만 엄격할 필요는 없다

### 8.4.1 확장성
- 일일 수십억 건의 요청으로 확장할 수 있어야 한다
- 속도 제한을 변경하는 요청은 조직 내부에만 열면 된다.


- 10억 사용자, 사용자당 최대 100개의 요청을 저장한다고 가정
- 사용자 ID와 사용자당 100개의 타임스탬프를 기록 -> 각 64비트
- 최대 100개의 서비스가 속도 제한이 필요하다고 해보자
- 실제로 10억명 데이터가 다 필요한가? -> 최근 10초 동안 데이터만 필요.
- 10초 이내 100만에서 1,000만 명 이상의 사용자가 있을지 논의할 수 있음.
- 보수적으로 1,000만 명이라고 하면 전체 저장 공간은 100 * 64 * 101 * 10M = 808GB
- 레디스 쓰고 각 사용자에게 키 할당하면 값의 크기는 64 * 100 = 800바이트
- 10초가 지난 데이터를 즉시 삭제하는 것은 현실적이지 않을 수 있으므로 실제 필요한 저장 공간은 오래된 데이터를 얼마나 빨리 삭제할 수 있는지에 따라 다르다

### 8.4.2 성능
- rate limit 확인은 blocking이므로 매우 낮은 지연 시간이 필요하다
- P99가 100ms
- 로그 조회나 분석에는 낮은 지연 시간이 필요하지 않다

### 8.4.3 복잡성
- 최대한 간단하고, 단순하고, 비용 최소화

### 8.4.4 보안과 프라이버시
- 외부에서 속도 제한 서비스에 못 접근하도록 해야하고
- 내부 사용자 서비스도 이를 위조하려할 수 있는데 이를 방지해야 한다

### 8.4.5 가용성과 내결함성
- 조금 다운되어도 무방하다
- 가용성이 높아질수록 비용이 증가한다
- 속도 제한 여부를 캐싱할 수도 있는데, 문제가 생기면 일부 클라이언트가 계속해서 속도 제한을 받을 수도 있다
- 이런 일이 발생할 통계적 가능성은 낮다.
- 발생할 때 이 몇 분 동안 부정적인 사용자 경험을 감수하고, 방화벽 등의 다른 기술로 서비스 중단을 방지할 수도 있다

### 8.4.6 정확성
- 과도한 클라이언트를 잘못 식별해서 제한해서는 안된다
- 그러나 10초당 10개 요청이라면 가끔 8개나 12개 요청에서도 제한되어도 무방하다
- 최소 10개 요청을 보장해야 한다면 아예 기준치를 12개로 설정하는 등도 가능하다

### 8.4.7 일관성
- 강한 일관성이 필요없다
- 속도 제한 업데이트할 때 새 요청에 즉시 적용될 필요도 없고, 몇 초 간의 불일치 허용될 수 있다
- 최종 일관성 쓰면 더 단순하고 저렴한 설계가 가능하다

## 8.5 사용자 스토리와 필요한 서비스 구성 요소
- 사용자 ID, 서비스 ID가 필수
- ID 형식은 각 서비스에 특화될 수 있다. 이는 속도 제한 서비스가 아니라 사용자 서비스에 정의되고 유지된다
- 서비스마다 속도 제한이 다르므로 서비스 ID로 속도 제한 값을 결정한다.


- 속도 제한기는 사용자 ID, 서비스 ID 데이터를 60초 동안 저장해야 한다
- 지연 시간 최소화하려면 인메모리 저장소나 캐싱이 필요하다
- 로그에는 일관성이나 지연 시간이 필요하지 않으므로 HDFS 등의 최종 일관성 저장소에 로그를 저장할 수 있다
- 사용자 서비스는 속도 제한을 만들고, 업데이트하기 위해 속도 제한 서비스에 가끔 요청할 수도 있다.


- 종합하면
	- 카운트에 빠른 읽기, 쓰기 가능한 DB. 스키마는 단순. 레디스 같은 인메모리 DB를 쓸 수 있음
	- 규칙 정의하고 검색할 수 있는 서비스. 이를 규칙 서비스라고 한다
	- 규칙 서비스와 레디스 DB에 요청 보내는 서비스. 이를 백엔드 서비스라고 한다
- 두 서비스를 분리한 이유는 규칙 추가, 수정이 속도 제한 여부를 결정하는 서비스를 방해해서는 안되므로

## 8.6 고수준 아키텍처
![](files/Pasted%20image%2020260523210558.png)
- 클라가 속도 제한 서비스 요청 보내면 프론트엔드나 서비스 메시를 통과한다
- 프론트 보안 메커니즘이 요청을 허용하면 백엔드로 이동, 다음 단계가 발생
	- 규칙 서비스에서 속도 제한 가져온다. 이는 지연시간 줄이고 규칙 서비스 요청 볼륨 줄이기 위해서 캐싱될 수 있다
	- 이 요청을 포함한 서비스의 현재 요청 속도를 결정
	- 제한 여부를 나타내는 응답 반환
- 전체 지연시간 줄이기 위해 각 단계 스레드 분기하거나, 1~2단계를 병렬화할 수도 있다

![](files/Pasted%20image%2020260523212308.png)
- 자주 규칙이 변경되지는 않으니 레디스 캐시 추가해 읽기 성능 더 올릴 수도 있다
- 백엔드 서비스도 레디스에 규칙을 캐싱할 수 있다
- 과도한 사용자의 ID도 캐시할 수 있다. 속도 제한 초과하면 더 이상 적용되지 않는 만료 시간과 사용자 ID를 캐싱. 이렇게 하면 사용자 요청 거부를 위해 규칙 서비스 쿼리할 필요가 없다

![](files/Pasted%20image%2020260523212459.png)

- AWS를 쓴다면 레디스나 SQL 대신 AWS DynamoDB써도 된다.
- 초당 수백만 개의 요청 처리 가능하고, 강한 일관성을 지닌다


- 요구사항 고려하면 초기 아키텍처가 과도하게 설계되고, 복잡하며 비용이 많이 들 수도 있다
- 매우 정확하고 강한 일관성을 가지지만, 둘 다 비기능적 요구사항에 해당하지는 않는다
- 비용을 낮추기 위해 정확성과 일관성을 일부 포기할 수 있을까?
- 일단 확장하기 위한 두 가지 접근부터 보자
	- 호스트가 상태를 유지하지 않고 공유 DB에서 데이터 가져와 모든 사용자를 서비스할 수도 있다. 상태 비접근 방식
	- 호스트가 고정된 사용자 집합을 서비스하고, 해당 사용자의 데이터를 저장. 상태 저장 접근 방식

## 8.7 상태 저장 접근 방식/샤딩
![](files/Pasted%20image%2020260523212622.png)
- 비기능적 요구사항에 더 가까운 상태 저장 솔루션의 백엔드
- 요청이 도착하면 LB가 해당 호스트로 라우팅
- 각 호스트는 자신의 클라 수를 메모리에 저장. 
- 호스트는 사용자가 속도 제한을 초과했는지 판단하고 참이나 거짓 반환
- 호스트가 다운된 경우, 500 오류를 반환하고 요청은 속도 제한이 적용되지 않는다.


- 상태 접근 방식은 L7 LB 필요. 


- 내결함성 질문이 나와야 한다. 호스트가 다운되었을 때 데이터 손실을 방지해야 하는지?
- 그렇다면 복제, 장애 조치, 핫 샤드, 재조정 같은 주제 설명으로 이어진다
- 복제에 고정 세션을 사용할 수도 있다. 그러나 앞에서 일관성, 고가용성, 내결함성이 필요하지 않다고 했다. 
- 호스트가 다운되면 다른 호스트에서 요청 수 0부터 다시 시작할 수 있다.
- 이렇다면 호스트 중단 감지, 대체 호스트 할당, 프로비저닝, 트래픽 재조정 등을 이야기해야 한다.


- 500 오류는 새 호스트 프로비저닝하는 자동화된 응답을 트리거해야 한다
- 새 호스트는 구성 서비스에서 주소 목록을 가져와야 한다.
- 이는 S3 등의 수동 업데이트 파일이거나 주키퍼 등의 복잡한 솔루션일 수도 있다
- 파일이 고가용성을 위해 단일 호스트가 아니라 분산 저장소에 저장되어야 한다.
- 속도 제한 서비스 개발할 때 호스트 설정 프로세스가 몇 분을 초과하지 않게 해야한다. 이를 모니터링하고, 몇 분을 초과하면 낮은 긴급도의 알림을 트리거해야 한다.


- 핫 샤드를 모니터링하고, 주기적으로 호스트 간 트래픽 재조정해야 한다
- 요청 로그 읽고, 많은 요청을 받는 호스트 식별하고, 적절한 로드 밸런싱 구성 결정 후 이를 구성 서비스에 쓰는 배치 ETL 주기적으로 실행
- ETL 작업은 새 구성을 LB로 푸시할 수도 있지만 LB 호스트가 다운되었을 때를 대비해 구성 서비스에 쓴다


![](files/Pasted%20image%2020260523213917.png)
- 이 재조정은 많은 수의 부하가 특정 호스트에 할당되어 다운되는 것을 방지한다
- 실패한 호스트를 다른 호스트로 분산시키는 장애 조치 메커니즘이 없으므로, 과도한 트래픽으로 인해 호스트가 연쇄 실패하는 문제가 없다


- 이 방식의 트레이드오프는 DoS, DDoS 공격 복원력이 떨어진다는 건
- 요청 속도가 매우 높으면 결국 호스트가 처리할 수 없고, 이 호스트에 할당된 모든 사용자의 속도를 제한할 수 없다
- 알림을 설정하고, 모든 서비스에서 이 사용자의 요청을 차단해야 한다
- LB는 이 IP의 요청을 드롭해야 한다
- 요청을 어떤 백엔드 호스트로 보내지 않되, 그 요청의 로깅은 해야 한다.


- 상태 비저장 접근 방식과 비교하면 더 복잡하고, 일관성과 정확성이 높지만 다음 항목은 더 낮다
	- 비용, 가용성, 내결함성


- 대체로 이 방식은 분산 DB의 사촌 격
- 자체 분산 저장 솔루션 설계하려 했지만, 널리 사용되는 분산 DB만큼 정교하거나 성숙하지 않다
- 단순성과 낮은 비용에 최적화되어 있으며 강한 일관성이나 높은 가용성은 없다.

## 8.8 모든 호스트에 모든 카운트 저장
- 레디스 사용
- 분산, 확장 가능, 고가용성 지님
- 낮은 지연 시간, 정확한 솔루션이 될 수 있음
- 그런데 속도 제한기를 외부 레디스 서비스에 의존해도 될까?


- 앞에서 본 상태 저장 설계는 백엔드에 상태 저장해 이 조회를 피하지만 LB가 모든 요청을 처리해 어떤 호스트로 보낼지 결정하고, 핫샤드 방지를 위해 재조정도 필요
- 모든 사용자 요청 타임스탬프가 단일 호스트의 메모리에 맞게 저장될 수 있다면? 어떨까?

### 8.8.1 고수준 아키텍처
- 앞서 100개 서비스에 대해서 808GB 저장 공간 이야기 함
- 서비스 별로 인스턴스 만들고, 적절히 라우팅하면 8GB 수준으로 줄일 수 있다. 이는 호스트 메모리에 맞출 수 있다
- 보통 높은 요청 수로 단일 호스트를 사용하지는 않으므로 128개의 호스트 사용한다면 64MB 저장하면 충분하다.
- 최종적으로 1~128 사이에서 결정

![](files/Pasted%20image%2020260523214536.png)

- L4 LB가 요청을 무작위로 분배하므로 정확하게 계산하려면 속도 제한 동기화가 필요
- 여러 방법이 있다.
- 여기서는 스트리밍을 사용한다.


- 이 설계는 일관성과 정확성을 낮은 지연 시간 및 높은 성능과 맞바꿨다
- 호스트가 속도 제한 결정 내리기 전에 모든 타임스탬프가 메모리에 있지 않을 수 있으므로 실제 값보다 낮은 요청 속도를 계산할 수 있다
- 다음 특성을 가진다
	- L4 LB로 어떤 호스트로든 요청 전달한다
	- 호스트는 메모리 데이터로 속도 제한 결정 내릴 수 있다
	- 데이터 동기화는 독립적인 프로세스에서 수행


- 호스트가 다운되고 데이터 손실되면?
- 잠시 동안 더 많은 요청을 할 수 있겠지만, 이는 허용 가능하다.

### 8.8.2 카운트 동기화
- 호스트는 사용자 요청 카운트를 어떻게 동기화?
- all to all 제외 다 우리의 속도 제한기에 적용 가능


- 동기화 메커니즘은 풀 vs 푸시?
- 일관성과 정확성 양보하는 대신 더 높은 성능, 낮은 리소스 소비, 낮은 복잡도 얻기 위해 푸시를 선택할 수 있다
- 호스트가 다운되면 해당 카운트 무시하고 사용자가 속도 제한되기 전 더 많은 요청을 하게 할 수 있다
- 이런 고려사항을 바탕으로 TCP 대신 UDP 써 비동기적으로 타임스탬프 공유하게 결정할 수 있다


- 호스트가 두 가지 주요 유형의 요청 트래픽 처리할 수 있어야 한다
	- 속도 제한 결정을 내리는 요청
	- 호스트의 메모리 내 타임스탬프 업데이트 요청

#### all to all
![](files/Pasted%20image%2020260523215137.png)
- 그룹 내 모든 노드가 다른 모든 노드에게 전송
- 브로드캐스팅보다 더 일반적
- 이를 위해서는 다른 모든 노드와 연결되는 full mesh가 필요
- 노드 수에 따라 제곱으로 확장되므로 확장 가능하지 않다
	- 128호스트에 대해서 이를 사용하면 128 * 128 * 64MB -> 1TB 이상 필요하며, 실현 불가능

#### 가십 프로토콜
![](files/Pasted%20image%2020260523215236.png)
- Gossip protocol에서는 주기적으로 무작위의 다른 노드를 선택해 메시지를 보낸다
- 야후는 이걸 쓴다
- 일관성과 정확성 낮춰 더 높은 성능, 더 낮은 리소스 소비를 얻을 수 있으나 더 복잡하기도 하다


- 이 또한 직접 서로에게 메시지 보내야 한다
- 모든 노드가 다른 노드의 IP 주소를 알아야 한다는 의미
- 주키퍼 등의 구성 서비스가 있어야 함

#### 외부 저장소와 조정 서비스
![](files/Pasted%20image%2020260523215241.png)
- 호스트는 리더 호스트를 통해 서로 통신
- 이 호스트는 주키퍼 등에 의해서 선출됨
- 각 호스트는 리더 호스트 IP만 알면 되고, 리더 호스트는 주기적으로 호스트 목록을 업데이트해야 한다

#### 무작위 리더 선출
![](files/Pasted%20image%2020260523215434.png)
- 리더 선출 알고리즘으로 복잡성 낮추는 대신 더 높은 리소스 소비 선택
- 각 리더가 다른 모든 호스트와 통신하는 한, 모든 호스트는 모든 요청 타임스탬프로 업데이트된다
- 불필요한 메시징 오버헤드가 있을 것

## 8.9 속도 제한 알고리즘
- 면접관은 우리가 속도 제한 알고리즘에 익숙할 것을 기대하지 않을 수 있다
- 이는 지원자의 의사소통 기술과 학습 능력 평가할 기회
- 면접관은 속도 제한 알고리즘을 설명하고 요구사항을 충족하는 솔루션 설계하기 위해 협력하는 능력을 평가할 수 있다
- 면접관은 광범위한 일반화나 오류가 있는 진술을 할 수 있고, 이를 비판적으로 평가하고 재치있고 단호하고 명확하고 간결하게 지적인 질문을 하고 기술적 의견을 표현하는 능력을 기를 수 있다

### 8.9.1 토큰 버킷
![](files/Pasted%20image%2020260523215630.png)
- 3개의 특성
	- 최대 토큰 수
	- 현재 사용 가능한 토큰 수
	- 버킷에 토큰이 추가되는 보충 속도


- 요청이 도착할 때마다 버킷에서 토큰을 하나씩 제거
- 토큰이 없으면 요청이 거부되거나 속도 제한
- 버킷은 일정 속도로 보충


- 간단 구현에서는 각 사용자 요청 시 다음이 발생
	- 호스트는 해시맵 사용해서 키-값 쌍 저장
	- 사용자 ID가 키
	- 키 없었으면 최대 토큰에서 1 뺀 값으로 생성
	- 키 있고, 0보다 크면 카운트 감소하고 거짓 반환. 0이면 참 반환(속도 제한)
	- 시스템은 매초마다 모든 값이 10보다 작으면 1씩 증가


- 장점
	- 이해하기 쉽고, 구현 쉽고 메모리 효율적
	- 토큰 세는 데 단일 정수 변수만 필요


- 각 호스트가 모든 키를 어떻게 증가?
	- 인메모리 해시 맵에서는 쉬움
	- 그러나 레디스처럼 외부라면?
	- `MSET` 등이 있지만 키 수 제한이 있다
	- 각 키가 64비트라면 1,000만 개의 키 업데이트 요청은 8.08GB
	- 업데이트를 여러 요청으로 나누면 각 요청은 리소스 오버헤드와 네트워크 지연 초래


- 키 삭제 메커니즘이 없음
	- 사용자의 마지막 요청 타임스탬프를 기록하는 별도의 저장 메커니즘과 오래된 키를 삭제하는 프로세스가 별도로 필요할 것


- 앞에서 본 분산 구현에서 모든 호스트는 자체 토큰 버킷을 가지고, 이 버킷으로 속도 제한 결정을 내릴 수 있다
- 버킷을 동기화할 수도 있다
- 버킷 동기화 전에 자신의 버킷 기준으로 속도 제한 여부를 판단하면 사용자가 설정된 속도 제한보다 더 높은 속도로 요청할 수도 있다.


### 8.9.2 누수 버킷
![](files/Pasted%20image%2020260525145038.png)
- 최대 토큰 수가 있고, 고정 속도로 누수되며 비어있을 때 누수가 멈춘다
- 요청이 도착할 때마다 버킷에 토큰을 추가. 버킷이 가득차면 요청이 거부되거나 속도가 제한됨


- 일반적인 구현은 고정 크기의 FIFO 큐 사용
- 고정 큐이므로 토큰 버킷보다 메모리 효율성이 낮다


- 이 알고리즘은 토큰 버킷처럼 몇 가지 문제를 가진다
	- 매초 호스트는 모든 키의 모든 큐를 디큐
	- 오래된 키를 삭제하는 별도 메커니즘 필요
	- 큐는 용량을 초과할 수 없으므로, 분산 구현에서는 동기화 전에 속도 제한을 초과할 수 있다


- 또 다른 설계는 토큰 대신 타임스탬프를 인큐
- 요청이 도착하면 먼저 큐에 남아있는 타임스탬프가 보존 기간보다 오래될 때까지 타임스탬프를 디큐한 후, 큐에 공간이 있으면 요청의 타임스탬프를 인큐
- 매초 모든 큐를 디큐해야 하는 요구사항을 피한다


- 단, 이는 일관성 문제가 있다
	- 호스트가 리더 호스트에 키-값 쌍을 쓰고, 즉시 다른 호스트에 덮어 쓰이는 경쟁 조건
	- 시간이 동기화되지 않아 호스트가 약간 다른 시간을 가진 호스트가 작성한 타임스탬프를 사용해 속도 제한 결정을 내릴 수 있다
- 이런 부정확성은 허용 가능하다
- 이 두 가지 문제는 타임스탬프를 사용하는 모든 알고리즘에도 적용된다.

### 8.9.3 고정 윈도우 카운터
- fixed window counter
- 키-값 쌍으로 구현
- 클라이언트 ID와 타임스탬프 조합일 수 있고, 값은 요청 횟수
- 클라가 요청하면 해당 키가 존재하면 증가하고, 없으면 생성
- 카운트가 설정된 속도 제한 내에 있으면 요청이 수락되고 카운트가 설정된 속도 제한을 초과하면 거부


- 윈도우 간격은 고정. 예를 들어 각 분의 [0, 60)초 사이일 수도 있다.
- 윈도우가 지나면 모든 키가 만료된다.
- `user0_1628825241` 키는 GMT 3:27:00 ~ 3:27:59까지 유효하다. 저 타임스탬프가 3:27:21로, 3:27의 AM의 1분 이내이므로

![](files/Pasted%20image%2020260525150422.png)

- 단점은 설정된 속도 제한을 최대 2배까지의 요청 속도를 허용할 수도 있다는 것

![](files/Pasted%20image%2020260525150432.png)
- 쿼리할 적절한 키를 결정. 속도 제한이 10라면 3:50:00~3:50:10 사이의 모든 시간이 키가 될 것
- MGET으로 한번에 조회할 수 있음
- 없으면 새로운 키-값 쌍 만들고, 하나의 키가 발견되면 그 값을 증가
- 하나 이상의 키가 발견되면 경쟁 조건으로 인해 반환된 모든 키의 값을 합산하고, 이 합계를 1 증가. 이것이 지난 10초 동안의 요청 수
- 병렬 방식
	- 생성되거나 업데이트된 키-값 쌍을 리더 호스트나 레디스에 쓴다. 여러 키가 있었다면 가장 오래된 키를 제외한 모든 키를 삭제
		- 그래야 너무 오랫동안 살아남는 key가 안 생김
	- 카운트가 10보다 크면 참을 반환하고, 그렇지 않으면 거짓을 반환


- 경쟁 조건으로 인해 여러 개의 키가 발견될 수 있는 이유는?
	- 여러 호스트가 동시에 조회 후 각각 키를 만들 수 있으므로
	- 혹은 clock skew


- 레디스는 TTL 설정이 있으므로 키가 10초 후에 만료되게 설정한다
- 아니라면 이를 지속적으로 찾아서 삭제하는 별도의 프로세스를 구현
- 이 프로세스가 필요하다면, 별도의 호스트로 구현할 수 있다
- 독립적으로 개발할 수 있어 테스트와 디버깅이 더 쉽다.


### 8.9.4 슬라이딩 윈도우 로그
![](files/Pasted%20image%2020260525150751.png)
- 각 클라이언트 키-값 쌍
- 키는 클라이언트 ID. 값은 정렬된 타임스탬프 목록
- 각 요청 타임스탬프를 저장


- 새로운 요청이 들어오면 해당 타임스탬프 추가하고, 첫 번째 타임스탬프가 만료되었는지 확인
- 만료되었다면 이진 검색으로 마지막으로 만료된 타임스탬프를 찾고 그 이전의 모든 타임스탬프를 제거
- 큐는 이진 검색 지원하지 않으므로 리스트를 대신 사용
- 리스트에 10개 이상의 타임스탬프가 있으면 참을 반환하고, 아니면 거짓


- 장점은 정확하다. 물론 분산 구현에서는 다양한 예외가 있다
- 그러나 모든 요청에 타임스탬프 값을 저장하는 것은 토큰 버킷보다 메모리를 더 쓴다
- 이는 속도 제한 초과 후에도 요청을 계속 집계하므로 사용자의 요청 속도를 측정할 수 있다.

### 8.9.5 슬라이딩 윈도우 카운터
- 고정된 윈도우 카운터 + 슬라이딩 윈도우 로그,
- 여러 개의 고정된 윈도우 간격을 사용. 각 간격은 속도 제한 시간 윈도우의 1/60
- 속도 제한 간격이 1시간이라면 1시간짜리 윈도우 하나가 아니라 1분짜리 윈도우 60개를 사용
- 현재 속도는 마지막 60개 윈도우 합산해서 결정
- 요청을 약간 과소 계산할 수 있다.
	- 11:00:35에 요청을 집계할 때 [10:01:00, 10:01:59] 윈도우부터 [11:00:00, 11:00:59] 윈 도우까지 60개의 1분 윈도우를 합산하고 [10:00:00, 10:00:59] 윈도우는 무시한다.
	- 그래도 고정 윈도우 카운터보다 정확하다

## 8.12 클라이언트 라이브러리로 기능 제공
- 사용자 서비스가 모든 요청의 속도 제한 서비스에 질의해야 하나? 사용자 서비스가 집계 후 특정 상황에서만 질의하는 방법도 있다
	- 배치할 사용자 요청이 쌓였을 때
	- 요청 속도가 갑자기 증가한 것을 감지했을 때
- 이를 일반화하면 속도 제한을 서비스 대신 라이브러리로 구현할 수 있지 않을까?
	- 8.7절 접근 방식을 써야 한다.
	- 호스트가 모든 사용자 요청을 메모리에 담고 서로 동기화
	- 타임스탬프 동기화하려면 주키퍼 등의 구성 서비스 설정이 필요한데, 이는 개발자에게 지나치게 복잡하고 오류 발생하기 쉬우므로 클라가 일부 처리를 수행해 속도 제한 서비스의 성능을 개선하는 라이브러리를 제공할 수도 있다
	- 이를 통해 서비스 요청 속도를 낮출 수 있다.
- 단, 이는 클라와 서버 간의 긴밀한 결합이 발생. 일반적으로 안티 패턴
- 클라가 데이터 처리를 하고자 한다면 다음 조건 중 최소한 하나를 충족
	- 처리가 단순해 서버 애플리케이션의 향후 버전에서도 이 클라 라이브러리를 쉽게 계속 지원
	- 처리가 리소스 집약적이어서 클라에서 이런 처리 수행하는 유지보수 부담이 서비스 운영 비용의 상당한 절감과 맞바꿀 가치가 있어야 한다
	- 클라가 더 이상 지원되지 않을 시기를 사용자에게 명확히 알리는 명시된 지원 수명 주기가 있어야 한다.


- 클라가 요청 속도도 측정하고, 요청 속도가 설정된 임계값 초과할 때만 속도 제한하는 건 어떤가?
- 이 방식의 문제는 클라끼리 통신하지 않으므로 클라가 설치된 특정 호스트의 요청 속도만 측정할 수 있고, 특정 사용자의 요청 속도를 측정할 수 없다는 것
- 이는 특정 사용자가 아닌 모든 사용자의 요청 속도를 기반으로 속도 제한이 활성화 됨
- 사용자가 이전에는 속도 제한을 받지 않던 요청 속도에서 갑자기 속도 제한을 받게되면 불만을 제기할 수도 있다
- 대안으로, 클라가 이상 감지를 사용해 요청 속도 급격한 증가를 감지한 후, 서버에 속도 제한 요청을 보내는 방법도 있다
