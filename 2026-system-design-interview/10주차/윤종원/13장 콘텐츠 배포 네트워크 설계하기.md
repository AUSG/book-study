# 13. 콘텐츠 배포 네트워크 설계하기

## 13.1 CDN의 장단점

### 13.1.1 CDN 사용의 장점
- 낮은 지연 시간
	- 사용자가 데이터 센터로부터 가깝다.
- 확장성
	- 우리 시스템 확장할 필요가 없다
- 낮은 단위 비용
	- 대량 할인을 일반적으로 제공
- 높은 처리량
	- CDN 회사의 처리량이 높다
- 높은 가용성
	- CDN이 SLA 유지만 할 수 있다면 우리 호스트가 실패할 때 대체 수단으로 활용 가능

### 13.1.2 CDN 사용의 단점
- 추가적인 복잡성
	- 추가 DNS 조회
	- 추가 실패 지점
- 트래픽이 적으면 단위 비용이 높아질 수 있다. CDN이 서드파티 네트워크 사용하면 데이터 전송 비용 등
- 다른 CDN으로 마이그레이션이 어렵다. 높은 비용 + 오랜 시간
	- 마이그레이션해야 하는 이유는
		- CDN이 커버하지 못하는 지역이 많아 옮기거나
		- CDN 회사가 망하거나
		- CDN 회사가 SLA를 잘 지키지 못하거나
- 특정 국가나 조직이 CDN IP 주소 차단
- 서드파티에 데이터 저장하는 문제로 보안, 프라이버시 우려.
- js 등에 악성 코드 삽입, 원격 호스팅 라이브러리 보안 등
- CDN 회사에 문제가 생기면 언제 해결될지 장담할 수 없다
- CDN이나 일반적인 서드파티, 서비스의 구성 관리가 특정 사용 사례를 충분히 사용자 정의할 수 없어 예상치 못한 문제가 발생할 수 있다.

### 13.1.3 이미지 제공 CDN을 사용할 때 발생할 수 있는 예상치 못한 문제의 예
- CDN은 GET 요청의 User Agent 헤더를 읽어 웹브라우저에서 온 걸 확인하고 PNG나 JPEG 대신 WebP 등을 반환할 수도 있다
- 일부 서비스에서는 이상적일 수 있지만 원본 이미지를 받고 싶은 경우 세 가지 선택지가 있다
	- User Agent 헤더를 재정의
	- 특정 서비스에서는 WebP 이미지를 제공하고, 다른 서비스에는 원본 형식으로 이미지를 제공하게 CDN 구성한다
	- 요청을 백엔드 서비스로 라우팅
		- 이는 CDN의 이점을 잃는다

## 13.2 요구사항
- 권한이 있는 사용자는 디렉터리를 만들고, 10GB 크기 제한 파일 업로드 및 다운로드 가능


- 비기능
	- 확장성 : 페타바이트 규모의 저장 용량, 하루 테라바이트 수준의 다운로드 용량 지원
	- 고가용성 : 4~5개의 9 가동 시간
	- 고성능 : 가장 빠르게 제공될 수 있는 데이터센터에서 다운로드. 업로드 성능은 덜 중요
	- 내구성
	- 보안, 프라이버시

## 13.3 CDN 인증과 권한 부여
- CDN 인증, 권한 부여는 쿠키 기반 인증이나 토큰 기반 인증
- 토큰 기반 인증은 메모리 덜 쓰고 보안 전문성이 더 높은 서드파티 서비스를 사용할 수 있고, 세밀한 접근 제어가 가능하다
- 이런 이점 외에도 CDN을 위해 토큰 인증은 접근자를 허용된 IP 주소나 특정 사용자 계정으로 제한할 수 있다.

### 13.3.1 CDN 인증과 권한 부여 단계
![](files/Pasted%20image%2020260628144602.png)
- CDN은 각 사용 업체에게 비밀 키를 발급하고 다음 정보로 접근 토큰 생성하는 SDK나 라이브러리를 제공한다
	1. 사용자가 우리에게 인증 요청을 보낸다.
	2. 우리가 다음 정보를 가지고 CDN SDK로 CDN 접근 토큰을 생성한다.
		 - 비밀 키
		 - CDN URL
		 - 만료 기한
		 - Referrer
			 - 클라, 사용자가 CDN에 HTTP 요청할 때는 CDN 사용 업체 URL을 Referrer HTTP 헤더로 포함해야 한다. CDN은 승인된 Referrer만 허용한다.
			 - 그러나 이는 쉽게 위조가 가능하다.
		 - 허용된 IP
		 - 허용된 국가나 지역
			 - 국가, 지역의 블랙리스트나 화이트리스트 포함 가능
			 - IP가 이미 지역, 국가를 나타내지만 편의를 위해 따로 유지할 수도 있다
	3. 우리가 토큰 저장 후 사용자에게 반환. 추가 보안을 위해 토큰을 암호화해서 저장할 수도 있다.
	4. 우리가 사용자에게 CDN URL 제공할 때 마다, CDN 자산에 GET 요청할 때마다 접근 토큰으로 서명해야 한다. 이를 URL 서명이라고 한다.
	5. 사용자 로그아웃하면 우리는 사용자의 토큰 파기.


- 토큰 삭제는 비동기적일 수 있고, 로그아웃이 자주 일어나는 이벤트가 아니므로 동기일 수도 있다.
- 토큰 삭제가 비동기면 CDN 업체에 일시적으로 문제가 생기면 토큰이 삭제되지 않을 수도 있다
- 이 문제를 해결하는 한 가지 방법은 이런 케이스를 무시하고 허용하는 거다
- 다른 방법은 이벤트 기반 접근을 사용하는 것.
- 또 다른 방법은 동기식, 차단 방식으로 구현하는 것

### 13.3.2 키 교체
- 해커가 키를 훔쳐도 영향을 줄이기 위해 주기적으로 변경할 수 있다
- 키는 갑자기 변경되는 게 아니라 교체된다. key rotation은 이전 키와 새 키 모두 유효한 기간을 포함하는 키 갱신 과정

## 13.4 상위 수준 아키텍처
![](files/Pasted%20image%2020260628145015.png)

- 작업을 읽기 및 쓰기로 일반화할 수 있다
- 초기 설계를 단순화하려면 모든 파일을 모든 데이터센터에 복제
- 그렇지 않으면 다음 복잡성 처리 필요

	- 사용자 쿼리 메타데이터를 주기적으로 사용해 데이터 센터 간 최적의 파일 분배를 결정하는 파일 분배 시스템이 필요하다. 복제본 수, 위치 등이 포함된다.

## 13.5 저장소 서비스
- 대용량 파일 저장이라 DB를 쓸 수 없다
- 파일은 호스트의 파일 시스템에 저장해야 함
- 가용성, 내구성을 위해 복제가 필요
- 각 파일을 여러 호스트에 할당
- 가용성 모니터링과 메타데이터 서비스를 업데이트하고 대체 노드를 프로비저닝하는 장애 조치 프로세스 필요

### 13.5.1 클러스터 내부
- 주키퍼를 클러스터 내부 관리자로 포함하는 HDFS 같은 분산 파일 시스템
- 클러스터 내부 관리자 역시 신뢰성, 확장성, 고성능 필요한 매우 정교한 구성 요소
- 이런 구성 요소를 피할 수 있는 대안은 클러스터 외부 관리자

### 13.5.2 클러스터 외부
- 외부 관리자가 관리하는 각 클러스터는 여러 데이터 센터에 분산된 3개 이상의 노드로 구성된다
- 파일을 읽거나 쓰려면 메타데이터 서비스가 파일이 저장되어 있거나 저장되어야 할 클러스터를 식별한 다음 클러스터에서 무작위로 선택한 노드에서 파일을 읽거나 쓴다
- 이 노드는 클러스터 내의 다른 노드로의 복제를 담당한다
- 리더 선출은 필요없지만 파일을 클러스터에 매핑하는 것은 필요하다
- 클러스터 외부 관리자는 파일과 클러스터의 매핑을 유지한다.

### 13.5.3 평가

|**클러스터 내부 관리자**|**클러스터 외부 관리자**|
|---|---|
|메타데이터 서비스가 클러스터 내부 관리자에게 요청하지 않는다.|메타데이터 서비스가 클러스터 외부 관리자에게 요청을 한다.|
|클러스터 내 개별 역할의 파일 할당을 관리한다.|클러스터로의 파일 할당을 관리하지만 개별 노드로는 관리하지 않는다.|
|클러스터의 모든 노드를 알아야 한다.|각 개별 노드에 대해서는 알지 못할 수 있지만 각 클러스터에 대해서는 알아야 한다.|
|노드의 하트비트를 모니터링한다.|각 독립 클러스터의 상태를 모니터링한다.|
|호스트 실패를 처리한다. 노드가 죽을 수 있고 새 노드가 클러스터에 추가될 수 있다.|각 클러스터의 사용률을 추적하고 과부하된 클러스터를 처리한다. 용량 한계에 도달한 클러스터에는 더 이상 새 파일을 할당하지 않을 수 있다.|

## 13.6 일반적인 작업
- CDN 도메인에 질의하면 GeoDNS가 가장 가까운 호스트의 IP 주소 할당 후 LB가 API Gateway 호스트로 요청 보낸다.
- API Gateway는 캐싱 등 여러 작업을 수행한다.

### 13.6.1 읽기: 다운로드
- 다운로드에서 다음 단계는 이 요청을 처리할 저장소 호스트를 선택하는 것
- 메타데이터 서비스는 아래 메타데이터 유지하고 제공해 이를 지원한다 (Redis, SQL 모두 사용 가능)
	- 파일을 포함하는 저장소 서비스 호스트, 데이터 센터 정보
	- 호스트의 현재 부하 측정. 이는 제공 중인 파일 크기 합으로 비슷하게 산정할 수 있다
	- 파일 다운로드하는 데 걸리는 시간을 예측하거나 동일한 이름을 가진 파일들을 구분. 후자일 때는 주로 MD5, SHA 해시 등을 사용
	- 파일 소유권, 접근 제어
	- 호스트의 상태

#### 다운로드 과정
![](files/Pasted%20image%2020260628154941.png)

1. 속도 제한 서비스로 요청이 제한 초과하는지 검사. 속도 제한기가 요청을 허용한다고 가정
2. 메타데이터 서비스에 쿼리해 이 자산을 포함하는 저장소 서비스 호스트를 가져온다
3. 저장소 호스트를 선택하고 자산을 클라로 스트리밍
4. 호스트 부하 증가로 메타데이터 서비스 업데이트. 메타데이터 서비스가 자산의 크기를 기록하고 있다면 3단계와 병렬 수행 가능. 아니면 API Gateway가 자산의 크기를 측정해 정확한 부하 증가를 메타데이터 서비스로 업데이트해야 한다.
	- 이는 비동기로 수행 가능
	- 메타데이터 서비스가 업데이트를 받지 못했다면 이 오류를 무시하고 허용된 양보다 더 많이 CDN을 사용하게 허용할 수도 있다
	- 혹은 이벤트 방식으로 처리한다거나

![](files/13장%20콘텐츠%20배포%20네트워크%20설계하기.png)

- CDN에 자산이 없을 수도 있다
	- 보존 기간이 지났거나
	- CDN 문제로 업로드되지 않았거나
	- CDN의 기타 오류
- 이런 경우 원본에서 다운로드해야 한다. 이로 인해 지연시간이 증가한다
- 그 다음 저장소 서비스에 업로드하고 메타데이터 서비스를 업데이트해 저장해야 한다
- 지연 시간 최소화하기 위해 저장 프로세스는 클라에 자산을 반환하는 작업과 병렬로 수행할 수 있다.

#### 저장 시 암호화가 적용된 다운로드 과정
![](files/Pasted%20image%2020260628155006.png)
- 자산을 암호화된 형태로 저장해야 한다면?
- 암호화 키 인증이 필요한 비밀 관리 서비스에 저장할 수 있다
- API Gateway 호스트가 초기화될 때 비밀 관리 서비스로 인증하면 비밀 관리 서비스는 향후 요청을 위해 토큰을 호스트에 전달
- 인가된 사용자가 자산을 요청할 때 호스트는 먼저 비밀 관리 서비스에서 자산의 암호화 키를 얻고 암호화된 자산을 가져와 복호화 후 반환
- 자산이 크면 여러 블록으로 저장되어 있을 수 있으므로 각 블록을 따로 가져와 복호화해야 한다.

![](files/Pasted%20image%2020260628155012.png)
- CDN이 보유하지 않은 암호화된 자산을 가져오는 경우
- 이 또한 원본에서 자산을 가져와야 함
- API Gateway는 임의의 암호화 키 생성 후 자산 암호화 후 자산을 저장소 서비스에 쓰고 키를 비밀 관리 서비스에 쓴다.

### 13.6.2 쓰기: 디렉터리 생성, 파일 업로드, 파일 삭제
- 파일은 내용이 아닌 ID로 식별 (이름 같을 수 있으니까)
- 파일이 완전 똑같지만 소유자가 다른 파일은 따로 저장할까 아니면 복사본 하나만 유지?
- 이런 방식으로 저장 공간을 절약하려면 소유자 그룹을 따로 관리하는 등 복잡성 계층을 추가 구축해야 한다
- 그래야 소유자가 다른 그룹에 속한 소유자가 아니라 자신이 인식하는 또 다른 소유자가 파일에 접근할 수 있음을 확인할 수 있다.
- 동일한 파일 수가 전체 파일 중 적은 비율이라면 과도한 설계일 수 있다
- 단, 이 가정은 바뀔 수 있으므로 나중에는 복잡성을 감수해서라도 비용을 절감할만 할 수도 있다.


- 파일 크기가 GB, TB가 된다면 업로드나 다운로드 중 실패가 발생했을 때 처음부터 하는 건 낭비다
- 체크포인팅이나 벌크헤드와 같은 유사 프로세스를 개발해 청크를 나눠야만 한다.
- 클라는 완료되지 않은 청크에만 업로드나 다운로드 작업을 하면 된다
- 이를 멀티파트 업로드라고 하며, 같은 원리를 다운로드에 적용할 수도 있다.


- 멀티파트 업로드를 위한 프로토콜을 설계할 수도 있다
- 청크 업로드는 독립 파일 업로드와 같다.
- 단순화를 위해 청크 크기는 128MB와 같이 고정할 수도 있다
- 클라가 청크 업로드를 시작할 때 사용자 ID, 파일 이름, 크기 등 일반적인 메타데이터가 포함된 초기 메시지를 보낼 수 있다. 업로드할 청크 번호도 포함할 수 있다
- 저장소 호스트는 파일을 저장할 적절한 디스크 주소 범위 할당하고 정보를 기록
- 청크 업로드를 받기 시작하면 적절한 주소에 청크를 써야 함
- 메타데이터 서비스는 어떤 청크 업로드가 완료되었는지 추적할 수 있다
- 마지막 청크 업로드를 완료하면 메타데이터 서비스는 파일을 복제 및 다운로드 준비가 완료된 것으로 표시한다.
- 청크 업로드가 실패하면 클라는 전체 파일이 아닌 해당 청크만 다시 업로드


- 모든 청크 업로드 전에 중단하면 저장소 호스트의 공간을 쓸데없이 차지
- 주기적으로 이를 삭제하는 크론 작업이나 배치 ETL 작업을 구현할 수 있다.


- 다른 설명 가능한 주제
	- 클라가 청크 크기 선택할 수 있게 하는 것
	- 파일을 업로드하는 동안 복제해 CDN 전체에서 더 빨리 다운로드할 수 있게 하는 것. 추가적인 복잡성을 도입하며 필요할 가능성은 낮지만 그런 고성능이 필요하다면 논의할 수도 있다
	- 클라가 첫 번째 청크 다운로드하자마자 파일 재생을 시작할 수 있게 하는 것
	- 특정 데이터 센터가 이런 작업을 수행하고 다른 데이터 센터에 전파하게 지정하는 단일 리더 접근 방식. 이는 모든 데이터 센터에서 변경 사항을 신속하게 사용할 필요가 없다면 충분할 수도 있다
	- 튜플 포함한 다중 리더 접근 방식
	- 클라가 모든 데이터 센터에서 이 파일의 잠금을 획득하고, 모든 데이터 센터에서 작업 수행 후 잠금을 해제

#### 모든 데이터 센터에 파일 사본 유지하지 않기
- 모든 파일이 모든 곳에서 사용되지는 않는다. 특정 파일이 특정 지역에서만 사용될 수도 있으니 모든 데이터 센터에 사본이 복사될 필요는 없다.
- 복제 기준을 설정해 파일을 특정 데이터센터에 복사해야 하는 시기를 결정할 수 있다 (지난 달 해당 파일의 요청 수나 사용자 수 등)
- 이는 내결함성을 위해 데이터 센터 내에 파일을 복제해야 함을 의미


- 특정 콘텐츠는 요구사항 때문에 여러 파일로 분리된다.
- 비디오 파일은 모든 사용자에게 제공되고 특정 언어의 오디오 파일은 일부 유저에게만 제공된다거나 등
- 이런 로직은 애플리케이션 단에서 처리할 수도 있다.

#### 배치 ETL 작업의 재조정
- 다양한 데이터 센터에 파일 분산하고, 수요 충족하기 위해 적절한 수의 호스트에 파일 복제하는 주기적인 배치 작업이 필요하다
- 배치 작업은 로깅 서비스에서 이전 기간의 파일 다운로드 로그를 가져와 파일의 요청 수를 확인하고, 이 숫자로 각 파일 저장소 호스트 수를 조정한다
- 그다음 각 노드에 추가하거나 삭제해야 할 파일의 맵을 만들고 이 맵을 사용해 해당 셔플링을 수행한다.


- 실시간 동기화를 위해 메타데이터 서비스를 더 발전시켜 파일 위치와 접근을 지속적으로 분석하고 파일을 재분배할 수도 있다
- 데이터 센터 간 복제는 고난이도 주제로, 관련 전문 지식이 요구되지 않는 면접에서는 깊이 다루지 않을 수도 있다.


- 메타데이터 서비스에 새 파일 메타데이터를 쓰고, 그에 따라 저장소 서비스의 데이터 센터 또는 호스트 간에 파일을 셔플하는 접근 방식
	- 다양한 데이터 센터의 호스트 간에 파일을 전송하려면 저장소 서비스에 요청해야 함
	- 쓰기 요청 실패 시 메타데이터 서비스와 저장소 서비스 간 불일치 방지하기 위해 메타데이터 서비스는 저장소 서비스로부터 파일이 새 위치에 성공적으로 기록되었다는 성공 응답을 받았을 때만 파일 위치 메타데이터를 업데이트해야 한다
	- 저장소 서비스는 자체 노드, 호스트 내 일관성 보장을 위해 관리자(클러스터 내외)에 의존
	- 이는 파일이 해당 파일이 성공적으로 기록되기 전에 메타데이터 서비스가 파일 위치를 반환하지 않게 보장한다.
- 또한 파일이 새 위치에 성공적으로 기록된 후에만 이전 노드에서 삭제되어야 함
- 그래야 새 위치 쓰기 실패 시 문제를 방지할 수 있음

## 13.8 로깅, 모니터링, 경보
- 다음 모니터링, 경보
	- 업로더가 파일 업로드 중 완료나 실패 상태 추적 가능해야 함
	- CDN 미스 기록하고 모니터링해 낮은 긴급도의 경보 트리거
	- 프론트엔드 서비스가 파일의 요청 속도를 기록할 수 있어야 함. 공유 로깅 서비스에서 수행 가능
	- 비정상적이거나 악의적인 활동 모니터링

## 13.9 미디어 파일 다운로드에 대한 기타 가능한 논의
- 미디어 파일은 완전 다운로드 전에 재생하고 싶을 수 있다
- 해결책은 미디어 파일을 더 작게 나누는 것
- 이를 위해서는 부분 파일을 받아서 재생할 수 있는 클라 사이드 미디어 플레이어가 필요하다
- 이런 세부사항은 면접 범위를 벗어난다


- 단, 순서가 중요하므로 메타데이터가 필요하다.
- 시스템은 파일을 더 작게 나누고 각 작은 파일에 시퀀스 번호 할당한다
- 파일 순서, 총 개수 정보도 필요
- 특정 순서로 파일 효율적으로 다운로드하려면 어떻게 해야할지에 대한 논의도 필요
